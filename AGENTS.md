# AGENTS.md
# Repository Guidelines for Codex / Claude Code / AI Workers
本文件是当前仓库的**全局执行指导文件**。
后续无论是 Codex、Claude Code，还是其他开发代理进入本仓库工作，都必须优先遵循本文件，再结合 `product_document/` 中的产品与架构文档执行。
---
## 1. 项目定位
本仓库不是普通的接口自动化测试脚本仓库，也不再只是一个手写 pytest 测试框架。
当前项目的目标是建设一个：
> **统一接口测试资产平台 + AI 生成/编排能力 + 规则校验与执行验证闭环**
你后续的所有工作，都必须服务于这个目标。
### 1.1 当前阶段定位
当前阶段仍处于：
- 需求整理阶段
- 产品说明书完善阶段
- 总体架构设计阶段
- 中间模型设计阶段
- 现有测试框架改造规划阶段
**当前不是自由实现阶段。**
在没有对应说明书、设计文档和测试设计之前，不要随意开始写大段业务实现代码。
---
## 2. 最高优先级规则
### 2.1 产品说明书优先
后续所有开发、设计、重构、改造，都必须优先参考：
- `product_document/产品需求说明书(全局).md`
- `product_document/总体架构设计说明书.md`
- `product_document/中间模型设计说明书.md`
- `product_document/现有接口自动化测试框架改造方案.md`
- 后续新增的 V1 / 模块 / 测试说明书
如果代码现状与文档冲突：
- 不要擅自按旧代码逻辑延续；
- 应以当前确认后的产品与架构文档为准；
- 如文档不清晰，先补文档，再改代码。
### 2.2 TDD 强制执行
后续开发必须采用 **TDD 思路**：
1. 先明确需求说明
2. 先设计详细测试用例
3. 先写测试
4. 再做最小实现
5. 跑测试验证
6. 通过后再继续推进
### 2.3 不允许直接把 AI 输出当成最终结果
AI 生成内容只能作为：
- 草稿
- 候选方案
- 补全结果
- 修订建议
不允许把自由生成内容未经：
- 模板规范
- 规则校验
- 测试验证
直接落为最终生产资产。
### 2.4 不允许破坏平台化方向
为了快速完成局部功能，不允许：
- 在旧结构上无规则堆代码；
- 临时写死目录、命名、字段和流程；
- 绕过中间模型直接拼最终代码；
- 让生成逻辑与 UI、执行、资产记录强耦合。

## 3. 当前认可的产品与架构方向
### 3.1 平台核心方向
平台未来需要同时支持三条输入路线：
1. 文档驱动
2. 功能测试用例驱动
3. 抓包驱动
但开发上必须分阶段推进，当前 V1 优先聚焦：
- 底座改造
- 文档驱动最小闭环
- 中间模型雏形
- 模板体系雏形
- 规则体系雏形
- 执行闭环雏形
### 3.2 技术栈方向
当前架构方向已明确：
- 核心语言：**Python**
- 测试执行：**pytest + requests**
- 模板体系：优先考虑 **Jinja2**
- Web 服务层：优先考虑 **FastAPI**
- 前端：优先考虑 **React + TypeScript**
- Desktop：采用 **前端壳 + 本地服务内核** 路线，优先评估 **Tauri**
如无明确批准，不要自行改成其他核心路线。
### 3.3 运行模式方向
- 当前以 **本地单机模式优先** 设计核心
- 后续再扩展服务化 / Web / 多人协作能力
### 3.4 桌面与 Web 关系
无论未来是 Web 访问还是桌面应用：
- 核心业务逻辑必须共用
- 不允许把核心业务写死在前端壳里
- Desktop 与 Web 应只是交互层的不同承载方式

## 5. 后续开发必须遵守的工程原则
### 5.1 中间模型优先
后续所有生成能力，都必须优先围绕统一中间模型建设。
不允许：
- 输入文档直接拼 Python 文件
- 输入抓包直接拼 pytest 用例
- 输入功能测试用例直接自由生成最终代码
必须走：
> 输入 → 解析标准化 → 中间模型 → 模板/规则 → 生成资产 → 执行验证
### 5.2 模板化优先
后续所有代码生成都必须优先通过模板体系生成，包括但不限于：
- 接口方法
- 测试用例
- 断言
- 数据文件
- 生成记录
### 5.3 规则校验优先
所有生成结果都必须可被规则层检查，包括：
- 命名规范
- 模块归档
- 参数规范
- 引用关系
- 断言规范
- 目录结构
### 5.4 资产化优先
后续输出结果不是一次性代码，而是测试资产。
每一类生成物后续都应考虑：
- 来源
- 生成时间
- 审核状态
- 执行状态
- 可追溯性
### 5.5 可扩展优先
实现方案必须优先考虑：
- V2 的功能用例驱动
- V2 的抓包驱动
- Web 访问
- Windows 桌面应用
- 后续 macOS 扩展
不要只为了当前一个小用例写死结构。

## 6. 编码与改造要求
### 6.1 改造策略
当前阶段采用：
- **增量式改造**
- 先稳住底座
- 再接最小闭环
不建议：
- 一次性全量推翻原有框架
- 一次性做大重构而不留过渡层
### 6.2 文件职责要求
后续新建文件时，要尽量保持单一职责：
- 解析逻辑独立
- 模型定义独立
- 模板独立
- 规则独立
- 执行器独立
- 报告独立
### 6.3 命名要求
保持清晰、稳定、可预测：
- 类：`PascalCase`
- 函数 / 变量：`snake_case`
- 常量：`UPPER_SNAKE_CASE`
- 文档文件：中文名称可保留，但代码路径保持清晰
---
## 7. 文档优先要求
后续遇到以下情况，优先补文档而不是直接写代码：
1. 需求边界不清晰
2. 模块职责不清晰
3. 目录结构还没定
4. 生成策略还没定
5. 测试方式还没定
6. 阶段范围还没定
### 7.1 当前优先文档顺序
当前阶段的执行顺序应优先围绕：
1. 中间模型设计
2. 框架改造方案
3. AGENTS / README / 阶段说明文档
4. V1 最小闭环说明书
5. 详细测试用例说明
6. 再进入实现
---
## 8. README 与阶段说明书要求
### 8.1 README
README 不是简单项目介绍，而是当前仓库的统一说明入口。
后续如果变更平台方向、阶段目标、使用方式或结构，应同步更新 README。
### 8.2 阶段说明书
后续每个阶段（例如 V1）都必须有单独说明书，并且要包含：
- 目标范围
- 不做内容
- 详细设计方案
- 开发方案
- 测试方案
- 开发进度记录
- 测试进度记录
- 风险与阻塞记录
阶段说明书不是一次性文档，而是持续更新的执行文档。
---
## 9. Git 与提交要求
### 9.1 提交要求
每轮完成明确文档或代码改动后，应及时提交。
提交信息要求：
- 简洁
- 语义明确
- 优先使用英文动词 + 范围
例如：
- `docs: add intermediate model design spec`
- `docs: refine PRD and add architecture design spec`
- `refactor: split api client and utility layers`
### 9.2 推送要求
推送远端 GitHub 属于外部动作。
在实际推送前，应确认：
- 远端仓库配置是否存在
- 当前分支是否正确
- 是否存在未提交改动
- 推送目标是否符合当前阶段预期
---
## 10. 当前明确不允许的做法
1. 不允许绕过产品说明书直接自由开发大功能
2. 不允许为了快而破坏中间模型和模板化方向
3. 不允许把 AI 输出直接当最终事实
4. 不允许在没有测试设计的情况下宣称功能完成
5. 不允许把当前仓库重新做成与原技术栈完全无关的新项目
6. 不允许为了 UI 方便把核心逻辑耦合到 Web 或桌面层
7. 不允许把 V1 做成大而全的失控版本
---
## 11. 当前最重要的执行目标
如果你是 Codex 或其他代理，当前最重要的事情不是直接造功能，而是：
1. 遵循 `product_document/` 中已经确认的文档体系
2. 继续补齐阶段设计文档
3. 围绕统一中间模型与框架改造推进 V1 设计闭环
4. 为后续 TDD 开发做好清晰、稳定、可执行的结构准备
---
## 12. 结论
当前仓库已经进入从“测试框架”向“测试平台”演进的关键阶段。
从现在开始，所有工作都必须遵守以下主线：
> **产品说明书优先 → 架构说明优先 → 中间模型优先 → 模板与规则优先 → TDD 开发 → 执行验证 → 资产沉淀**
偏离这条主线的实现，即使短期看起来能跑，也会在后续平台化过程中产生高额返工成本。
