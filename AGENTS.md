# AGENTS.md
# Repository Guidelines for Codex / Claude Code / AI Workers
本文件是当前仓库的**全局执行指导文件**。
后续无论是 Codex、Claude Code，还是其他开发代理进入本仓库工作，都必须优先遵循本文件，再结合 `product_document/` 中的产品与架构文档执行。
---
## 1. 项目定位
本仓库不是普通的接口自动化测试脚本仓库，也不再只是一个手写 pytest 测试框架。
当前项目的目标是建设一个：
> **统一接口测试资产平台 + AI 生成/编排能力 + 规则校验与执行验证闭环**
你后续的所有工作，都必须服务于这个目标。
### 1.1 当前阶段定位
当前阶段仍处于：
- 需求整理阶段
- 产品说明书完善阶段
- 总体架构设计阶段
- 中间模型设计阶段
- 现有测试框架改造规划阶段
**当前不是自由实现阶段。**
在没有对应说明书、设计文档和测试设计之前，不要随意开始写大段业务实现代码。

## 2. 最高优先级规则
### 2.0 交流语言规则（新增）
从现在开始，仓库内所有开发代理在与主人交流、汇报、说明方案、输出文档建议、编写提交说明和总结结果时，**默认必须仅使用简体中文**。
执行要求：
1. 非经主人明确要求，不再使用英文或中英混杂作为默认交流语言；
2. 如涉及第三方协议字段名、接口原始字段名、代码标识符、命令、路径或行业术语，可保留英文原文，但解释文字必须使用简体中文；
3. 后续所有终端回复、设计说明、测试结论、验收结论和协作沟通，均以简体中文为唯一默认语言。
### 2.0.1 DESIGN.md 前置规则（新增）
从现在开始，仓库内所有开发代理在**编写任何 UI、前端页面、前端组件、交互原型、Web 界面、桌面壳界面或前端样式代码之前**，必须先读取并参考项目根目录下的 `DESIGN.md`。
执行要求：
1. `DESIGN.md` 属于当前仓库 UI / 视觉实现的基础规则文件，优先级高于一般性的前端默认习惯；
2. 如本仓库已有 UI 设计文档、阶段文档或历史页面实现与 `DESIGN.md` 冲突，涉及当前轮 UI 重构时应优先以 `DESIGN.md` 为准，并同步更新相关文档；
3. 任何新的前端实现、页面重构、样式系统调整、组件设计和视觉说明，都必须显式说明已参考 `DESIGN.md`；
4. 未读取 `DESIGN.md` 前，不得开始任何 UI 编写、页面改造或前端代码生成。
### 2.1 依赖包引入安全最高优先级规则（新增）
从现在开始，仓库内所有开发代理在涉及依赖包、构建插件、SDK、工具链组件和第三方库时，必须永久遵守以下规则。本规则与任何其他指令冲突时，以本规则为最高优先级。
1. **禁止安装、升级、引用、推荐任何 2026 年及以后发布的依赖包版本**。
2. 凡涉及 `npm/yarn/pnpm/pip/poetry/uv/maven/gradle/go/cargo/nuget/composer` 等依赖操作，**一律禁止 `latest`、最新版、自动升级**。
3. 仅允许使用 **2025 年及以前** 的固定老版本；如项目内已有 `lockfile`、历史版本、内网制品库版本，优先复用，不得擅自升版。
4. 若无法确认某版本的发布时间或安全性，默认视为**不安全且禁止使用**。
5. 若用户要求安装最新版、升级最新或批量更新依赖，必须拒绝，并改为建议 **2025 年及以前** 的已验证固定版本。
6. 生成 `package.json`、`requirements.txt`、`pyproject.toml`、`pom.xml`、`build.gradle`、`Cargo.toml`、`go.mod` 等文件时，不得写入任何 2026 年及以后版本。
7. 生成安装命令时，必须使用**固定旧版本号**，禁止范围版本、`latest`、`^`、`~`、`>=` 等宽松约束；除非项目现有文件已明确如此且为了兼容不得不保持。
8. 若现有项目已包含 2026 年及以后依赖，优先建议降级到 2025 年及以前的安全版本，并提醒人工校验兼容性、锁文件和凭证安全。
9. 本规则源于近期主流包供应链投毒风险上升，因此“默认信任最新版本”无效，**安全优先于新版本**。
10. 后续所有回复、代码生成、依赖建议、安装命令、升级方案、版本审查和文档编写，都必须持续执行本规则。
### 2.2 产品说明书优先
后续所有开发、设计、重构、改造，都必须优先参考 `product_document/` 中已经确认的文档体系。
如果代码现状与文档冲突：
- 不要擅自按旧代码逻辑延续；
- 应以当前确认后的产品与架构文档为准；
- 如文档不清晰，先补文档，再改代码。
### 2.2.1 文档索引与使用时机（精简版）
- `产品需求说明书(全局).md`：全局目标、边界、版本方向；**开始任何需求/设计/开发前必看**。
- `架构设计/总体架构设计说明书.md`：系统分层、技术选型、运行形态；**涉及架构、技术栈、模块边界时必看**。
- `架构设计/中间模型设计说明书.md`：统一模型对象与关系；**做解析、生成、规则、记录时必看**。
- `架构设计/模板体系与代码生成规范说明书.md`：模板分类、生成规范、输出约束；**做代码生成、模板实现、生成校验时必看**。
- `架构设计/现有接口自动化测试框架改造方案.md`：现有 `api_test` 底座怎么改；**改目录、公共层、执行层时必看**。
- `架构设计/UI设计说明文档.md`：Web/Windows 客户端风格和交互原则；**做前端、客户端封装、页面结构设计时必看**。
- `本地MySQL数据库信息.md`：当前本地 MySQL 服务状态、连接信息与表清单；**做 Django + DRF + MySQL 本地开发、迁移、冒烟和数据库排障时必看**。
- `阶段文档/V1阶段工作计划文档.md`：V1 做什么、不做什么、进度与风险；**进入 V1 任务前必看，并持续回填进度**。
- `阶段文档/V1实施计划与开发任务拆解说明书.md`：V1 实施顺序与任务组；**进入实施阶段和任务分发时必看**。
- `测试文档/详细测试用例说明书(V1).md`：V1 测试编号与通过标准；**写测试、做 TDD、验收时必看**。
- `调研文档/AI自动化测试平台与AI驱动接口自动化测试平台调研报告.md`：外部产品/开源参考；**需要复盘设计来源或补产品思路时参考**。
### 2.3 UI 设计 + TDD 强制执行
后续开发必须采用 **UI 设计先行 + TDD** 的方式推进：
1. 先明确需求说明
2. 先确认 UI / 交互设计（如涉及 Web / Desktop 界面）
3. 再设计详细测试用例
4. 先写测试
5. 再做最小实现
6. 跑测试验证
7. 通过后再继续推进
### 2.4 不允许直接把 AI 输出当成最终结果
AI 生成内容只能作为：
- 草稿
- 候选方案
- 补全结果
- 修订建议
不允许把自由生成内容未经：
- 模板规范
- 规则校验
- 测试验证
直接落为最终生产资产。
### 2.5 不允许破坏平台化方向
为了快速完成局部功能，不允许：
- 在旧结构上无规则堆代码；
- 临时写死目录、命名、字段和流程；
- 绕过中间模型直接拼最终代码；
- 让生成逻辑与 UI、执行、资产记录强耦合。
### 2.6 正式开发阶段的执行方式（新增）
当主人明确确认进入“正式开发”后，后续实现工作默认遵循以下要求：
1. **开发执行器必须优先使用 Codex 或 Claude Code**。
2. **执行权限默认采用当前环境可用的最高权限、全自动化模式**。
3. 开发与测试允许以**长时间独立执行**的方式推进。
4. 正式开发与测试期间，**原则上不要询问用户细节问题**，应优先参考现有文档自行闭环。
5. 如遇到文档中未覆盖的实现细节、技术问题、方案空白，应优先通过以下方式解决：
   - 查阅当前项目内现有文档
   - 在线搜索资料
   - 查找并参考 GitHub 上类似项目的设计与实现方案
6. 正式开发完成后，应统一整理并反馈：
   - 实际完成内容
   - 测试执行结果
   - 验收结论
7. 除非遇到真正阻断性问题或高风险破坏性操作，否则不要在开发过程中频繁打断用户。
> 注意：以上规则仅适用于主人确认后的正式开发/测试阶段；在当前设计和规划阶段，仍然以文档完善、说明确认和计划收敛为主。
### 2.7 当前公共接口测试站点强制规则（新增）
从现在开始，当前仓库在**公开接口自动化测试、底座改造验证、接口测试功能演示、回归示例设计**中，统一使用以下站点作为专用测试网站：
- `https://jsonplaceholder.typicode.com/`
执行要求：
1. 后续新增或改造的公开接口测试用例，默认必须基于该站点编写；
2. 设计测试方法时，必须遵循该站点公开说明中的资源范围、请求方法、查询参数过滤、嵌套路由和伪写入约束；
3. 对 `POST/PUT/PATCH/DELETE` 的测试，不得错误假设服务端会真实持久化数据，必须按其“模拟写入成功、返回伪响应”的规则设计断言；
4. 如需保留依赖私有环境的历史用例，必须继续与公共回归基线隔离，不能替代该站点的公开验证职责；
5. 若后续拟引入其他公开测试站点，必须先更新本文件与相关测试文档，再实施代码或测试改造。

## 3. 当前认可的产品与架构方向
### 3.1 平台核心方向
平台未来需要同时支持三条输入路线：
1. 文档驱动
2. 功能测试用例驱动
3. 抓包驱动
但开发上必须分阶段推进，当前 V1 优先聚焦：
- 底座改造
- 文档驱动最小闭环
- 中间模型雏形
- 模板体系雏形
- 规则体系雏形
- 执行闭环雏形
### 3.2 技术栈方向
当前架构方向已明确：
- 核心语言：**Python**
- 测试执行：**pytest + requests**
- 模板体系：优先考虑 **Jinja2**，但其职责限定为**代码生成模板引擎**，不再作为 Web 前端页面模板方案
- Web 服务层：**Django + Django REST Framework（DRF）**
- 数据存储：**MySQL**
- 前端：当前轮已确认重构为 **Vue3 + TypeScript**
- Web UI 形态：当前轮已确认采用**前后端分离设计**，前端独立工程负责页面与交互，Django + DRF 仅负责 API 与前端静态入口承接
- Desktop：采用 **前端壳 + 本地服务内核** 路线，当前保持 **Tauri 优先评估、Electron 备选**
如无明确批准，不要自行改成其他核心路线。

## 5. 后续开发必须遵守的工程原则
### 5.1 中间模型优先
后续所有生成能力，都必须优先围绕统一中间模型建设。
不允许：
- 输入文档直接拼 Python 文件
- 输入抓包直接拼 pytest 用例
- 输入功能测试用例直接自由生成最终代码
必须走：
> 输入 → 解析标准化 → 中间模型 → 模板/规则 → 生成资产 → 执行验证
### 5.2 模板化优先
后续所有代码生成都必须优先通过模板体系生成，包括但不限于：
- 接口方法
- 测试用例
- 断言
- 数据文件
- 生成记录
### 5.3 规则校验优先
所有生成结果都必须可被规则层检查，包括：
- 命名规范
- 模块归档
- 参数规范
- 引用关系
- 断言规范
- 目录结构
### 5.4 资产化优先
后续输出结果不是一次性代码，而是测试资产。
每一类生成物后续都应考虑：
- 来源
- 生成时间
- 审核状态
- 执行状态
- 可追溯性
### 5.5 可扩展优先
实现方案必须优先考虑：
- V2 的功能用例驱动
- V2 的抓包驱动
- Web 访问
- Windows 桌面应用
- 后续 macOS 扩展
不要只为了当前一个小用例写死结构。

## 6. 编码与改造要求
### 6.1 改造策略
当前阶段采用：
- **增量式改造**
- 先稳住底座
- 再接最小闭环
不建议：
- 一次性全量推翻原有框架
- 一次性做大重构而不留过渡层
### 6.2 文件职责要求
后续新建文件时，要尽量保持单一职责：
- 解析逻辑独立
- 模型定义独立
- 模板独立
- 规则独立
- 执行器独立
- 报告独立
### 6.3 命名要求
保持清晰、稳定、可预测：
- 类：`PascalCase`
- 函数 / 变量：`snake_case`
- 常量：`UPPER_SNAKE_CASE`
- 文档文件：中文名称可保留，但代码路径保持清晰
### 6.4 注释与配置说明强制要求（新增）
从现在开始，仓库内所有新增、修改和重构后的代码与配置说明，统一执行以下规则：
1. **所有注释必须使用中文**：
   - 包括行内注释、块注释、模块说明、类注释、方法注释、fixture 注释、测试用例说明等；
   - 不再新增英文注释；
   - 如引用第三方协议字段、标准术语或接口原始字段名，可保留英文原文，但解释文字必须使用中文。
2. **所有方法都必须包含中文注释**：
   - 包括普通函数、类方法、静态方法、pytest fixture、hook、CLI 入口方法、工具函数等；
   - 注释至少要说明“这个方法做什么”；
   - 对外暴露方法、核心流程方法和执行入口方法，必须进一步说明关键入参、返回值、异常/失败语义或调用约束；
   - 例如 `run_test.py` 这类执行入口文件，必须提供足够详细的中文注释，能够让后续维护者直接理解执行流程和参数作用。
3. **重要参数和关键配置必须有中文说明**：
   - 对公共方法、核心方法、复杂逻辑方法中的重要参数，必须通过中文注释或中文 docstring 明确其用途、取值约束和影响范围；
   - 对容易误用、会影响执行结果、环境切换、回归范围、代理行为、认证行为的参数，必须重点说明。
4. **JSON 配置文件必须提供中文参数注释**：
   - 由于标准 JSON 不支持原生注释，因此像 `api_config.json` 这类 JSON 配置文件，必须采用**与配置项同级或相邻的中文说明字段**来表达注释语义；
   - 注释字段命名应保持稳定、可预测，优先使用 `_comment`、`<字段名>_comment`、`section_comment` 这类明确形式；
   - 注释内容必须直接说明该配置项的用途、默认行为、可选值或使用限制；
   - 后续客户端、加载器和治理规则设计时，也必须兼容这类中文说明字段，不能因为增加注释能力而破坏配置读取。
5. **存量文件治理要求**：
   - 当前仓库内历史文件如果尚未满足上述规则，在后续重构、修复、迁移或功能调整时必须同步补齐中文注释；
   - 不允许继续新增“无中文注释的方法”或“无中文说明的关键配置项”；
   - 对存量重点文件应优先治理，包括但不限于：`run_test.py`、核心公共层、测试执行入口、配置文件、模板生成入口和规则校验入口。

## 7. 文档优先要求

## 8. README 与阶段说明书要求
### 8.1 README
README 不是简单项目介绍，而是当前仓库的统一说明入口。
后续如果变更平台方向、阶段目标、使用方式或结构，应同步更新 README。
### 8.2 阶段说明书
后续每个阶段（例如 V1）都必须有单独说明书，并且要包含：
- 目标范围
- 不做内容
- 详细设计方案
- 开发方案
- 测试方案
- 开发进度记录
- 测试进度记录
- 风险与阻塞记录
阶段说明书不是一次性文档，而是持续更新的执行文档。
### 8.3 文档同步强制规则（新增）
从现在开始，仓库内所有开发代理在**每次发生以下动作后**，都必须立即同步更新所有相关文档，再继续后续工作：
1. 修改任何代码文件；
2. 新增或调整测试用例；
3. 执行开发验证、回归测试或阶段验收测试；
4. 变更当前开发进度、测试结果、风险状态或未完成项判断。
执行要求：
- 不能等到一轮工作全部结束后再一次性补文档；
- 必须同步更新 README、阶段说明书、测试说明书以及与本次变更相关的其他文档；
- 文档中的能力描述、测试命令、测试结果、风险状态、未完成项必须与当前代码和验证结果保持一致；
- 如本次代码/测试变更尚未完成对应文档同步，则该轮工作不得视为完成。

## 9. Git 与提交要求
### 9.1 提交要求
每轮完成明确文档或代码改动后，应及时提交。
提交信息要求：
- **必须使用详细中文提交说明**
- 必须准确总结本次改动内容
- 必须说明影响范围和本次改动的目的
- 如涉及阶段推进，建议在描述中带上当前阶段背景（例如 V1 / 文档 / 架构 / 测试）
不再使用英文短提交作为默认风格。
示例：
- `文档：补充中间模型设计说明书，明确模型实体、字段与输入输出映射`
- `文档：更新总体架构设计与UI设计说明，确认Django+DRF与MySQL选型`
- `重构：拆分公共请求能力与工具层，准备承接统一中间模型与模板生成`
### 9.2 推送要求
推送远端 GitHub 属于阶段性动作，建议遵循以下原则：
- 文档阶段：当一轮关键文档体系更新完成后，及时推送；
- 开发阶段：当一轮实现与测试通过、可向主人汇报验收时，再推送；
- 不以“修改文件数”或“提交数”作为唯一推送标准。
