## 2. 调研目标
本次调研围绕以下两个方向展开：
1. **AI自动化测试平台**
   - 关注行业内成熟产品如何把 AI 引入测试设计、测试生成、执行、分析、治理等环节。
2. **AI驱动的接口自动化测试平台**
   - 关注 API / OpenAPI / Contract / 请求链路 / 抓包 / 代码生成等方向的具体实现思路。
本次调研的最终目标不是简单列举产品，而是提炼：
- 哪些设计是被市场反复验证过的；

- 哪些架构与实现思路适合沿用到本项目中。

## 3. 调研结论摘要（先看结论）
### 3.1 核心结论
当前赛道中做得较好的产品与项目，并不是把 AI 当作“自动写代码按钮”，而是普遍采用以下模式：
1. **以规范/上下文/现有资产为事实基础**
   - OpenAPI / Swagger / Postman Collection / 既有请求 / 历史测试资产 / 用户输入场景
2. **用 AI 做生成、补全、编排、解释、修复**
   - 不是完全凭空生成，而是在上下文约束下生成草稿
3. **生成后必须进入治理流程**
   - 审批、规则校验、执行验证、报告分析、自愈修复、回归复用
4. **平台化能力比“代码生成”更重要**
   - 真正成熟的产品都在做测试资产平台，而不是一次性代码生成器
5. **接口测试的高价值方向不只是单接口生成，而是场景链路与依赖推断**
   - create → query → update → delete 这类 producer-consumer 链路，决定平台上限
### 3.2 对本项目最重要的启发
当前项目未来应被定义为：
> **统一接口测试资产平台 + AI生成/编排能力 + 规则校验与执行验证闭环**
而不是：
> 仅是一个“AI 自动生成 requests + pytest 代码”的工具。
这一定义将直接影响后续的目录设计、核心模块划分、V1/V2规划与现有框架改造方式。

## 4. 外部产品调研
## 4.1 Postman Agent Mode
### 4.1.1 调研观察
从公开页面可观察到，Postman 的 Agent Mode 强调：
- 通过自然语言完成 API 探索、调试、修复、文档补全等任务；
- 支持把 request、folder、collection、workspace、Flow 等作为上下文输入给 AI；
- 支持 approval / auto-run；
- 强调与整个 API 生命周期深度集成，而不是孤立功能。
### 4.1.2 可借鉴设计
1. **上下文注入机制**
   - AI 不是裸跑，而是带着“具体请求 / 接口集合 / 工作空间”运行。
2. **审批与自动执行分离**
   - 高风险动作需要确认，低风险动作可加速执行。
3. **AI 贯穿 API 生命周期**
   - 不只生成测试，还参与调试、文档、探索、分析。
### 4.1.3 对本项目的借鉴建议
可直接借鉴到当前平台的设计：
- 设计统一“上下文包”输入机制；
- 设计“仅生成草稿 / 生成并预览 / 生成并执行”三段式操作；
- 后续 AI 不只用于生成代码，也用于解释、分析、修订建议。

## 5. GitHub / 开源项目调研
## 5.1 Schemathesis
### 5.1.1 项目定位
Schemathesis 是一个基于 OpenAPI / GraphQL schema 自动生成大量测试用例的工具，强调：
- 自动从 schema 生成测试；
- 基于 property-based testing / Hypothesis；
- 支持 stateful testing；
- 支持 CLI、Python 测试、CI 集成。
### 5.1.2 关键可借鉴点
1. **Schema-first 生成模式**
   - 以规范为输入，自动生成测试而不是手写。
2. **CLI + Python API 双入口**
   - 既可作为工具，也可嵌入已有测试体系。
3. **Stateful Testing 支持**
   - 不是只测单接口，而是支持 workflow。
4. **公共 API 与扩展点清晰**
   - 从其公开导出结构可看出，项目在配置、schema、generation、hooks、transport 等层面做了明确分层。
### 5.1.3 可直接沿用到本项目的方案
- 当前平台应采用 **Schema/Input → Intermediate Model → Code/Test Generation** 的标准化路线；
- 设计统一 CLI / Service API / Desktop 调用入口；
- 预留 hooks / check / serializer / auth / transport 等扩展位；
- 中间模型中应原生考虑 stateful workflow，而不是后补。
---
## 5.2 RESTler
### 5.2.1 项目定位
RESTler 是微软研究院推出的 **stateful REST API fuzzing** 工具，核心特点是：
- 输入 OpenAPI 规范；
- 推断 producer-consumer 依赖关系；
- 生成请求序列而非仅单请求；
- 支持 compile / test / fuzz-lean / fuzz 多模式；
- 强调智能探索深层服务状态。
### 5.2.2 关键可借鉴点
1. **Producer-Consumer 依赖推断**
   - 这是接口测试平台从“单接口生成器”进化为“场景测试平台”的核心能力。
2. **多阶段执行模式**
   - compile → test → fuzz-lean → fuzz
   - 非常适合本项目设计“生成 → 预检 → 轻量执行 → 深度验证”流程。
3. **Grammar / 中间表示层**
   - 先把 OpenAPI 编译成中间结构，再进入执行与探索。
4. **不同强度策略并存**
   - 不是所有场景都直接上最重模式，而是分层验证。
### 5.2.3 可直接沿用到本项目的方案
- 在平台内设计统一 **中间模型 / 测试语法层**；
- 设计接口依赖关系推断机制；
- 设计不同执行级别：
  - 语法级检查
  - 生成级检查
  - smoke 执行
  - 深度场景执行
- 把“场景链路编排”列为 V2 的重点能力。
---
## 5.3 EvoMaster
### 5.3.1 项目定位
EvoMaster 是自动生成系统级测试用例的开源工具，强调：
- AI-driven；
- 面向 REST / GraphQL / RPC 等接口；
- 可生成用于回归测试的小型有效测试套件；
- 输出不仅是发现问题，也包括可保留的测试资产；
- 默认生成交互式 Web 报告。
### 5.3.2 关键可借鉴点
1. **生成结果不是临时输入，而是回归资产**
2. **支持多协议，而不仅限于 REST**
3. **有报告与结果可视化意识**
4. **支持黑盒模式，从 Swagger 直接起步**
### 5.3.3 可直接沿用到本项目的方案
- 平台生成的测试资产必须可沉淀、可持续执行；
- 后续架构上建议预留 GraphQL / RPC 扩展位；
- Web / Desktop 形态应考虑执行报告可视化；
- V1 虽先聚焦 requests + pytest，但不要把模型写死成只支持单协议单场景。

## 5.4 OpenAPI Generator
### 5.4.1 项目定位
OpenAPI Generator 是成熟的 OpenAPI 代码生成项目，覆盖大量语言与目标类型。
### 5.4.2 关键可借鉴点
1. **模板驱动代码生成**
2. **生成器与模板解耦**
3. **大规模语言/场景扩展能力来自规范化模板体系**
4. **对不可信输入有明确安全提醒**
### 5.4.3 可直接沿用到本项目的方案
- 本项目生成接口方法 / pytest 用例时，不应直接拼字符串，而应建立模板体系；
- 模板应至少拆成：
  - 接口方法模板
  - 测试用例模板
  - 数据文件模板
  - 断言模板
  - 模块索引模板
- 对外部文档/抓包输入应有安全过滤和规则校验。

## 6. 综合分析：赛道中被验证有效的设计模式
### 6.1 设计模式一：规范/上下文优先，AI 补全其次
成熟产品与开源项目几乎都不是“纯自然语言直接生成所有测试”。
它们普遍采用：
- OpenAPI / Collection / 文档 / 既有请求 / 场景描述 作为事实基础；
- AI 在其上进行补全、推断、生成、修订。
**结论**：本项目必须坚持“事实输入优先，AI 补全次之”。
### 6.2 设计模式二：中间模型至关重要
无论是 RESTler 的 grammar，还是 Schemathesis 的 schema/operation/case 结构，本质都说明：
> 不应直接从输入跳到代码，而应先转成统一中间模型。
**结论**：本项目必须建设统一中间模型层。
### 6.3 设计模式三：场景链路能力决定平台上限
单接口生成并不难，真正难的是：
- 接口依赖推断；
- 动态变量提取；
- create/get/update/delete 链路拼接；
- 鉴权、上下文、数据清理；
- 业务状态转移验证。
**结论**：V2 必须重点建设场景链路与依赖机制。
### 6.4 设计模式四：代码生成必须模板化、规则化
成熟方案不会依赖一次性 prompt 输出最终资产，而是：
- 先做结构化；
- 再套模板；
- 再校验。
**结论**：当前项目要尽快建立模板层和规则层。
### 6.5 设计模式五：平台闭环比生成能力更重要
真正可落地的平台都具备：
- 生成
- 预览
- 审核
- 执行
- 报告
- 回归
- 修订
**结论**：本项目 V1 虽可先做最小闭环，但架构上必须从第一天考虑完整闭环。

## 7. 对当前项目的产品与架构建议
## 7.1 产品定位建议
建议将本项目定义为：
> **面向测试与开发人员的接口自动化测试资产平台，支持基于文档、功能用例、抓包等输入，借助 AI 生成和编排接口测试资产，并通过规则校验与执行验证实现持续沉淀与复用。**
---
## 7.2 核心能力分层建议
建议平台采用以下核心分层：
1. **输入接入层**
   - OpenAPI / Swagger / Markdown 文档输入
   - 功能测试用例输入
   - 抓包文件 / 代理数据输入
2. **解析标准化层**
   - 统一抽取接口、参数、返回结构、依赖关系、场景步骤
3. **中间模型层**
   - API Definition
   - API Operation
   - Test Scenario
   - Variable Binding
   - Assertion Candidate
4. **生成编排层**
   - 接口方法生成
   - 用例生成
   - 断言生成
   - 场景链路拼接
5. **模板与规则层**
   - 模板渲染
   - 命名规范
   - 模块归类
   - 参数与依赖校验
6. **资产管理层**
   - 接口资产
   - 用例资产
   - 数据资产
   - 生成记录
   - 审核记录
7. **执行与报告层**
   - pytest 执行
   - smoke / regression 分级
   - 报告生成与展示
8. **AI 协作层**
   - 提示编排
   - 生成解释
   - 修订建议
   - 差异对比
---
## 7.3 V1 建议范围
结合调研和当前项目现状，建议 V1 聚焦：
1. 现有 pytest 接口测试框架改造为“适合 AI 生成接入”的底座；
2. 建立统一目录规范、模板规范、命名规范；
3. 建立统一中间模型雏形；
4. 打通 **文档驱动最小闭环**：
   - 输入 OpenAPI / 基础接口文档
   - 生成接口方法骨架
   - 生成基础 pytest 用例骨架
   - 执行 smoke 级验证
5. 建立生成记录、预览确认、执行结果沉淀的基础能力；
6. 在架构上预留功能用例驱动与抓包驱动入口。
---

V2 建议重点扩展：

2. 抓包驱动生成；
3. 动态变量提取与引用；
4. producer-consumer 依赖识别；
5. 场景编排；
6. Web 使用能力完善；
7. Windows 应用封装增强；
8. 更强的审核、修订、报告与结果管理。

## 8. 可直接沿用到当前平台的关键原则
### 原则 1：事实来源优先级固定
建议采用：
- OpenAPI / 文档 / 抓包 / 功能测试用例 > AI 推断
### 原则 2：生成结果必须是“资产”，不是一次性输出
建议所有生成结果都带：
- 来源
- 时间
- 输入摘要
- 生成版本
- 审核状态
- 执行状态
### 原则 3：一切生成先走中间模型，再走模板
避免直接 prompt 输出生产代码。
### 原则 4：执行级别分层
建议至少支持：
- 结构校验
- 模板校验
- smoke 执行
- 深度执行
### 原则 5：专业用户优先，低门槛用户后置支持
当前先服务测试 / 开发用户最合理。
