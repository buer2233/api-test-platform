# UI设计说明文档
## 1. 文档说明
- **文档名称**：UI设计说明文档
- **所属项目**：接口自动化测试平台
- **文档目标**：明确平台 Web 前端与 Windows 客户端封装后的界面设计方向、交互层级、视觉风格、组件原则和实施要求，作为后续前端开发与客户端封装的设计依据。
- **当前阶段**：V1 设计阶段
- **适用范围**：Web 界面、Windows 客户端封装界面、后续前端实现阶段
---
## 2. UI 设计定位
UI 设计在本项目中不是附属工作，而是正式开发前置环节。
后续前端开发和 Windows 客户端封装开发，必须优先遵循本设计文档，再进入具体实现。
### 2.1 UI 设计的目标
1. 建立统一、专业、可持续扩展的平台型视觉风格；
2. 保证 Web 前端与 Windows 客户端封装后的体验一致；
3. 保证后续平台页面结构清晰、交互层级合理；
4. 保证前端界面满足主人对品质和风格的要求。

## 4. 设计原则
## 4.1 统一性原则
Web 前端与 Windows 客户端应使用统一设计语言，包括：
- 色彩体系
- 字体风格
- 间距体系
- 卡片/面板样式
- 按钮与输入控件样式
- 状态反馈样式
## 4.2 平台感原则
本项目是测试资产平台，不是普通宣传页或管理后台套壳页面。
界面要体现：
- 平台感
- 工具感
- 专业感
- 稳定感
## 4.3 Windows 客户端适配原则
虽然前端主要面向 Web 技术栈，但最终 Windows 客户端封装后，视觉上应具有：
- 桌面工具感
- 分层清晰
- 面板化布局
- 稳定、沉稳、专业的 UI 气质
## 4.4 高质量设计原则
后续实现时不允许走“普通后台模板风格”路线，不允许只是默认组件库堆叠。
应通过：
- 字体选择
- 层次布局
- 边框/阴影/圆角
- 配色与强调色
- 面板结构
来形成有辨识度的视觉系统。
## 4.5 可实施原则
UI 设计要足够具体，能直接指导 React 前端实现，而不是停留在空泛描述层面。
---
## 5. 当前推荐的视觉方向
结合产品定位、平台属性和截图参考，当前推荐的 UI 方向为：
> **专业平台型 + Windows 客户端工具感 + 深色中性基调 + 清晰分层 + 高级感细节强化**
### 5.1 关键词
- 专业
- 沉稳
- 深色
- 平台化
- 工具型
- 清晰分层
- 高质量桌面感
### 5.2 不建议的方向
1. 不建议使用过度轻浮、营销页式风格；
2. 不建议使用过度花哨、动画堆砌的炫技风格；
3. 不建议使用普通开源后台模板默认视觉；
4. 不建议使用廉价感强的渐变堆叠与随意配色。
---
## 6. 界面结构设计建议
## 6.1 总体布局建议
建议采用平台型三段式或两段式布局：
1. **左侧导航区**
   - 功能导航
   - 模块入口
   - 阶段任务入口
2. **顶部状态区**
   - 当前项目
   - 当前环境
   - 当前执行状态
   - 全局操作入口
3. **主工作区**
   - 文档输入
   - 生成预览
   - 场景配置
   - 执行结果
   - 报告查看
### 6.2 页面类型建议
后续至少会涉及：
1. 仪表盘 / 首页
2. 输入导入页（文档 / 用例 / 抓包）
3. 中间模型 / 结构预览页
4. 生成结果预览页
5. 执行结果与报告页
6. 设置与配置页
---
## 7. 组件设计原则
## 7.1 导航组件
要求：
- 层级清晰
- 选中状态明显
- 支持未来模块扩展
## 7.2 卡片 / 面板组件
要求：
- 具备桌面工具式面板感
- 层级明确
- 不宜过度轻飘
## 7.3 表单组件
要求：
- 适合复杂输入
- 支持多区域、分步骤填写
- 支持错误提示和校验反馈
## 7.4 结果展示组件
要求：
- 适合展示生成结果、断言、执行日志、报告摘要
- 强调可读性与信息层次
## 7.5 状态反馈组件
要求：
- 清晰表达：成功 / 失败 / 待审核 / 执行中 / 已完成 / 阻塞
---
## 8. 色彩与主题建议
## 8.1 当前推荐主题方向
建议优先采用：
- **深色主题优先**
- 中性灰/深蓝/石墨黑为主基调
- 通过少量高亮强调色突出关键状态与操作
## 8.2 设计理由
- 更符合平台工具感；
- 更贴近 Windows 客户端封装后的专业感；
- 更适合长时间使用；
- 与接口测试、执行日志、报告展示场景适配更好。
## 8.3 配色原则
- 主色调：深色中性
- 强调色：单一明确，不宜过多
- 状态色：成功/失败/警告/信息应统一规范
---
## 9. 字体与层次建议
## 9.1 字体原则
- 标题与正文需要有明显层级区分；
- 不使用廉价感强或过于普通的字体组合；
- 需兼顾 Web 与 Windows 客户端显示效果。
## 9.2 层级原则
- 页面标题 > 模块标题 > 区块标题 > 正文 > 辅助说明
- 层级要通过字号、字重、颜色和间距共同表达。
---
## 10. 动效与交互反馈建议
## 10.1 动效原则
动效应服务于：
- 状态变化
- 面板切换
- 数据加载
- hover / focus 反馈
而不应喧宾夺主。
## 10.2 当前建议
- 使用克制、稳定、专业的微交互；
- 避免过度夸张的动画；
- 更偏向桌面工具型流畅反馈，而不是营销页动画感。
---
## 11. 与技术选型的结合要求
## 11.1 前端实现
当前端建议技术栈为：
- React + TypeScript
### 要求
- UI 实现必须可组件化；
- 样式体系必须统一；
- 后续要适配 Web 和 Windows 客户端壳。
## 11.2 后端配合
当前后端确认：
- Django + DRF
### 要求
- 前端页面结构需要围绕平台业务流程设计，而不是围绕后端接口零散拼装。
## 11.3 桌面封装配合
当前桌面方向：
- 前端壳 + 本地服务内核
- Tauri 优先评估，Electron 备选
### 要求
- UI 设计要考虑桌面窗口环境中的边距、层次、区域组织与状态展示。
