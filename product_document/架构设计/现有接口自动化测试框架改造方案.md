# 现有接口自动化测试框架改造方案
## 1. 文档说明
- **文档名称**：现有接口自动化测试框架改造方案
- **所属项目**：接口自动化测试平台
- **目标**：明确当前 `api_test` 框架存在哪些结构性问题、为什么不适合直接承接 AI 生成能力，以及后续应如何分阶段改造，使其演进为平台化底座。
- **关联文档**：
  - `product_document/产品需求说明书(全局).md`
  - `product_document/架构设计/总体架构设计说明书.md`
  - `product_document/架构设计/中间模型设计说明书.md`
---
## 2. 当前框架现状概览
当前项目主体位于 `api_test/`。截至 2026-04-01，现状已进入通用框架重构阶段，当前同时存在“旧底座残留”和“新结构起步”两类内容：
1. `core/base_api.py`：已进一步收口为通用 HTTP 客户端，非 HTTP 工具已迁移到 `core/common_tools.py`。
2. `tests/`：已切换到 JSONPlaceholder 公开基线，并保留配置治理、运行时治理和公开示例测试。
3. `conftest.py`：当前主要承载公开基线 fixture 与 HTML 报告钩子。
4. `api_config.json` + `core/config_loader.py`：已形成唯一配置源和配置加载基础。
5. `run_test.py`：已作为正向公开基线执行入口。
6. 历史专用配置、目录桥接和账号文件已删除，不再作为通用框架组成部分。
从用途上看，它已经是一个“可以运行的接口自动化测试框架”，但还不是“可以承接 AI 生成、统一资产管理和平台化扩展”的测试平台底座。
---
## 3. 当前框架存在的核心问题
## 3.1 定位问题：更像手写测试框架，而不是平台底座
当前目录和代码组织更适合人工写少量测试，不适合：
- 批量生成接口方法；
- 批量生成测试用例；
- 接收文档驱动 / 抓包驱动 / 功能用例驱动的统一输入；
- 管理生成记录、审核记录与执行记录。
## 3.2 结构问题：缺少中间承接层
目前从项目结构看，没有专门的：
- 中间模型层
- 模板层
- 规则校验层
- 生成记录层
这意味着一旦接入 AI 生成，生成结果很容易直接落成 Python 文件，无法稳定治理。
## 3.3 责任问题：`base_api.py` 职责过重
历史上的 `base_api.py` 曾同时承担：
- HTTP 请求
- 加解密
- 时间工具
- 数据处理
问题在于：
- 公共职责边界不清晰；
- 难以按平台化方式扩展；
- AI 很难稳定理解“哪些能力属于请求层，哪些属于工具层”。
截至 2026-04-02，旧私有登录链路已删除，非 HTTP 工具已迁移到 `core/common_tools.py`，本问题已完成当前轮首个治理目标。
## 3.4 接口资产问题：接口封装结构不够标准化
历史业务接口封装文件虽然已删除，但通用接口资产模板和既有资产导入边界仍需继续标准化：
- 还没有统一接口资产模板；
- 缺少明确模块目录规范；
- 接口命名和归类未来可能越来越散；
- 不利于 AI 批量生成与维护。
## 3.5 用例资产问题：场景与断言体系未标准化
当前测试用例更偏示例，缺少：
- 场景模型统一表达；
- 断言模板统一表达；
- 动态变量引用规范；
- 前后置依赖与清理规范；
- 参数化数据结构规范。
## 3.6 配置问题：配置方式不利于平台化
历史 `config.py` 中存在明显的环境耦合问题：
- 基础地址硬编码；
- 服务映射硬编码；
- 账号与环境管理未平台化；
- 不利于多环境、多项目和后续 Web / Desktop 配置界面支持。
2026-04-01 已新增 `api_config.json` 与 `config_loader.py`，并已完成运行时引用迁移；`config.py` 已删除，当前配置收口已完成首轮落地。
同日已新增 `proxy.enabled` 与 `proxy.url` 两个配置字段，用于统一承接当前代理开关与代理地址；后续客户端应直接控制这两个字段，而不是新增独立代理配置源。本轮验证时曾临时将代理开关切为开启状态，用于验证对外请求连通性与框架改造结果，仓库默认值仍保持关闭。
## 3.7 资产追踪问题：缺少元数据和记录体系
当前项目对生成物、执行结果、来源信息缺少结构化记录：
- 没有生成记录
- 没有审核记录
- 没有统一执行记录模型
- 没有资产元信息跟踪
这会直接影响后续 AI 生成结果的可追溯性。
## 3.8 执行问题：执行入口较薄，不适合平台化调度
`run_test.py` 能运行 pytest，但还不够：
- 缺少执行级别划分；
- 缺少按模块/场景/来源执行能力；
- 缺少执行上下文记录；
- 缺少平台级反馈结构。
---
## 4. 当前框架与未来平台的差距
从“框架”到“平台”，至少还差以下层级：
1. **输入接入能力**
2. **解析标准化能力**
3. **统一中间模型能力**
4. **模板化生成能力**
5. **规则校验能力**
6. **测试资产管理能力**
7. **执行结果治理能力**
8. **报告分析能力**
9. **多端访问能力（Web / Desktop）**
---
## 5. 改造目标
本次改造不是为了“推倒重来”，而是为了让当前框架演进为可承接平台化能力的底座。
### 5.1 改造后的目标状态
改造后，当前框架应逐步具备：
1. 可承接统一中间模型输出；
2. 可承接模板化生成的接口资产；
3. 可承接模板化生成的测试用例资产；
4. 可通过规则层校验生成结果；
5. 可统一执行并输出结构化执行结果；
6. 可为 Web / Desktop / CLI 提供共用核心。
---
## 6. 改造总策略
## 6.1 总体策略
建议采用：
> **先改造底座，再接入文档驱动最小闭环**
原因：
- 当前项目已有可复用基础，不需要全量推翻；
- 若不先改造底座，AI 生成资产会直接堆入现有结构，后续很难治理；
- V1 重点是打稳承接能力，而不是追求一次生成很多代码。
## 6.2 改造方式
建议采用：
- **增量式改造**，而不是一次性重构全部代码；
- 优先新增标准结构；
- 再逐步迁移旧能力；
- 避免在当前阶段做过度清理式重构。
---
## 7. 改造方向详细说明
## 7.1 方向一：建立标准目录结构
### 当前问题
目录结构围绕手写测试展开，不适合平台能力扩展。
### 改造建议
后续逐步建立清晰目录边界，例如：
1. `models/`：中间模型
2. `parsers/`：文档/用例/抓包解析
3. `templates/`：模板体系
4. `rules/`：规则校验
5. `services/`：应用服务层
6. `executors/`：执行器
7. `assets/`：资产管理
8. `api_test/`：保留并逐步重构为核心测试资产目录
### 改造目标
让生成资产、平台能力、现有测试框架逐步解耦。
---
## 7.2 方向二：拆分 `base_api.py` 的职责
### 当前问题
`base_api.py` 历史上过于庞杂。2026-04-02 当前轮已完成第一步拆分：`BaseAPI` 只保留 HTTP 请求能力，`core/common_tools.py` 承接嵌套取值、时间、加解密和随机数据工具。
### 改造建议
逐步按职责拆分为：
1. 请求客户端层（当前轮已由 `base_api.py` 收口）
2. 会话与重试配置层（当前已由 `session.py` 承接）
3. 通用工具层（当前轮已由 `common_tools.py` 承接首轮）
4. 后续如工具继续增加，再细拆为时间、加解密和测试数据子模块
### 改造目标
- 降低复杂度；
- 提高 AI 理解与复用稳定性；
- 为模板化接口生成提供更清晰的目标结构。
---
## 7.3 方向三：建立标准接口资产模板
### 当前问题
接口封装缺少统一模板。
### 改造建议
统一接口资产规范，明确：
- 文件命名规范
- 类/函数命名规范
- 入参规范
- 返回值规范
- 公共调用入口
- 日志与异常规范
### 改造目标
使文档驱动生成接口方法时，有稳定模板可落。
---
## 7.4 方向四：建立标准测试用例模板
### 当前问题
当前用例更多是演示，不是平台级模板。
### 改造建议
建立统一测试用例模板，至少明确：
- 场景命名
- 场景步骤写法
- 公共断言入口
- 动态变量使用方式
- 参数化组织方式
- 前后置清理写法
### 改造目标
使功能用例驱动和抓包驱动后续能稳定生成 pytest 用例。
---
## 7.5 方向五：建立断言模板与断言层
### 当前问题
断言未标准化。
### 改造建议
建立：
- 状态码断言模板
- JSON 字段断言模板
- schema 断言模板
- 业务规则断言模板
### 改造目标
让 AI 补全断言时有规则可依，而不是自由发挥。
---
## 7.6 方向六：建立动态变量与依赖机制
### 当前问题
当前项目没有系统化的变量绑定与依赖机制。
### 改造建议
逐步建立：
- 提取变量规范
- 变量作用域规范
- 依赖关系表达规范
- 场景链路引用规范
### 改造目标
为后续场景编排和抓包驱动打基础。
---
## 7.7 方向七：建立生成记录与元数据机制
### 当前问题
生成资产不可追溯。
### 改造建议
所有后续生成物都应记录：
- 来源
- 生成时间
- 生成策略
- 审核状态
- 执行状态
### 改造目标
让平台具备资产级治理能力。
---
## 7.8 方向八：增强执行器与报告体系
### 当前问题
执行能力较薄。
### 改造建议
在现有 `run_test.py` 基础上，逐步增强：
- 分级执行
- 模块执行
- 场景执行
- 执行记录输出
- 报告归档
### 改造目标
使执行层可以支撑平台闭环，而不只是手工跑 pytest。
---
## 7.9 方向九：配置层平台化改造
### 当前问题
配置与环境绑定明显。
### 改造建议
逐步建立：
- 环境配置抽象
- 账号配置抽象
- 本地配置文件规范
- 后续 Web / Desktop 配置入口预留
### 改造目标
为多环境、多项目和本地应用封装提供支持。
---

## 9. V1 框架改造建议顺序
建议后续改造按以下顺序推进：
1. 先定义中间模型
2. 再定义模板体系
3. 再定义规则体系
4. 再规划目录结构
5. 再抽离公共基础能力
6. 再对接文档驱动最小闭环
7. 再完善执行与报告闭环
---
## 10. 对 Codex / 后续开发的直接要求
后续无论是人工开发还是 Codex 开发，都应遵循：
1. 不直接在旧结构上无规则堆代码；
2. 先按文档与架构落地新结构；
3. 所有新增能力优先考虑是否可被中间模型和模板层承接；
4. 所有变更要考虑未来 Web / Desktop 共用；
5. 不为了短期实现便利破坏平台化方向。
---
## 11. 结论
当前 `api_test` 可以继续作为基础底座，但必须改造为：
> **面向统一中间模型、模板化生成、规则校验和执行验证闭环的测试资产底座**

## 11.1 2026-04-17 主线重构当前实现进展
截至 2026-04-17，围绕 `api_test` 主线重构已先落地以下最小承接能力：
1. `scenario_service` 已补齐三段式主工作台壳层和子模块默认测试用例列表交互；
2. 已补齐三主题样式切换与主题偏好记录，确保主题仅切样式、不切结构；
3. 已新增模块级抓包前置过滤能力 `CaptureProxyFilter`；
4. 已新增抓包会话落库对象 `CaptureProxyRecord`；
5. 已新增抓包治理候选构建能力 `CaptureCandidateBuilder`，先完成静态资源过滤和最小候选输出；
6. 已新增 `ApiTestMethodRegistry`，先完成按 `HTTP 方法 + 完整路径` 的已有接口方法命中能力；
7. 已在服务层补齐抓包候选的 `reused / parameter_completion_required / create_required` 状态标注入口；
8. 已新增 `build_asset_paths()`，固化 `api_test/core/<project>/<model>/` 与 `api_test/tests/<project>/<model>/` 的目录落点；
9. 已新增 `build_testcase_steps()`，保证测试用例方法链严格保持抓包选择顺序；
10. 已新增 `render_testcase_module()`，先生成符合 Allure 规范的最小测试模块；
11. 已新增 `write_generated_assets()` 与 `evaluate_generation_gate()`，分别承接文件落盘和 `pytest` 提交门禁；
12. 已新增 `GenerationJobRecord`，为后续生成任务记录、提交状态和审计回写预留最小事实层。

当前仍未完成的后续主线项包括：
1. `api_test` 目录扫描与接口方法识别；
2. 复用 / 补参 / 新增三种匹配状态；
3. 接口方法与测试用例落盘；
4. `pytest + allure` 生成门禁；
5. 失败重试与报告回看闭环。

当前主线的固定目录口径与治理要求应保持为：
1. 核心接口方法文件落到 `api_test/core/<project>/<model>/`
2. 核心测试用例文件落到 `api_test/tests/<project>/<model>/`
3. 抓包阶段必须支持 URL / IP `抓包前置过滤`
4. 过滤后只记录命中的抓包数据，不做“先全量抓、后列表过滤”
5. 生成测试代码统一遵守 `allure.feature / allure.story / allure.step` 规范
如果这一步不先做，后续 AI 生成能力越强，项目结构会越快失控。
因此，框架改造不是附属工作，而是平台 V1 的核心前置工作之一。
