# 中间模型设计说明书
## 1. 文档说明
- **文档名称**：中间模型设计说明书
- **所属项目**：接口自动化测试平台
- **文档目标**：定义平台统一中间模型（Unified Intermediate Model）的设计原则、模型结构、字段建议、关系约束、输入输出映射与后续扩展策略。
- **适用阶段**：V1 设计阶段 / V2 扩展设计基础
- **关联文档**：
  - `product_document/产品需求说明书(全局).md`
  - `product_document/架构设计/总体架构设计说明书.md`
---
## 2. 为什么必须建设中间模型
平台未来需要同时支持：
1. 接口文档驱动
2. 功能测试用例驱动
3. 抓包驱动
如果三条路线分别直接产出代码，会产生以下问题：
- 生成结果风格不一致；
- 审核与校验方式无法统一；
- Web / Desktop / CLI 无法共享核心逻辑；
- 后续修改成本极高；
- AI 输出结果难以被结构化复用；
- 动态变量、依赖关系、断言候选无法统一管理。
因此，平台必须建立统一中间模型层，作为所有输入、生成、校验、执行和沉淀的共同基础。
---
## 3. 中间模型在平台中的定位
中间模型位于：
- 输入接入层
- 解析标准化层
- 生成编排层
之间的核心位置。
### 3.1 核心作用
1. 承接不同来源输入
2. 统一表达接口、场景、依赖、断言和元数据
3. 为模板生成提供稳定输入
4. 为规则校验提供统一检查对象
5. 为执行验证提供可编排的结构化对象
6. 为资产沉淀提供可追溯元信息
### 3.2 设计要求
中间模型必须满足：
- 结构化
- 可序列化
- 可扩展
- 可追溯
- 可被 AI、模板、规则、执行层共同消费
---
## 4. 中间模型设计原则
## 4.1 事实优先原则
中间模型中的字段优先来源于：
- OpenAPI / Swagger / 文档事实
- 功能测试用例事实
- 抓包事实
- 已有接口资产事实
AI 推断只作为补充信息，不应覆盖事实输入。
## 4.2 最小可用原则
V1 只设计并落地平台当前真正需要的模型字段，不为了未来可能支持的所有协议一次性堆满字段。
## 4.3 分层解耦原则
- 解析层负责把输入转换为模型；
- 模型层负责统一表达；
- 模板层负责消费模型生成代码；
- 规则层负责校验模型与产物；
- 执行层负责基于模型编排测试。
## 4.4 稳定标识原则
中间模型中的核心对象需要有稳定 ID / key，用于：
- 生成记录追踪
- 审核状态追踪
- 再生成
- 资产更新
- 差异对比
## 4.5 元信息必备原则
所有核心对象都应允许挂载 metadata，不允许只保留业务字段而丢失来源与状态。
---
## 5. 中间模型总体分层
建议将中间模型拆为以下几类：
1. **源信息模型（Source Models）**
2. **接口资产模型（API Asset Models）**
3. **场景模型（Scenario Models）**
4. **变量与依赖模型（Variable & Dependency Models）**
5. **断言模型（Assertion Models）**
6. **生成与审核模型（Generation & Review Models）**
7. **执行与报告引用模型（Execution Reference Models）**
---
## 6. 核心模型定义建议
## 6.1 SourceDocument（输入源文档模型）
### 6.1.1 用途
统一记录输入源。
### 6.1.2 建议字段
- `source_id`：输入源唯一标识
- `source_type`：输入类型
  - `openapi`
  - `swagger`
  - `markdown_doc`
  - `functional_case`
  - `traffic_capture`
  - `manual_input`
- `source_name`
- `source_path`
- `source_version`
- `source_summary`
- `imported_at`
- `imported_by`
- `raw_reference`
### 6.1.3 设计说明
- 所有解析结果都应能回溯到 SourceDocument。
---
## 6.2 ApiModule（接口模块模型）
### 6.2.1 用途
描述接口资产的模块归属。
### 6.2.2 建议字段
- `module_id`
- `module_name`
- `module_code`
- `module_path_hint`
- `module_type`
  - `api`
  - `bs`
  - `app`
  - `common`
- `module_desc`
- `source_ids`
- `tags`
### 6.2.3 设计说明
- 用于解决接口按模块归类的问题；
- 后续可映射到代码目录结构。
---
## 6.3 ApiOperation（接口操作模型）
### 6.3.1 用途
是中间模型中的核心对象，描述一个具体接口。
### 6.3.2 建议字段
- `operation_id`
- `module_id`
- `operation_name`
- `operation_code`
- `http_method`
- `path`
- `summary`
- `description`
- `tags`
- `auth_type`
- `request_headers`
- `path_params`
- `query_params`
- `body_schema`
- `content_type`
- `response_schemas`
- `success_codes`
- `error_codes`
- `source_ids`
- `confidence_score`
- `metadata`
### 6.3.3 设计说明
- 这是模板生成接口方法的核心输入对象；
- 需要支持部分字段“事实输入 + AI 补全”共存。
---

## 6.5 ResponseField（响应字段模型）
### 6.5.1 用途

### 6.5.2 建议字段

### 6.5.3 设计说明

---
## 6.6 VariableBinding（变量绑定模型）
### 6.6.1 用途
描述动态变量如何从响应中提取，并在后续步骤中复用。
### 6.6.2 建议字段

- `variable_name`
- `source_operation_id`
- `source_field_path`

- `target_scope`
  - `step`
  - `scenario`
  - `module`
  - `global`
- `target_operations`
- `fallback_value`
- `required`
- `metadata`
### 6.6.3 设计说明
- 这是场景链路能力的基础；
- 未来功能用例驱动和抓包驱动都高度依赖该模型。
---
## 6.7 DependencyLink（依赖关系模型）
### 6.7.1 用途
表示接口之间的 producer-consumer 关系。
### 6.7.2 建议字段
- `dependency_id`
- `upstream_operation_id`
- `downstream_operation_id`
- `dependency_type`
  - `id_dependency`
  - `token_dependency`
  - `business_state_dependency`
  - `resource_lifecycle_dependency`
- `binding_id`
- `required`
- `confidence_score`
- `source`
### 6.7.3 设计说明
- 该模型后续是场景自动拼接的重要基础；
- 支持来源于文档、抓包和 AI 推断。
---
## 6.8 AssertionCandidate（断言候选模型）
### 6.8.1 用途
描述平台建议生成的断言。
### 6.8.2 建议字段
- `assertion_id`
- `operation_id` 或 `scenario_step_id`
- `assertion_type`
  - `status_code`
  - `json_field_equals`
  - `json_field_exists`
  - `schema_match`
  - `business_rule`
- `target_path`
- `expected_value`
- `priority`
- `source`
- `confidence_score`
- `review_status`
### 6.8.3 设计说明
- 断言候选不等于最终断言实现；
- 后续可经过审核或 AI 修订再进入模板层。
- V1 当前轮中，`business_rule` 已落地两类最小语义：使用 `target_path + expected_value={"rule_code":"non_empty_string"}` 表达“目标字段必须为非空字符串”，使用 `target_path + expected_value={"rule_code":"positive_integer"}` 表达“目标字段必须为正整数”；
- 当前 `business_rule` 仅支持手工构造候选进入模板层和规则层，不由解析器自动推断生成。
---

## 7. 模型关系设计
建议的核心关系如下：
1. `SourceDocument` → 提供输入来源
2. `ApiModule` → 包含多个 `ApiOperation`
3. `ApiOperation` → 包含多个 `ApiParam`
4. `ApiOperation` → 包含多个 `ResponseField`
5. `VariableBinding` → 从 `ApiOperation` 或 `ResponseField` 提取变量
6. `DependencyLink` → 连接不同 `ApiOperation`
7. `TestScenario` → 包含多个 `ScenarioStep`
8. `ScenarioStep` → 引用 `ApiOperation`、`VariableBinding`、`AssertionCandidate`
9. `GenerationRecord` / `ReviewRecord` / `ExecutionRecord` → 关联任意核心资产对象
10. `RouteCapabilitySummary` / `ServiceCapabilitySnapshot` / `DocumentPipelineRunSummary` / `WorkspaceAssetInventorySummary` / `WorkspaceInspectionSummary` → 作为服务层对外摘要对象，消费核心模型与执行结果

## 8. 三条输入路线到中间模型的映射
## 8.1 文档驱动映射
### 输入
- OpenAPI / Swagger / Markdown 接口文档
### 映射优先产物
- `SourceDocument`
- `ApiModule`
- `ApiOperation`
- `ApiParam`
- `ResponseField`
- 初步 `AssertionCandidate`
### 特点
- 结构信息最强
- 场景信息较弱
- 依赖关系可能部分缺失
---
## 8.2 功能测试用例驱动映射
### 输入
- 功能测试用例
- 场景步骤说明
### 映射优先产物
- `SourceDocument`
- `TestScenario`
- `ScenarioStep`
- `AssertionCandidate`
- 补充 `DependencyLink`
### 特点
- 场景信息强
- 接口结构依赖已有接口资产或 AI 补全
---
## 8.3 抓包驱动映射
### 输入
- 流量记录
- 请求顺序
- 响应数据
### 映射优先产物
- `SourceDocument`
- `ApiOperation`
- `ApiParam`
- `VariableBinding`
- `DependencyLink`
- `TestScenario`
- `ScenarioStep`
### 特点
- 真值性较强
- 噪声多
- 需要清洗与归一化
---

## 10. V1 中间模型落地范围建议
V1 不要求一次性把所有模型都完全落地，但建议至少优先落地：
1. `SourceDocument`
2. `ApiModule`
3. `ApiOperation`
4. `ApiParam`
5. `ResponseField`
6. `AssertionCandidate`
7. `GenerationRecord`
8. `ExecutionRecord`
### V1 暂以“最小支持”形式预留的模型
1. `VariableBinding`
2. `DependencyLink`
3. `TestScenario`
4. `ScenarioStep`
5. `ReviewRecord`
原因：
- V1 重点还是文档驱动最小闭环；
- 场景编排与抓包驱动更多在 V2 逐步增强。

## 11. 技术实现建议
## 11.1 实现语言
建议继续采用 Python 定义中间模型。
## 11.2 模型实现方式
建议优先使用：
- `pydantic` 或同类数据模型机制
### 原因
1. 便于字段校验
2. 便于 JSON 序列化
3. 便于 Django / DRF 服务层复用
4. 便于后续模型版本演进
## 11.3 序列化策略
建议支持：
- JSON
- 本地文件持久化（用于临时中间产物或调试）
- 后续与 MySQL 持久化结构衔接
---
## 12. 风险与控制
## 12.1 风险一：模型设计过重
控制：
- V1 只落地最小必需字段。
## 12.2 风险二：模型设计过轻
控制：
- 至少保证接口、参数、响应、断言、元数据五类核心能力。
## 12.3 风险三：模型与模板强耦合
控制：
- 模型只表达结构，不内嵌模板逻辑。
## 12.4 风险四：模型字段来源不清晰
控制：
- 所有核心对象保留 source / confidence / metadata。

## 14. 结论
统一中间模型是该平台的核心设计枢纽。
如果没有中间模型，平台会退化成多个零散生成器；
