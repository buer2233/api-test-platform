# 详细测试用例说明书（V1）
## 1. 文档说明
- **文档名称**：详细测试用例说明书（V1）
- **所属项目**：接口自动化测试平台
- **文档目标**：定义 V1 阶段需要覆盖的详细测试用例，用于指导后续 TDD 开发、测试实现、阶段验证与验收。
- **适用范围**：V1 范围内的架构底座、中间模型、模板体系、规则体系、文档驱动最小闭环、执行闭环
- **文档状态**：第一版（持续维护）
- **关联文档**：
  - `product_document/产品需求说明书(全局).md`
  - `product_document/架构设计/总体架构设计说明书.md`
  - `product_document/架构设计/中间模型设计说明书.md`
  - `product_document/架构设计/现有接口自动化测试框架改造方案.md`
  - `product_document/阶段文档/V1阶段工作计划文档.md`
  - `product_document/架构设计/模板体系与代码生成规范说明书.md`
---
## 2. V1 测试目标
V1 测试的目标不是验证最终完整产品，而是验证 V1 最小闭环是否成立。
V1 测试重点验证以下内容：
1. 说明书约束是否可落地；
2. 中间模型是否能稳定承接文档输入；
3. 模板体系是否能生成稳定资产；
4. 规则体系是否能发现基础不合规问题；
5. 当前框架是否能承接生成资产，并能基于统一公开测试站点完成接口验证；
6. pytest 执行最小闭环是否成立；
7. 生成记录与执行记录是否具备最小可追溯能力。
---
## 2.1 当前公开接口测试基线站点
当前 V1 阶段涉及公开接口自动化测试、底座验证示例和回归用例设计时，统一使用：
- `https://jsonplaceholder.typicode.com/`
测试设计必须遵循其公开 Guide 约束：
1. 资源范围固定在 `posts/comments/albums/photos/todos/users` 等示例资源；
2. 支持 `GET/POST/PUT/PATCH/DELETE`；
3. 支持查询参数过滤与一层嵌套路由；
4. `POST/PUT/PATCH/DELETE` 为伪写入，不做真实持久化，断言应验证返回契约而不是持久化结果。
---
## 3. 测试设计原则
## 3.1 说明书驱动原则
所有测试用例必须可追溯到产品说明书与 V1 说明书中的明确目标。
## 3.2 TDD 原则
后续实现阶段必须优先按本说明书中的测试项编写测试，再进行实现。
## 3.3 最小闭环优先原则
V1 优先保证最小闭环成立，而不是追求覆盖未来所有能力。
## 3.4 分层测试原则
测试用例应覆盖：
- 文档/设计层校验
- 模型层
- 模板层
- 规则层
- 执行层
- 集成闭环层
## 3.5 可维护原则
测试用例说明应清晰、编号稳定、便于阶段更新。
---
## 4. 测试分层结构
建议 V1 测试分为以下六层：
1. **文档约束测试**
2. **中间模型测试**
3. **模板生成测试**
4. **规则校验测试**
5. **执行与记录测试**
6. **最小闭环集成测试**
---
## 5. 文档约束测试用例
## 5.1 目标
验证 V1 的设计输出是否仍然符合当前已确认的产品边界与架构原则。
### TC-V1-DOC-001
- **名称**：V1 范围不应越界到完整三路线实现
- **目标**：验证 V1 设计与实现计划不包含“一次性完整实现文档驱动、功能用例驱动、抓包驱动全部成品能力”的越界内容。
- **前置条件**：已有 V1 范围说明书
- **检查点**：
 1. V1 范围中明确“文档驱动最小闭环”为主线；
 2. 功能用例驱动与抓包驱动列为后续增强方向；
 3. 不将 V1 宣称为最终产品成品版。
- **预期结果**：通过。
### TC-V1-DOC-002
- **名称**：V1 必须围绕统一中间模型设计
- **目标**：验证当前设计未绕过中间模型直接进入生成实现。
- **检查点**：
 1. 中间模型设计文档存在；
 2. 相关架构说明中已明确“输入 → 中间模型 → 模板/规则 → 生成资产”的路径。
- **预期结果**：通过。
### TC-V1-DOC-003
- **名称**：V1 必须采用模板化生成策略
- **目标**：验证平台没有把自由文本生成直接定义为最终落地方案。
- **检查点**：
 1. 模板体系文档存在；
 2. 全局产品说明书中已明确模板生成原则；
 3. AGENTS.md 中已明确禁止直接自由生成最终生产代码。
- **预期结果**：通过。
### TC-V1-DOC-004
- **名称**：公开接口测试基线必须统一使用 JSONPlaceholder
- **目标**：验证当前公开接口自动化测试、底座演示和回归示例未继续分散到多个临时站点。
- **检查点**：
 1. `AGENTS.md` 中已明确 JSONPlaceholder 为当前专用公开测试站点；
 2. `api_test` 公开示例和相关测试文档已切换到该站点；
 3. 写操作断言未错误假设持久化成功。
- **预期结果**：通过。
### TC-V1-DOC-005
- **名称**：`api_test` 公开示例应符合 JSONPlaceholder Guide 的测试方法
- **目标**：验证底座公开示例已覆盖该站点允许的核心测试路径，而不是继续停留在无关公开网站。
- **检查点**：
 1. 至少覆盖单资源读取、过滤查询、嵌套路由；
 2. 至少覆盖 `POST/PUT/PATCH/DELETE` 的伪写入契约；
 3. 底座请求层可兼容 `201` 等标准 REST 状态码。
- **预期结果**：通过。
### TC-V1-DOC-006
- **名称**：`api_test` 公开测试底座应提供可复用的 JSONPlaceholder 资源与 fixture
- **目标**：验证公开测试不只是零散示例，而是已经具备复用的资源级 API 封装和公共 fixture。
- **检查点**：
 1. 存在可复用的 JSONPlaceholder API 封装；
 2. 存在公共 fixture 供多文件复用；
 3. 至少覆盖 users 和 todos 两类资源。
- **预期结果**：通过。

## 7. 模板生成测试用例
## 7.1 目标
验证模板体系能够根据中间模型稳定生成统一风格的资产。
### TC-V1-TPL-001
- **名称**：接口方法模板应能生成标准接口代码骨架
- **目标**：验证单个 ApiOperation 可生成统一风格的接口方法代码。
- **输入**：一个完整的 ApiOperation 模型实例
- **检查点**：
 1. 文件内容包含统一 import；
 2. 生成的方法命名符合规则；
 3. 请求方法与路径正确；
 4. 参数组织符合模板约定；
 5. 返回结构符合模板约定。
- **预期结果**：通过。
### TC-V1-TPL-002
- **名称**：pytest 用例模板应能生成基础测试骨架
- **目标**：验证接口测试骨架可被模板稳定输出。
- **输入**：一个 ApiOperation + AssertionCandidate
- **检查点**：
 1. 生成 `test_*.py` 风格文件；
 2. 包含基础 fixture 引用；
 3. 包含接口调用示例；
 4. 包含状态码断言；
 5. 包含基础响应断言占位。
- **预期结果**：通过。
### TC-V1-TPL-003
- **名称**：断言模板应能生成统一断言片段
- **目标**：验证状态码/字段断言能被模板统一输出。
- **输入**：不同类型的 AssertionCandidate
- **检查点**：
 1. 状态码断言格式统一；
 2. JSON 字段断言格式统一；
 3. 断言错误信息表达清晰。
- **预期结果**：通过。
### TC-V1-TPL-004
- **名称**：生成记录模板应能生成最小记录结构
- **目标**：验证生成记录可被稳定落盘。
- **检查点**：
 1. 可记录来源；
 2. 可记录生成时间；
 3. 可记录目标资产路径；
 4. 可记录审核状态占位。
- **预期结果**：通过。
### TC-V1-TPL-005
- **名称**：响应示例应可生成 `json_field_equals` 断言模板片段
- **目标**：验证模板层能够根据响应示例值输出稳定的 JSON 字段相等断言。
- **输入**：包含响应字段示例值的 `AssertionCandidate(assertion_type=json_field_equals)`。
- **检查点**：
 1. 断言片段可正确读取嵌套 JSON 路径；
 2. 断言值与输入示例保持一致；
 3. 渲染结果可直接嵌入 pytest 测试骨架。
- **预期结果**：通过。

## 8. 规则校验测试用例
## 8.1 目标
验证基础规则体系能发现明显不合规生成结果。
### TC-V1-RULE-001
- **名称**：接口方法命名不符合规范时应被识别
- **目标**：验证规则层可检查接口方法命名。
- **输入**：不符合命名规范的接口方法定义
- **预期结果**：规则校验失败，返回命名不合规提示。
### TC-V1-RULE-002
- **名称**：测试文件命名不符合规范时应被识别
- **目标**：验证测试文件命名规则生效。
- **输入**：非 `test_*.py` 文件名
- **预期结果**：规则校验失败。
### TC-V1-RULE-003
- **名称**：模块归类缺失时应被识别
- **目标**：验证模块归档规则生效。
- **输入**：缺失 module_id / module_name 的操作对象
- **预期结果**：规则校验失败。
### TC-V1-RULE-004
- **名称**：接口核心字段缺失时应被识别
- **目标**：验证接口定义必要字段检查生效。
- **输入**：缺少 method/path 的操作对象
- **预期结果**：规则校验失败。
### TC-V1-RULE-005
- **名称**：生成记录缺少基础追溯字段时应被识别
- **目标**：验证 `GenerationRecord` 至少具备 `source_ids`、`target_asset_path`、`template_reference` 和 `generation_version`。
- **输入**：缺失上述字段之一的生成记录对象。
- **预期结果**：规则校验失败，并返回缺失字段提示。
### TC-V1-RULE-006
- **名称**：AI 辅助生成记录缺少 `prompt_reference` 时应被识别
- **目标**：验证 AI 辅助生成结果不会绕过最小追溯要求。
- **输入**：`generator_type=ai_assisted` 且未设置 `prompt_reference` 的生成记录对象。
- **预期结果**：规则校验失败，并返回 `prompt_reference` 缺失提示。
### TC-V1-RULE-007
- **名称**：可执行接口缺少 `status_code` 断言时应被识别
- **目标**：验证执行前的断言集合至少包含基础状态码校验。
- **输入**：包含 `success_codes` 但断言列表中无 `status_code` 的接口对象。
- **预期结果**：规则校验失败。
### TC-V1-RULE-008
- **名称**：资产清单缺少基础关联字段时应被识别
- **目标**：验证 `asset_manifest` 至少具备资产列表、生成记录关联、执行记录关联与报告路径。
- **输入**：缺失 `assets`、`generation_ids`、`execution_id` 或 `report_path` 的资产清单对象。
- **预期结果**：规则校验失败。
### TC-V1-RULE-009
- **名称**：既有接口资产模块命名或标记不合规时应被识别
- **目标**：验证 `existing_api_asset` 的模块目录不会绕过最小平台规则。
- **输入**：`module_code` 非 snake_case 或缺失 `legacy_public_api` 标签的既有接口模块对象。
- **预期结果**：规则校验失败，并返回模块命名或标签缺失提示。
### TC-V1-RULE-010
- **名称**：既有接口资产操作缺少私有环境或响应模式元数据时应被识别
- **目标**：验证旧接口操作目录必须显式声明私有环境依赖和响应模式。
- **输入**：缺失 `requires_private_env`、`private_env` 标签，或 `response_mode` / `source_layer` 非法的既有接口操作对象。
- **预期结果**：规则校验失败，并返回对应元数据违规提示。

## 9. 执行与记录测试用例
## 9.1 目标
验证生成资产能进入最小执行闭环，并保留执行记录。
### TC-V1-EXEC-001
- **名称**：生成后的基础 pytest 用例应能被 pytest 识别
- **目标**：验证模板生成的测试骨架至少符合 pytest 基本识别规则。
- **预期结果**：pytest 可正常收集该用例。
### TC-V1-EXEC-002
- **名称**：基础 smoke 级执行应可完成
- **目标**：验证 V1 的最小执行链路成立。
- **预期结果**：可调用 pytest 执行；能输出通过/失败结果。
### TC-V1-EXEC-003
- **名称**：执行后应生成基础执行记录
- **目标**：验证执行行为可被记录。
- **检查点**：
 1. 记录执行时间；
 2. 记录执行结果；
 3. 记录报告路径或结果摘要。
- **预期结果**：通过。
### TC-V1-EXEC-004
- **名称**：目录级执行应覆盖全部生成测试
- **目标**：验证批量生成多个接口测试时，执行器不会只运行首个测试文件。
- **输入**：包含多个接口操作的 OpenAPI / Swagger 文档。
- **检查点**：
 1. 执行目标为生成测试目录；
 2. junit 报告包含全部生成测试数量；
 3. 执行记录 `target_id` 为 `generated-suite`。
- **预期结果**：通过。
### TC-V1-EXEC-005
- **名称**：CLI 入口应可直接执行文档驱动最小闭环
- **目标**：验证命令行入口可触发解析、生成和执行，并输出执行摘要。
- **输入**：一份本地 OpenAPI / Swagger 文档和输出目录。
- **检查点**：
 1. 进程返回码与执行结果一致；
 2. 标准输出包含 source、modules、operations、execution_target、asset_manifest_path 等摘要；
 3. 输出目录生成接口、测试和记录文件。
- **预期结果**：通过。
### TC-V1-EXEC-006
- **名称**：闭环执行后应生成最小资产清单
- **目标**：验证生成资产不是一次性代码输出，而是具备来源和摘要信息的最小资产集合。
- **输入**：一份可执行的 OpenAPI / Swagger 文档。
- **检查点**：
 1. 生成 `asset_manifest.json`；
 2. 清单中包含 source、generation、execution 关联信息；
 3. 清单中包含资产类型、文件路径和内容摘要。
- **预期结果**：通过。
### TC-V1-EXEC-007
- **名称**：V1 应用服务层应阻止未来路线越界执行
- **目标**：验证 V1 当前仅开放文档驱动闭环，不允许功能用例驱动和抓包驱动直接进入实现。
- **输入**：调用 `functional_case` 或 `traffic_capture` 路由的服务层请求。
- **检查点**：
 1. `document` 路由保持可执行；
 2. `functional_case` 与 `traffic_capture` 返回明确阻止结果；
 3. 阻止信息可用于阶段性范围验收。
- **预期结果**：通过。
### TC-V1-EXEC-008
- **名称**：应用服务层应可检查既有工作区资产清单
- **目标**：验证 V1 最小资产管理边界不仅能落盘，还能通过服务层读取并返回检查结果摘要。
- **输入**：一份已生成并执行过的本地工作区。

- **预期结果**：通过。
### TC-V1-EXEC-009
- **名称**：CLI inspect 应可输出工作区资产检查摘要
- **目标**：验证命令行可直接读取既有工作区并输出结构化检查结果。
- **输入**：一个已生成资产清单的工作区目录。

 1. 返回码与检查结果一致；
