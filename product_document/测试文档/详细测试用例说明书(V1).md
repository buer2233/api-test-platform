# 详细测试用例说明书（V1）
## 1. 文档说明
- **文档名称**：详细测试用例说明书（V1）
- **所属项目**：接口自动化测试平台
- **文档目标**：定义 V1 阶段需要覆盖的详细测试用例，用于指导后续 TDD 开发、测试实现、阶段验证与验收。
- **适用范围**：V1 范围内的架构底座、中间模型、模板体系、规则体系、文档驱动最小闭环、执行闭环
- **文档状态**：第一版（持续维护）
- **关联文档**：
  - `product_document/产品需求说明书(全局).md`
  - `product_document/架构设计/总体架构设计说明书.md`
  - `product_document/架构设计/中间模型设计说明书.md`
  - `product_document/架构设计/现有接口自动化测试框架改造方案.md`
  - `product_document/阶段文档/V1范围与最小闭环说明书.md`
  - `product_document/架构设计/模板体系与代码生成规范说明书.md`
---
## 2. V1 测试目标
V1 测试的目标不是验证最终完整产品，而是验证 V1 最小闭环是否成立。
V1 测试重点验证以下内容：
1. 说明书约束是否可落地；
2. 中间模型是否能稳定承接文档输入；
3. 模板体系是否能生成稳定资产；
4. 规则体系是否能发现基础不合规问题；
5. 当前框架是否能承接生成资产；
6. pytest 执行最小闭环是否成立；
7. 生成记录与执行记录是否具备最小可追溯能力。
---
## 3. 测试设计原则
## 3.1 说明书驱动原则
所有测试用例必须可追溯到产品说明书与 V1 说明书中的明确目标。
## 3.2 TDD 原则
后续实现阶段必须优先按本说明书中的测试项编写测试，再进行实现。
## 3.3 最小闭环优先原则
V1 优先保证最小闭环成立，而不是追求覆盖未来所有能力。
## 3.4 分层测试原则
测试用例应覆盖：
- 文档/设计层校验
- 模型层
- 模板层
- 规则层
- 执行层
- 集成闭环层
## 3.5 可维护原则
测试用例说明应清晰、编号稳定、便于阶段更新。
---
## 4. 测试分层结构
建议 V1 测试分为以下六层：
1. **文档约束测试**
2. **中间模型测试**
3. **模板生成测试**
4. **规则校验测试**
5. **执行与记录测试**
6. **最小闭环集成测试**

## 5. 文档约束测试用例
## 5.1 目标
验证 V1 的设计输出是否仍然符合当前已确认的产品边界与架构原则。
### TC-V1-DOC-001
- **名称**：V1 范围不应越界到完整三路线实现
- **目标**：验证 V1 设计与实现计划不包含“一次性完整实现文档驱动、功能用例驱动、抓包驱动全部成品能力”的越界内容。
- **前置条件**：已有 V1 范围说明书
- **检查点**：
 1. V1 范围中明确“文档驱动最小闭环”为主线；
 2. 功能用例驱动与抓包驱动列为后续增强方向；
 3. 不将 V1 宣称为最终产品成品版。
- **预期结果**：通过。
### TC-V1-DOC-002
- **名称**：V1 必须围绕统一中间模型设计
- **目标**：验证当前设计未绕过中间模型直接进入生成实现。
- **检查点**：
 1. 中间模型设计文档存在；
 2. 相关架构说明中已明确“输入 → 中间模型 → 模板/规则 → 生成资产”的路径。
- **预期结果**：通过。
### TC-V1-DOC-003
- **名称**：V1 必须采用模板化生成策略
- **目标**：验证平台没有把自由文本生成直接定义为最终落地方案。
- **检查点**：
 1. 模板体系文档存在；
 2. 全局产品说明书中已明确模板生成原则；
 3. AGENTS.md 中已明确禁止直接自由生成最终生产代码。
- **预期结果**：通过。

## 6. 中间模型测试用例
## 6.1 目标
验证中间模型是否能稳定承接文档输入，并为模板层和规则层提供结构化信息。
### TC-V1-MODEL-001
- **名称**：SourceDocument 模型应能承接文档来源信息
- **目标**：验证输入源可被统一记录。
- **输入**：一个 OpenAPI 文档来源信息
- **检查点**：
 1. 能记录 source_id；
 2. 能记录 source_type；
 3. 能记录来源路径或引用；
 4. 能记录导入时间。
- **预期结果**：通过。
### TC-V1-MODEL-002
- **名称**：ApiModule 模型应能表达模块归属
- **目标**：验证接口模块归类信息可被结构化表达。
- **输入**：模块名称、模块类型、路径提示
- **检查点**：
 1. 可记录 module_name；
 2. 可记录 module_type；
 3. 可记录路径映射提示。
- **预期结果**：通过。
### TC-V1-MODEL-003
- **名称**：ApiOperation 模型应能表达基础接口定义
- **目标**：验证单接口定义可完整表达。
- **输入**：接口方法、路径、参数、响应结构摘要
- **检查点**：
 1. 可记录 http_method；
 2. 可记录 path；
 3. 可记录 operation_name；
 4. 可记录基础参数信息；
 5. 可记录 success_codes。
- **预期结果**：通过。
### TC-V1-MODEL-004
- **名称**：ApiParam 模型应能表达不同参数位置
- **目标**：验证 header/query/path/body 参数都可表达。
- **输入**：4类不同参数定义
- **检查点**：
 1. 能区分 param_in；
 2. 能记录 required；
 3. 能记录数据类型；
 4. 能记录默认值或样例值。
- **预期结果**：通过。
### TC-V1-MODEL-005
- **名称**：ResponseField 模型应能表达断言候选来源
- **目标**：验证响应字段后续可为断言和变量提取提供基础。
- **输入**：响应字段定义
- **检查点**：
 1. 能记录字段路径；
 2. 能记录字段类型；
 3. 能标记是否可断言；
 4. 能标记是否可提取变量。
- **预期结果**：通过。
### TC-V1-MODEL-006
- **名称**：GenerationRecord 模型应能记录最小生成元数据
- **目标**：验证生成行为可追溯。
- **检查点**：
 1. 能记录生成时间；
 2. 能记录来源；
 3. 能记录目标文件；
 4. 能记录生成策略。
- **预期结果**：通过。
### TC-V1-MODEL-007
- **名称**：ExecutionRecord 模型应能记录最小执行信息
- **目标**：验证执行行为可追溯。
- **检查点**：
 1. 能记录执行级别；
 2. 能记录执行结果；
 3. 能记录开始和结束时间；
 4. 能关联报告路径。
- **预期结果**：通过。

## 7. 模板生成测试用例
## 7.1 目标
验证模板体系能够根据中间模型稳定生成统一风格的资产。
### TC-V1-TPL-001
- **名称**：接口方法模板应能生成标准接口代码骨架
- **目标**：验证单个 ApiOperation 可生成统一风格的接口方法代码。
- **输入**：一个完整的 ApiOperation 模型实例
- **检查点**：
 1. 文件内容包含统一 import；
 2. 生成的方法命名符合规则；
 3. 请求方法与路径正确；
 4. 参数组织符合模板约定；
 5. 返回结构符合模板约定。
- **预期结果**：通过。
### TC-V1-TPL-002
- **名称**：pytest 用例模板应能生成基础测试骨架
- **目标**：验证接口测试骨架可被模板稳定输出。
- **输入**：一个 ApiOperation + AssertionCandidate
- **检查点**：
 1. 生成 `test_*.py` 风格文件；
 2. 包含基础 fixture 引用；
 3. 包含接口调用示例；
 4. 包含状态码断言；
 5. 包含基础响应断言占位。
- **预期结果**：通过。
### TC-V1-TPL-003
- **名称**：断言模板应能生成统一断言片段
- **目标**：验证状态码/字段断言能被模板统一输出。
- **输入**：不同类型的 AssertionCandidate
- **检查点**：
 1. 状态码断言格式统一；
 2. JSON 字段断言格式统一；
 3. 断言错误信息表达清晰。
- **预期结果**：通过。
### TC-V1-TPL-004
- **名称**：生成记录模板应能生成最小记录结构
- **目标**：验证生成记录可被稳定落盘。
- **检查点**：
 1. 可记录来源；
 2. 可记录生成时间；
 3. 可记录目标资产路径；
 4. 可记录审核状态占位。
- **预期结果**：通过。

## 8. 规则校验测试用例
## 8.1 目标
验证基础规则体系能发现明显不合规生成结果。
### TC-V1-RULE-001
- **名称**：接口方法命名不符合规范时应被识别
- **目标**：验证规则层可检查接口方法命名。
- **输入**：不符合命名规范的接口方法定义
- **预期结果**：规则校验失败，返回命名不合规提示。
### TC-V1-RULE-002
- **名称**：测试文件命名不符合规范时应被识别
- **目标**：验证测试文件命名规则生效。
- **输入**：非 `test_*.py` 文件名
- **预期结果**：规则校验失败。
### TC-V1-RULE-003
- **名称**：模块归类缺失时应被识别
- **目标**：验证模块归档规则生效。
- **输入**：缺失 module_id / module_name 的操作对象
- **预期结果**：规则校验失败。
### TC-V1-RULE-004
- **名称**：接口核心字段缺失时应被识别
- **目标**：验证接口定义必要字段检查生效。
- **输入**：缺少 method/path 的操作对象
- **预期结果**：规则校验失败。

## 9. 执行与记录测试用例
## 9.1 目标
验证生成资产能进入最小执行闭环，并保留执行记录。
### TC-V1-EXEC-001
- **名称**：生成后的基础 pytest 用例应能被 pytest 识别
- **目标**：验证模板生成的测试骨架至少符合 pytest 基本识别规则。
- **预期结果**：pytest 可正常收集该用例。
### TC-V1-EXEC-002
- **名称**：基础 smoke 级执行应可完成
- **目标**：验证 V1 的最小执行链路成立。
- **预期结果**：可调用 pytest 执行；能输出通过/失败结果。
### TC-V1-EXEC-003
- **名称**：执行后应生成基础执行记录
- **目标**：验证执行行为可被记录。
- **检查点**：
 1. 记录执行时间；
 2. 记录执行结果；
 3. 记录报告路径或结果摘要。
- **预期结果**：通过。

## 10. 最小闭环集成测试用例
## 10.1 目标
验证从文档输入到测试执行的 V1 最小闭环整体可用。
### TC-V1-INT-001
- **名称**：文档驱动最小闭环集成测试
- **目标**：验证平台可从文档输入打通到测试骨架生成与执行。
- **步骤**：
 1. 输入一份结构化 OpenAPI / Swagger 文档；
 2. 解析为中间模型；
 3. 生成接口方法；
 4. 生成基础 pytest 测试骨架；
 5. 通过基础规则校验；
 6. 执行 pytest；
 7. 记录生成与执行结果。
- **预期结果**：闭环成立。
### TC-V1-INT-002
- **名称**：闭环结果应具备最小可追溯性
- **目标**：验证最小闭环完成后，资产不是一次性产物。
- **检查点**：
 1. 有来源信息；
 2. 有生成记录；
 3. 有执行记录；
 4. 有目标文件定位信息。
- **预期结果**：通过。

## 12. 测试通过标准
V1 阶段，以下条件同时满足时，才能宣称 V1 最小闭环通过：
1. 所有 P0 测试项完成设计并通过；

3. 模板层可生成统一风格资产；
4. 规则层能拦截明显不合规结果；
5. pytest 最小执行链路成立；
6. 生成与执行记录具备最小可追溯性。
