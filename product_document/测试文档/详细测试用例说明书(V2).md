## 2. V2 测试目标
V2 的测试目标不是验证完整最终平台，而是验证 V2 已确认范围内的正式闭环是否成立。
V2 测试重点验证以下内容：
1. 场景驱动核心模型可以正式落地，并能承接功能测试用例和抓包两条输入路线；
2. 功能测试用例驱动能够从场景描述进入“草稿 -> 审核 -> 确认 -> 执行 -> 回写”的正式闭环；
3. 抓包驱动能够完成“导入 -> 清洗 -> 去重 -> 归一化 -> 草稿生成 -> 审核接入”的草稿化闭环；
4. 变量绑定、依赖编排和多步骤场景执行可以稳定工作；
5. Django + DRF + MySQL 能成为正式事实承载层，本地工作区降级为导入导出与 pytest 执行载体；
6. Web / Windows 可用型入口能够打通导入、预览、审核确认、执行和结果查看；
7. AI 辅助改写不能绕过规则校验、人工确认和执行验证。

## 2.1 当前公开接口测试基线站点
当前 V2 阶段涉及公开接口自动化测试、服务化回归示例、场景执行演示与入口联调时，统一使用：
- `https://jsonplaceholder.typicode.com/`

测试设计必须继续遵循其公开 Guide 约束：
1. 资源范围固定在 `posts/comments/albums/photos/todos/users` 等示例资源；
2. 支持 `GET/POST/PUT/PATCH/DELETE`、查询参数过滤和一层嵌套路由；
3. `POST/PUT/PATCH/DELETE` 为伪写入，不做真实持久化，V2 场景断言不能错误假设远端状态已经被永久修改；
4. 涉及变量提取、依赖编排和多步骤执行时，应优先围绕响应契约、已有读取接口和伪写入返回值设计，不把公共基线误当成真实业务库；
5. 正式回归或验收如需访问该站点，建议沿用 V1 的方式临时开启 `api_test/api_config.json` 中的代理开关执行相关用例，完成后恢复 `proxy.enabled=false`。

## 3. 测试设计原则
## 3.1 说明书追溯原则
所有测试用例必须可追溯到全局产品说明书、总体架构设计说明书、中间模型设计说明书、UI 设计说明文档与 V2 阶段工作计划文档中的明确目标。
## 3.2 TDD 原则
后续实现阶段必须优先按本说明书中的测试项编写测试，再进行最小实现。
## 3.3 场景驱动优先原则
V2 的测试主线应围绕 `TestScenario`、`ScenarioStep`、`VariableBinding`、`DependencyLink` 与审核确认链路展开，而不是回退为单接口模板增强视角。
## 3.4 数据库事实源原则
V2 的正式资产、审核记录、执行记录和状态查询，以数据库为事实源；本地工作区只能作为导入导出、生成落盘和 pytest 执行载体。
## 3.5 审核门禁原则
草稿、修订、AI 建议和抓包结果都不能直接成为正式资产；“正式入库”与“触发正式执行”前必须经过确认门禁。
## 3.6 分层测试原则
V2 测试必须同时覆盖文档约束、模型、解析、模板、规则、服务、执行、入口和集成闭环，不允许只做单层验证后宣称完成。
## 3.7 一致性原则
服务层输出、数据库记录、本地工作区导出结果和交互入口展示口径必须一致，不能出现“双事实源各说各话”的情况。
## 3.8 AI 非最终事实原则
AI 生成或改写内容只能作为草稿、建议和候选输入；V2 测试重点验证其门禁、留痕和回退能力，而不把 AI 语义质量本身定义为阶段验收事实。

建议 V2 测试分为以下八层：
1. **文档与阶段约束测试**

4. **模板与规则测试**
5. **服务层与持久化测试**
6. **执行与调度测试**
7. **交互入口测试**
8. **主线集成闭环测试**

## 7. 解析与标准化测试用例
## 7.1 目标
验证功能测试用例驱动和抓包驱动都能先经过解析、清洗和标准化，再进入统一场景模型，而不是直接生成最终资产。

- **目标**：验证文本或结构化功能测试用例可以先转成场景草稿对象。
- **输入**：一份包含场景名称、目标、步骤和预期结果的功能测试用例
- **检查点**：
 1. 能生成 `TestScenario` 草稿；
 2. 能填充场景名称、描述、来源和优先级；
 3. 解析结果不直接跳到 pytest 代码。
- **优先级**：P0
