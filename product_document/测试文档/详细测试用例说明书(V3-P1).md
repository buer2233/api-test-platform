## 2. P1 测试目标
`P1` 测试目标不是重新验证 `P0` 的治理对象，而是在 `P0` 已稳定验收的前提下，验证平台能力深化是否成立。

`P1` 重点验证以下内容：
1. 权限体系、项目级访问边界和审计留痕是否建立在 `P0` 隔离边界之上；
2. 抓包是否从“草稿化接入”升级到“正式执行闭环”；
3. Web 正式入口是否已经从最小治理入口深化为更可用的产品入口；
4. Windows Demo 是否具备真实测试能力，并满足“浏览器先验、Tauri 优先、阶段性打包复验”的路线要求；
5. 调度与执行中心是否能支撑批量执行、重试和结果聚合，同时不破坏项目隔离；
6. `P1` 能形成独立验收，不依赖 `P2` 的 AI 治理实现。
---
## 2.1 当前公开接口测试基线站点
当前 `P1` 阶段涉及抓包正式执行、入口深化、Windows Demo 实测和调度中心回归时，统一使用：
- `https://jsonplaceholder.typicode.com/`

测试设计仍必须遵循其公开 Guide 约束：
1. 场景执行、抓包导入和 Demo 演示优先使用公开可复现资源；
2. 写操作断言仍基于伪写入返回契约，不假设远端状态真实持久化；
3. Windows Demo 和浏览器端入口的实测结果应围绕项目归属、权限状态、执行上下文和结果回写一致性设计。

## 3. P1 测试边界
### 3.1 P1 应覆盖
1. 权限体系与项目级访问边界；
2. 审计日志和关键操作留痕；
3. 抓包正式执行闭环；
4. Web 正式入口深化；
5. Windows Demo 的可实际测试链路；
6. 调度与执行中心；
7. `P1` 独立验收。

### 3.2 P1 不应越界
1. 不把 `P2` 的 AI 自愈和自动修复写成 `P1` 主实现目标；
2. 不把 `macOS` 正式客户端写成 `P1` 必达项；
3. 不让 Windows Demo 反向绑架治理模型和服务边界；
4. 不在 `P1` 中弱化 `P0` 的执行隔离和治理对象约束。

### 3.3 P1 测试设计原则
1. 任何深化能力都必须建立在 `P0` 隔离边界之上；
2. 权限与审计验证必须覆盖“允许”和“阻断”两类场景；
3. Windows Demo 必须优先验证真实测试能力和重复打包验证便捷性；
4. 调度中心必须同时验证执行能力和边界安全性。

## 5. 文档约束测试用例
### TC-V3-P1-DOC-001
- **名称**：P1 不应绕过 P0 独立验收和隔离边界
- **目标**：验证 `P1` 文档、测试和实现不会默认 `P0` 隔离边界已经可忽略。
- **背景**：权限、调度和 Windows Demo 一旦忽略 `P0`，会直接打穿项目隔离。
- **前置条件**：
  1. 已存在 `V3阶段工作计划文档.md`
  2. 已存在 `V3-P0` 和 `V3-P1` 测试文档
- **输入**：阶段计划文档、`P0` / `P1` 测试文档
- **执行步骤**：
  1. 检查 `P1` 测试边界和通过标准
  2. 检查 `P1` 是否把 `P0` 视为前置条件
  3. 检查是否存在无项目边界的权限、调度或入口设计
- **检查点**：
  1. `P1` 明确建立在 `P0` 验收通过之后
  2. `P1` 测试用例保留项目和环境上下文
  3. 未出现“调度可跨项目共享”之类越界口径
- **失败判定**：
  1. `P1` 用例忽略项目隔离
  2. Windows Demo 或调度用例不带项目归属检查
- **优先级**：P0
- **预期结果**：通过，`P1` 不削弱 `P0` 边界。
- **归属阶段验收**：P1
- **后续联动测试**：`TC-V3-P1-ISO-003`、`TC-V3-P1-INT-001`

- **检查点**：
  1. 浏览器端验证仍是主路径
  2. Windows Demo 明确以 Tauri 为优先壳方案

## 6. 模型测试用例
### TC-V3-P1-MODEL-001
- **名称**：权限对象应能表达项目级访问边界和关键操作授权
- **目标**：验证 `P1` 的权限对象足以承接查看、修改、执行和审核四类核心权限。
- **背景**：P1 的权限体系不做复杂全局 IAM，但必须先把项目级边界做实。
- **前置条件**：已定义权限对象和角色模型设计
- **输入**：包含用户、角色、项目归属和授权范围的权限定义
- **执行步骤**：
  1. 检查权限对象字段设计
  2. 检查角色到权限、权限到项目的映射
  3. 检查是否能区分查看、修改、执行、审核权限
- **检查点**：
  1. 可记录用户、角色和项目归属
  2. 可记录关键操作授权范围
  3. 可被 API、UI 和审计层共同消费
- **失败判定**：
  1. 权限对象缺少项目归属维度
  2. 权限无法区分执行和审核
- **优先级**：P0
- **预期结果**：通过，权限对象满足 P1 基础要求。
- **归属阶段验收**：P1
- **后续联动测试**：`TC-V3-P1-API-001`、`TC-V3-P1-ISO-001`

### TC-V3-P1-MODEL-002
- **名称**：审计日志对象应能记录关键操作上下文和审批链
- **目标**：验证审计日志不是弱流水，而是正式治理事实对象。
- **背景**：权限体系一旦落地，没有可追溯的审计日志就很难形成真正治理闭环。
- **前置条件**：已定义审计日志对象设计
- **输入**：一次授权变更、一次执行请求和一次审核动作
- **执行步骤**：
  1. 检查审计日志对象字段设计
  2. 检查操作人、项目、对象、动作、时间和结果字段
  3. 检查审批链和变更上下文是否可追溯
- **检查点**：
  1. 可记录项目、用户、动作和对象
  2. 可记录审批前后状态和执行结果
  3. 可支持后续查询和验收引用
- **失败判定**：
  1. 日志对象无法记录项目上下文
  2. 审批动作和执行动作没有区分
- **优先级**：P0
- **预期结果**：通过，审计日志具备正式治理价值。
- **归属阶段验收**：P1
- **后续联动测试**：`TC-V3-P1-API-002`、`TC-V3-P1-INT-003`

### TC-V3-P1-MODEL-003
- **名称**：抓包正式执行对象应能表达确认、绑定和执行状态
- **目标**：验证抓包从草稿升级到正式执行时，有稳定对象承接其状态变化。
- **背景**：V2 只做到抓包草稿化接入，P1 必须补正式执行状态对象。
- **前置条件**：已定义抓包正式执行相关对象
- **输入**：一条抓包草稿从确认到执行的状态流
- **执行步骤**：
  1. 检查抓包场景确认对象和状态字段
  2. 检查变量绑定确认和执行准备字段
  3. 检查正式执行后结果关联字段
- **检查点**：
  1. 可区分草稿、已确认、待执行、已执行等状态
  2. 可记录变量绑定确认结果和审核门禁
  3. 可与执行记录、版本和来源追溯关联
- **失败判定**：
  1. 抓包对象仍只有“草稿”一种语义
  2. 无法记录正式确认与执行之间的中间状态
- **优先级**：P0
- **预期结果**：通过，抓包正式执行对象结构稳定。
- **归属阶段验收**：P1
- **后续联动测试**：`TC-V3-P1-API-003`、`TC-V3-P1-EXEC-001`

### TC-V3-P1-MODEL-004
- **名称**：调度对象应能表达任务、队列、重试和聚合摘要
- **目标**：验证调度中心具备正式建模基础，而不是散落在脚本逻辑里。
- **背景**：P1 调度中心需要支持批量执行、队列和重试，必须有正式对象承接。
- **前置条件**：已定义调度任务、任务项和结果聚合对象
- **输入**：一批场景执行计划及其重试策略
- **执行步骤**：
  1. 检查调度对象字段设计
  2. 检查队列状态、重试状态和结果聚合字段
  3. 检查是否能记录项目 / 环境 / 版本上下文
- **检查点**：
  1. 可记录任务、子任务和队列状态
  2. 可记录重试策略和失败原因
  3. 可记录结果聚合摘要并附带治理上下文
- **失败判定**：
  1. 调度对象缺少项目上下文
  2. 队列状态和结果聚合无法结构化表达
- **优先级**：P1
- **预期结果**：通过，调度中心具备正式建模基础。
- **归属阶段验收**：P1
- **后续联动测试**：`TC-V3-P1-API-004`、`TC-V3-P1-EXEC-003`

## 7. 迁移与兼容测试用例
### TC-V3-P1-MIG-001
- **名称**：P0 治理对象升级到 P1 后不应破坏既有项目、环境和版本关系
- **目标**：验证权限、抓包正式执行和调度接入不会破坏 `P0` 已建立的治理对象。
- **背景**：P1 若打破 P0 治理边界，阶段独立验收将失去意义。
- **前置条件**：
  1. `P0` 已具备稳定项目 / 环境 / 场景集 / 版本关系
  2. 已接入 P1 新对象
- **输入**：升级前后的治理对象快照
- **执行步骤**：
  1. 对比升级前后的项目、环境、场景集和版本关系
  2. 对比历史执行记录和查询摘要
  3. 检查是否因 P1 对象加入而重写既有归属关系
- **检查点**：
  1. 升级后既有治理对象关系不变
  2. 历史执行与版本上下文不丢失
  3. 新权限、调度和抓包对象只是增量挂接
- **失败判定**：
  1. 升级后历史数据失去项目或版本归属
  2. P1 升级要求重建 P0 数据模型
- **优先级**：P0
- **预期结果**：通过，P1 升级不破坏 P0 事实边界。
- **归属阶段验收**：P1
- **后续联动测试**：`TC-V3-P1-INT-001`

### TC-V3-P1-MIG-002
- **名称**：Web 正式入口与 Windows Demo 应共享同一服务契约和治理上下文
- **目标**：验证浏览器端到 Windows Demo 的切换不引入第二套事实口径。
- **背景**：用户已明确要求“先浏览器测试，再阶段性打包到 Windows”，两者必须共用同一服务契约。
- **前置条件**：
  1. Web 正式入口已存在
  2. Windows Demo 路线已建立
- **输入**：同一项目和场景在浏览器端与 Windows Demo 中的访问请求
- **执行步骤**：
  1. 在浏览器端读取项目 / 环境 / 场景信息
  2. 在 Windows Demo 中读取同一数据
  3. 对比服务返回和入口展示
- **检查点**：
  1. 浏览器端与 Windows Demo 读取同一服务契约
  2. 治理上下文、项目归属和执行摘要一致
  3. Windows Demo 没有引入独立事实缓存
- **失败判定**：
  1. Windows Demo 需要独立数据转换层才能理解结果
  2. 两个入口显示不同的治理上下文
- **优先级**：P1
- **预期结果**：通过，浏览器和 Windows Demo 共用统一事实口径。
- **归属阶段验收**：P1
- **后续联动测试**：`TC-V3-P1-UI-002`

## 8. 隔离与边界测试用例
### TC-V3-P1-ISO-001
- **名称**：权限检查必须严格受项目边界约束
- **目标**：验证用户角色不能越权访问不属于自己的项目资产。
- **背景**：权限体系如果脱离项目边界，会直接推翻 P0 治理底座。
- **前置条件**：
  1. 已建立多个项目
  2. 已配置不同用户角色
- **输入**：跨项目查询、修改、执行和审核请求
- **执行步骤**：
  1. 以项目 A 权限用户访问项目 B 资产
  2. 尝试执行项目 B 场景
  3. 尝试审核项目 B 草稿或抓包场景
- **检查点**：
  1. 非授权访问被阻断
  2. 阻断结果有明确错误语义
  3. 审计日志记录越权尝试
- **失败判定**：
  1. 非授权用户可以查看或执行其他项目资产
  2. 审计日志未记录越权尝试
- **优先级**：P0
- **预期结果**：通过，项目级权限边界成立。
- **归属阶段验收**：P1
- **后续联动测试**：`TC-V3-P1-EXEC-002`

### TC-V3-P1-ISO-002
- **名称**：抓包正式执行不得绕过项目和环境绑定
- **目标**：验证抓包场景一旦进入正式执行，仍然受到与普通场景相同的治理边界约束。
- **背景**：抓包结果天然更“像事实”，更容易诱发越权跳过治理门禁。
- **前置条件**：
  1. 已存在抓包草稿
  2. 已配置项目、环境和审核流程
- **输入**：抓包草稿正式确认与执行请求
- **执行步骤**：
  1. 直接发起未绑定项目 / 环境的抓包执行
  2. 发起已确认但跨项目的抓包执行
  3. 发起合法项目 / 环境上下文下的正式执行
- **检查点**：

1. 权限体系只能展示，不能阻断关键操作；
2. 抓包场景仍无法进入正式执行闭环；
3. Windows Demo 不能进行真实测试；
4. Windows 路线偏离“浏览器先验、Tauri 优先、阶段性打包复验”；
5. 调度中心会打穿项目边界或结果聚合失真；
6. `P1` 验收需要依赖 `P2` 的 AI 治理能力才能完成。
