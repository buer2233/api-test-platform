## 2. V3 测试目标
V3 测试的目标不是只验证局部实现，而是验证 `P0 / P1 / P2` 三个阶段是否分别形成独立闭环，并共同支撑 V3 的平台治理深化目标。

V3 测试重点验证以下内容：
1. `P0` 的多项目、多环境、场景集、版本、执行记录治理是否能稳定成立；
2. `P0` 的执行隔离级边界是否真实成立，而不是停留在文档层；
3. V2 历史资产迁入默认项目后的可追溯性、可查询性与可执行性是否成立；
4. `P1` 的权限体系、抓包正式执行闭环、入口深化和调度中心是否建立在 P0 稳定边界之上；
5. `P1` 的 Windows Demo 是否具备真实测试能力，并符合“浏览器先验、Tauri 优先、阶段性打包复验”的路线；
6. `P2` 是否能在不越权的前提下定义高级 AI 治理边界、审批门禁、回退机制和独立验收要求；
7. `P0 / P1 / P2` 是否都能形成独立验收，而不是只依赖最终总体验收。

## 2.1 当前公开接口测试基线站点
当前 V3 阶段涉及公开接口自动化测试、治理入口演示、Windows Demo 实测、抓包正式执行回归和阶段验收示例时，统一使用：
- `https://jsonplaceholder.typicode.com/`

测试设计必须继续遵循其公开 Guide 约束：
1. 资源范围固定在 `posts/comments/albums/photos/todos/users` 等示例资源；
2. 支持 `GET/POST/PUT/PATCH/DELETE`、查询参数过滤和一层嵌套路由；
3. `POST/PUT/PATCH/DELETE` 为伪写入，不做真实持久化，V3 相关用例断言不能错误假设远端状态已被永久修改；
4. 涉及项目隔离、环境绑定、Windows Demo 或抓包正式执行的场景时，应优先围绕响应契约、环境切换和结果回写一致性设计断言；
5. 正式回归或阶段验收如需访问该站点，仍建议沿用“临时开启代理执行公网相关用例，完成后恢复默认关闭”的流程，避免破坏仓库默认配置契约。

## 4. 文档包结构说明
V3 测试文档包采用：
> **1 个总索引 + 3 个分文档**

各文档职责如下：

| 文档 | 职责 | 是否写详细用例 |
| --- | --- | --- |
| `详细测试用例说明书(V3-总索引).md` | 定义 V3 测试目标、编号规则、测试分层、阶段边界、验收矩阵与文档映射关系 | 否 |
| `详细测试用例说明书(V3-P0).md` | 承载 P0 全部详细测试用例，服务 P0 独立验收 | 是 |
| `详细测试用例说明书(V3-P1).md` | 承载 P1 全部详细测试用例，服务 P1 独立验收 | 是 |
| `详细测试用例说明书(V3-P2).md` | 承载 P2 全部详细测试用例，服务 P2 独立验收 | 是 |

使用原则：
1. 总索引只负责统一口径和引用关系；
2. 详细用例只写在 `P0 / P1 / P2` 分文档；
3. 阶段验收报告优先引用统一编号，不重复改写详细步骤。

## 5. 编号规则
### 5.1 统一编号格式
V3 详细测试用例统一采用：

> `TC-V3-Px-LAYER-XXX`

示例：
1. `TC-V3-P0-DOC-001`
2. `TC-V3-P1-API-012`
3. `TC-V3-P2-INT-003`

### 5.2 编号字段说明
1. `V3`：所属版本阶段；
2. `Px`：所属阶段，取值为 `P0 / P1 / P2`；
3. `LAYER`：所属测试层；
4. `XXX`：阶段内对应测试层的顺序号，使用三位数字递增。

### 5.3 编号使用规则
1. 总索引、阶段计划文档、README、后续验收报告都优先使用统一编号进行引用；
2. 同一编号在整个仓库内只允许出现一次，不允许跨文档复用；
3. 如果后续新增用例，必须在所属阶段和所属测试层下继续顺延，不允许打乱已存在编号。

## 6. 测试分层定义
V3 测试层固定定义如下：

| 代号 | 名称 | 说明 |
| --- | --- | --- |
| `DOC` | 文档与阶段边界测试 | 验证阶段边界、范围、不做项、独立验收原则是否仍然成立 |
| `MODEL` | 模型与治理对象测试 | 验证项目、环境、场景集、权限、AI 治理等对象是否被正确建模 |
| `MIG` | 迁移与兼容测试 | 验证阶段升级、历史资产迁移、配置兼容和升级连续性 |
| `ISO` | 隔离与边界测试 | 验证项目边界、环境边界、事实源边界和 Demo 边界是否成立 |
| `API` | 服务与接口契约测试 | 验证服务层、查询层和管理接口的正式契约 |
| `EXEC` | 执行与调度测试 | 验证执行触发、结果回写、抓包正式执行、批量执行和调度行为 |
| `UI` | 交互入口与 Demo 测试 | 验证浏览器治理入口、Web 正式入口和 Windows Demo 的测试链路 |
| `INT` | 主线集成与阶段验收测试 | 验证阶段主链路闭环和独立验收路径 |

补充规则：
1. 三份分文档都按以上测试层分章；
2. 某一阶段即使某层内容较轻，也需保留该层章节并显式说明其边界；
3. 不允许临时发明不在总索引中的测试层代号。

## 7. 分阶段独立验收原则
### 7.1 P0 独立验收原则
`P0` 独立验收必须证明：
1. 历史资产迁移成立；
2. 项目、环境、场景集、版本、执行记录形成统一归属关系；
3. 执行隔离级边界真实成立；
4. 浏览器端最小治理入口可用；
5. `P0` 不越界到 `P1 / P2` 的主实现范围。

### 7.2 P1 独立验收原则
`P1` 独立验收必须证明：
1. 权限体系建立在 P0 边界之上；
2. 抓包正式执行闭环成立；
3. Web 正式入口深化可用；
4. Windows Demo 可实际测试；
5. 调度中心不破坏项目隔离。

### 7.3 P2 独立验收原则
`P2` 独立验收必须证明：
1. AI 治理边界明确；
2. 审批门禁和回退机制成立；
3. AI 建议不能越权成为事实源；
4. P2 当前实现范围与边界范围被清晰区分。

### 7.4 V3 总体验收原则
V3 总体验收必须建立在：
1. `P0 / P1 / P2` 都已形成明确独立验收结论；
2. README、阶段文档、测试文档和正式验收报告口径一致；
3. V3 与后续阶段边界清晰；
4. 当前轮未做项已明确转入后续阶段。
