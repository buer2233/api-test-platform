## 2. P0 测试目标
`P0` 测试目标不是提前验证权限、调度和 AI 自愈，而是验证 V3 平台治理底座是否成立。

`P0` 重点验证以下内容：
1. 项目、测试环境、场景集、标签、归档、基线版本等治理对象能被正式建模；
2. V2 历史资产能迁移到默认项目 / 默认环境 / 默认场景集，并保持可追溯、可查询、可执行；
3. 项目级工作区、执行记录、导入导出边界和环境绑定执行能够真实隔离；
4. 浏览器端最小治理入口能支持项目切换、环境切换、场景集筛选、版本查看和隔离结果回看；
5. `P0` 能形成独立阶段验收，不依赖 `P1 / P2` 的实现。
---
## 2.1 当前公开接口测试基线站点
当前 `P0` 阶段涉及公开接口演示、迁移后资产执行验证、浏览器端治理入口回归和独立验收示例时，统一使用：
- `https://jsonplaceholder.typicode.com/`

测试设计必须继续遵循其公开 Guide 约束：
1. 资源范围固定在 `posts/comments/albums/photos/todos/users` 等示例资源；
2. 允许查询参数过滤与一层嵌套路由；
3. 写操作为伪写入，不做真实持久化，`P0` 的断言重点是治理归属、环境绑定和结果回写一致性；
4. 如正式回归需访问该站点，建议沿用代理临时开启、完成后恢复默认关闭的流程。

## 3. P0 测试边界
### 3.1 P0 应覆盖
1. 项目对象、测试环境对象、场景集对象、基线版本对象；
2. 历史资产默认项目迁移；
3. 项目级执行隔离；
4. 浏览器端最小治理入口；
5. `P0` 独立验收闭环。

### 3.2 P0 不应越界
1. 不把权限体系与审计治理当成 `P0` 主实现目标；
2. 不把抓包正式执行闭环当成 `P0` 主实现目标；
3. 不把 Windows Demo 当成 `P0` 必达项；
4. 不把 AI 自愈与自动修复当成 `P0` 测试重点。

### 3.3 P0 测试设计原则
1. 迁移优先验证事实连续性，不只验证新对象创建成功；
2. 隔离优先验证“不会串扰”，而不只验证“能执行”；
3. 入口优先验证治理能力承接，不把 UI 美观度当成 `P0` 成败标准；
4. 所有结果必须能回溯到项目、环境、场景集和基线版本。

## 6. 模型测试用例
### TC-V3-P0-MODEL-001
- **名称**：项目对象应能表达治理归属与生命周期状态
- **目标**：验证 `P0` 的项目对象足以成为后续所有治理关系的根对象。
- **背景**：没有稳定项目对象，后续环境、场景集、执行记录和入口切换都无法建立统一归属。
- **前置条件**：已定义 `P0` 的项目对象设计
- **输入**：一个项目定义，包括名称、编码、状态和元数据
- **执行步骤**：
  1. 检查项目对象字段设计
  2. 检查项目状态与归档字段
  3. 检查是否支持与环境、场景集、执行记录建立关联
- **检查点**：
  1. 可记录项目唯一标识和项目编码
  2. 可记录项目状态、归档状态和基础元数据
  3. 可作为环境、场景集和执行记录的上游归属对象
- **失败判定**：
  1. 项目对象缺少唯一归属标识
  2. 项目对象无法承载归档或状态信息
- **优先级**：P0
- **预期结果**：通过，项目对象可成为治理根对象。
- **归属阶段验收**：P0
- **后续联动测试**：`TC-V3-P0-API-001`、`TC-V3-P0-ISO-002`

### TC-V3-P0-MODEL-002
- **名称**：测试环境对象应能表达连接参数、变量集和数据隔离标识
- **目标**：验证环境对象达到“测试环境层”而不是仅有基地址的弱配置。

- **输入**：一个环境定义，包括基础连接参数、认证、变量集和隔离标识
- **执行步骤**：
  1. 检查环境对象字段设计
  2. 检查是否支持项目绑定和场景集绑定
  3. 检查变量集、认证和超时配置是否可被结构化表达
- **检查点**：
  1. 可记录 `base_url`、认证方式、请求头、代理和超时
  2. 可记录环境变量集和数据隔离标识
  3. 可记录与项目、场景集的绑定关系
- **失败判定**：
  1. 环境对象仅能记录基地址，无法承载变量集
  2. 环境与项目 / 场景集没有清晰绑定关系
- **优先级**：P0
- **预期结果**：通过，环境对象满足测试环境层要求。
- **归属阶段验收**：P0
- **后续联动测试**：`TC-V3-P0-ISO-004`、`TC-V3-P0-EXEC-001`

## 7. 迁移与兼容测试用例
### TC-V3-P0-MIG-001
- **名称**：V2 历史场景资产应能迁入默认项目 / 默认环境 / 默认场景集
- **目标**：验证 V2 历史资产迁移存在明确落点，而不是升级后出现“无项目归属”资产。
- **背景**：用户已明确确认历史资产承接采用“默认项目迁移”策略。
- **前置条件**：
  1. 数据库中存在 V2 场景资产
  2. 已定义默认项目迁移策略
- **输入**：包含场景、修订、建议和执行记录的 V2 历史资产集合
- **执行步骤**：
  1. 执行默认项目迁移流程
  2. 检查场景、修订、建议和执行记录的项目归属
  3. 检查默认环境和默认场景集绑定结果
- **检查点**：
  1. 正式资产全部归属到默认项目
  2. 执行记录和修订、建议记录继续可追溯
  3. 默认环境和默认场景集建立稳定映射
- **失败判定**：
  1. 迁移后仍存在无项目归属资产
  2. 修订、建议或执行记录丢失关联关系
- **优先级**：P0
- **预期结果**：通过，V2 历史资产完成默认项目迁移。
- **归属阶段验收**：P0
- **后续联动测试**：`TC-V3-P0-MIG-002`、`TC-V3-P0-INT-002`

### TC-V3-P0-MIG-002
- **名称**：历史迁移后应保留来源追溯、执行历史和结果查询能力
- **目标**：验证迁移不是只把主表记录搬过去，而是保留完整事实链。
- **背景**：V2 已形成来源追溯、建议治理和执行历史，V3 迁移不能破坏这些事实。
- **前置条件**：
  1. 已完成默认项目迁移
  2. V2 历史资产中存在执行历史、修订记录和建议记录
- **输入**：迁移后的默认项目数据集
- **执行步骤**：
  1. 查询迁移后场景详情
  2. 查询执行历史、修订记录和建议记录
  3. 比对迁移前后记录可追溯性
- **检查点**：
  1. 场景详情仍可显示来源摘要
  2. 执行历史、修订和建议记录保持关联
  3. 查询结果不因迁移而丢失上下文
- **失败判定**：
  1. 迁移后来源摘要消失
  2. 执行历史或修订、建议记录断链
- **优先级**：P0
- **预期结果**：通过，迁移后仍能保持完整追溯链。
- **归属阶段验收**：P0
- **后续联动测试**：`TC-V3-P0-API-002`

### TC-V3-P0-MIG-003
- **名称**：默认项目迁移应具备幂等性和异常回退判定
- **目标**：验证迁移脚本或迁移流程可重复执行，且异常时不会留下半完成污染。
- **背景**：默认项目迁移是一次性高风险动作，若无法重跑和回退，会直接影响 P0 验收可信度。
- **前置条件**：
  1. 已实现默认项目迁移流程
  2. 已准备可重复执行的测试数据集
- **输入**：同一批 V2 历史资产
- **执行步骤**：
  1. 首次执行迁移流程
  2. 对同一数据再次执行迁移流程
  3. 人为构造部分失败，检查回退或失败阻断机制
- **检查点**：
  1. 重复执行不会生成重复项目归属或重复关联
  2. 异常时能识别半完成状态
  3. 能给出明确的失败阻断或回退判定
- **失败判定**：
  1. 重跑后产生重复关联记录

- **优先级**：P0
