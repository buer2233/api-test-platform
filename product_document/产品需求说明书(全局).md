## 2. 项目背景
当前仓库内已经具备一个基于 Python + requests + pytest 的接口自动化测试框架雏形，但其定位仍偏向“可运行测试框架”，尚未达到“接口自动化测试平台”的产品级能力。
本项目未来要建设的，不只是一个手写接口自动化用例的执行框架，而是一个：
- 能围绕接口文档、功能测试用例、抓包结果等多种输入，
- 借助 AI 进行接口资产生成、测试场景编排、断言补全、规则校验与执行验证，
- 并最终形成统一测试资产沉淀、复用、扩展与治理能力的
**接口自动化测试平台**。
未来所有新增或生成的接口自动化测试用例，都需要基于该平台完成编写、组织、执行、验证与管理。
---
## 3. 产品定位
本平台的定位不是单点的“AI 自动生成测试代码工具”，而是：
> **统一接口测试资产平台 + AI 生成/编排能力 + 规则校验与执行验证闭环**
平台需要同时具备以下属性：
1. **资产平台**：统一管理接口方法、测试用例、测试数据、生成记录、执行结果与报告。
2. **AI 辅助平台**：AI 负责生成、补全、编排、解释、修订建议，但不直接替代事实来源与规则系统。
3. **本地优先平台**：优先支持本地单机闭环，适配本地目录、pytest 执行、抓包与桌面应用场景。
4. **可平台化扩展**：后续可扩展为 Web 服务访问、多项目管理、多人协作和更完整的平台产品。
---
## 4. 产品目标
建设一个接口自动化测试平台，使其具备以下总体目标：
1. 根据用户提供的不同输入信息，自动或半自动生成接口测试资产。
2. 所有生成资产统一适配项目既定技术体系：**Python + requests + pytest**。
3. 平台既支持全自动生成，也支持人工预览、人工确认、人工补充和半自动生成。
4. 平台最终支持通过 **Web 服务方式访问使用**。
5. 平台最终支持打包为 **桌面应用程序**。
6. 桌面应用优先支持 **Windows**，架构设计时需预留后续支持 **macOS** 的扩展能力。
7. 平台最终需要形成统一的测试资产沉淀、执行验证、报告展示和可追溯治理能力。

## 6. 开发流程要求（强约束）
后续开发工作必须严格遵循以下流程：
1. **先整理并确认产品说明书**。
2. **所有开发都必须基于详细产品说明书进行**。
3. **在进入实现前，先完成 UI / 交互设计确认**，确保前端与 Windows 客户端视觉风格符合要求。
4. **产品说明书和 UI 设计确认后，先按照说明书编写详细测试用例**。
5. **再按照产品说明书、UI 设计和测试用例进行开发**。
6. **开发完成后，执行测试，所有测试通过才算开发完成**。
7. **最后对照产品说明书与 UI 设计说明进行验收核对**。
8. **整体开发模式要求采用 TDD 思路推进**。
### 6.1 当前达成的一致原则
- 当前优先事项是“需求整理、产品说明书完善、架构设计”，不是直接堆砌功能。
- 所有后续新功能都应先确认说明书，再进入实现。
- 需求讨论内容必须优先沉淀到本地文档，避免上下文丢失。
- 架构设计必须为后续 V2/V3 扩展预留能力，但 V1 不追求一次性大而全。

## 7. 当前项目现状
当前仓库已有一个基础接口自动化测试框架，主要特征如下：
1. 技术栈为 Python + requests + pytest。
2. 已具备基础 API 请求封装能力。
3. 已具备 pytest 执行能力。
4. 已有少量示例测试用例。

### 7.1 当前问题判断

1. 更偏手写测试框架，不具备资产平台能力。
2. 缺少统一的生成规范、模板体系、中间模型与规则校验机制。
3. 缺少对 AI 生成测试资产的结构化承接能力。
4. 缺少面向文档驱动、场景驱动、抓包驱动的标准输入路径。
5. 缺少平台化的记录、审核、报告与可追溯机制。

## 9. 输入路线（已确认）
平台未来需要同时支持三条输入路线，但开发上分阶段推进。
### 9.1 路线一：接口文档驱动
适用场景：
- 用户提供 OpenAPI / Swagger / Markdown / 其他接口文档；
- 平台根据接口文档生成接口定义、接口方法、基础测试骨架。
价值判断：
- 是平台自动生成能力最重要的基础路线；
- 最适合作为 V1 / V2 的主线能力；
- 对文档质量依赖较高。
### 9.2 路线二：功能测试用例驱动
适用场景：
- 用户提供详细功能测试用例或业务场景描述；
- 平台基于已有接口资产自动或半自动编排为接口自动化测试场景。
价值判断：
- 是从“业务场景”到“接口自动化测试”的关键桥梁；
- 非常适合测试工程师实际使用；
- 依赖接口资产层先稳定沉淀。
### 9.3 路线三：抓包驱动
适用场景：
- 缺少完整文档或完整用例；
- 用户通过页面或客户端实际操作，由平台或代理抓取请求流量；
- 平台分析并提取接口、参数、依赖、顺序，进一步生成接口资产与测试场景。
价值判断：
- 是重要的补充路线；
- 特别适合无完整文档但有真实业务链路的项目；
- 后续需重点解决抓包清洗、去重、参数归一化、动态值识别、依赖提取等问题。

## 10. 输出资产要求
平台输出的不只是单个代码文件，而是统一的测试资产。
### 10.1 输出资产类型
平台后续至少需要支持输出：
1. 接口方法资产
2. 测试用例资产
3. 测试数据资产
4. 断言资产 / 断言模板
5. 生成记录与元数据
6. 执行记录与测试报告
### 10.2 输出资产原则
所有输出资产需要满足：
- 可读
- 可执行
- 可维护
- 可审查
- 可追溯
- 可复用
- 可持续演进

## 11. 接口封装规范要求
1. 每个接口应封装为单独的接口方法。
2. 接口方法应统一放在独立 `.py` 文件中。
3. 接口方法需要按业务模块归类。
4. 类似 `/api/module1`、`/api/bs/module1`、`/api/app/module1` 等接口，应根据项目约定归入对应模块文件。
5. 实际模块分组和命名规范，需要结合项目目录与业务特点进一步细化。
6. 接口方法需要统一风格、统一入参与统一返回约定，便于 AI 生成与长期维护。
---
## 12. 测试用例组织要求
1. 测试用例使用独立 `.py` 文件组织。
2. 测试用例应围绕业务场景、功能测试用例、用户操作链路组织。
3. 测试用例通过调用已有接口方法拼接形成完整场景。
4. 测试用例仍需按模块分目录管理。
5. 生成结果必须统一接入现有或改造后的接口自动化测试平台框架中。
6. 测试用例应支持参数化、动态变量引用、前后置依赖和清理逻辑。

## 15. AI 在平台中的职责边界
平台需要引入 AI 能力，但 AI 的职责必须被边界化。
### 15.1 AI 适合负责
- 将文档转换为接口方法草稿
- 将功能测试用例转换为自动化测试草稿
- 将抓包结果整理为接口定义与场景草稿
- 补充断言建议
- 辅助识别依赖关系
- 辅助生成测试数据
- 解释生成结果与给出修订建议
### 15.2 AI 不应直接等同于最终正确结果
- AI 生成结果不能默认视为最终可信资产；
- 必须经过规则校验；
- 必须经过人工确认或半自动确认；
- 必须经过测试执行验证。
### 15.3 事实来源优先级
建议固定以下优先级：
- 接口文档 / 功能测试用例 / 抓包记录 / 既有接口资产
- 高于 AI 自由推断
### 15.4 AI 角色定位
AI 在平台内的定位应为：
- 生成器
- 补全器
- 编排器
- 推荐器
- 解释器
- 修订助手
而不是唯一事实源。

## 16. 基于调研明确的产品原则
结合本轮产品与开源项目调研，明确以下原则：
1. **平台定位原则**：平台应定位为“统一接口测试资产平台 + AI 生成/编排能力”，而不是单点代码生成工具。
2. **事实优先原则**：OpenAPI / Swagger / 功能测试用例 / 抓包记录等事实输入优先于 AI 自由推断。
3. **中间模型原则**：所有输入不允许直接跳到最终代码，必须先进入统一中间模型层。
4. **模板生成原则**：接口方法、测试用例、断言、数据文件等生成结果，应优先通过模板化方式生成，而不是完全依赖一次性自由文本生成。
5. **依赖编排原则**：平台高价值能力不仅是单接口生成，更要支持 producer-consumer 依赖识别、动态变量提取与场景链路编排。
6. **分级执行原则**：平台应支持结构校验、规则校验、smoke 执行、深度执行等分级验证能力。
7. **资产沉淀原则**：所有生成物必须保留来源、时间、输入摘要、审核状态、执行状态等元信息。

## 18. 中间模型要求（新增）
后续平台必须建设统一中间模型，用于承接三条输入路线。
### 18.1 中间模型至少应承载
1. 接口定义
2. 接口路径与方法
3. 请求参数结构
4. Header / Cookie / Body 模型
5. 响应结构与断言候选
6. 变量提取点
7. 接口依赖关系
8. 场景步骤与顺序
9. 模块归类信息
10. 生成来源与元信息
### 18.2 建设中间模型的原因
- 统一三条输入路线的承接方式；
- 降低直接生成代码的混乱度；
- 支持模板化生成；
- 支持审核、修订与再生成；
- 支持 Web / Desktop / CLI 多入口复用。
---
## 19. 模板体系要求（新增）
平台后续不应依赖“自由文本直接生成最终代码”，而应建立模板体系。
### 19.1 需建设的模板类型
1. 接口方法模板
2. 测试用例模板
3. 断言模板
4. 测试数据模板
5. 模块索引模板
6. 生成记录模板
7. 报告展示模板
### 19.2 模板体系价值
- 统一风格
- 降低生成风险

- 便于规则校验
- 便于 AI 生成接入

## 21. 外部调研提炼出的可借鉴方向
通过本轮调研，可重点借鉴以下方向：
1. **Postman Agent Mode**
   - 借鉴“上下文注入 + 审批确认 + 自动执行”的 AI 协同模式。
2. **KushoAI**
   - 借鉴“文档/契约驱动 + 持续维护 + 自愈/风险感知”的思路。
3. **ACCELQ / Katalon**
   - 借鉴“统一测试资产平台 + 多角色支持 + 平台化流程”思路。
4. **Schemathesis**
   - 借鉴 schema-first、stateful testing、Python 生态友好的设计。
5. **RESTler**
   - 借鉴 producer-consumer 依赖推断、请求序列生成、分阶段执行模式。
6. **OpenAPI Generator**
   - 借鉴模板化生成体系与生成器/模板解耦思路。
7. **EvoMaster**
   - 借鉴“生成结果即回归资产”、“多协议扩展预留”、“可视化报告”思路。

## 22. 版本规划（阶段性结论）
### 22.1 V1 方向
V1 采用：**A + 明确 V2 路线**。
即：
- V1 优先定位为“需求验证版 / 架构奠基版”；
- 优先证明整体平台路线可行；
- 不追求一次性覆盖三条输入路线的完整能力；
- 明确 V2 扩展方向与边界。
### 22.2 V1 目标范围（初步）
V1 优先聚焦以下内容：
1. 全局产品说明书、总体架构设计说明书、UI 设计说明文档、版本规划文档沉淀；
2. 现有接口自动化测试框架的系统化分析与首轮改造设计；
3. 平台核心架构骨架设计；
4. 明确 Web 与 Windows 客户端统一风格的 UI 设计方向；
5. 面向“接口文档驱动”的最小可行闭环设计；
6. 面向“测试执行验证”的最小可行闭环设计；
7. 建立模板体系与基础规则体系；
8. 建立统一中间模型雏形；
9. 为 AI 生成接口方法与测试用例预留统一生成与校验接口。
### 22.3 V1 不追求一次性完成的内容
V1 原则上不强求：
1. 三条路线全部完整打通；
2. 完整抓包驱动能力；
3. 完整桌面应用使用体验；
4. 完整多人协作 / 中心化平台能力；
5. 面向非技术用户的低门槛闭环。
### 22.4 V2 方向
V2 在 V1 稳定基础上重点扩展：
1. 功能测试用例驱动的场景化生成能力；
2. 抓包驱动的接口识别、依赖提取、场景拼接能力；
3. 更完整的 Web 使用能力；
4. 更完整的 Windows 本地应用封装能力；
5. 更强的审核、预览、修订、确认能力；
6. 更完整的执行调度、报告展示与结果管理能力。
### 22.5 V3 及后续方向
后续版本可进一步考虑：
1. 更成熟的桌面应用体验；
2. macOS 打包支持；
3. 多项目管理能力；
4. 协作与权限管理能力；
5. 面向非技术用户的简化使用模式；
6. 更高级的 AI 自愈、自动修复、差异分析与风险治理能力。

## 23. 当前明确不采用或暂不采纳的需求/方向
为了控制范围和降低早期架构风险，当前阶段不直接采纳以下不合理或过早绑定的要求：
1. **不要求 V1 一次性覆盖三条路线的完整功能闭环**；
2. **不要求 V1 一次性实现完整桌面应用体验**；
3. **不要求 V1 一次性支持多人协作与集中化服务部署**；
4. **不要求当前阶段就为非技术用户设计完整低门槛交互闭环**；
5. **不将 AI 直接定义为最终正确结果来源**；
6. **不采用“自由文本直接生成最终生产代码”作为核心生成策略**；
7. **不把产品定位为简单脚本生成器**。

## 24. 当前结论
当前已形成一致共识：
1. 该项目未来目标是建设“接口自动化测试平台”，而不只是保留为“接口自动化测试框架”。
2. 平台未来需要同时支持文档驱动、功能测试用例驱动、抓包驱动三条路线，但采用分阶段建设策略。
3. 当前阶段优先完成全局需求梳理、产品说明书完善、总体架构设计、UI 设计说明、现有框架改造分析。
4. 后续所有开发工作必须先基于确认后的产品说明书和 UI 设计说明，再按 TDD 模式推进。
5. 平台最终需要同时支持 Web 访问与桌面应用，Windows 优先，架构预留 macOS 扩展能力。
6. 平台核心运行模式先按本地单机闭环设计，同时保留后续服务化扩展空间。
7. Web 服务后端当前确认为 Django + DRF。
8. 数据存储当前确认为 MySQL。
9. 目标用户优先满足测试工程师与开发工程师，后续再逐步尝试推广给非技术用户。
10. 平台应优先定位为“统一接口测试资产平台 + AI 生成/编排能力”。
11. 后续架构应围绕统一中间模型、模板化生成、规则校验、分级执行、资产沉淀展开。
