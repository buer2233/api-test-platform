## 1. 文档说明
- **文档名称**：产品需求说明书（全局）
- **项目名称**：接口自动化测试平台
- **当前阶段**：需求整理与总体架构设计阶段
- **文档状态**：持续完善中
- **维护原则**：后续所有需求核对、范围确认、架构调整、功能拆分，优先直接更新本文件，确保需求信息持续沉淀、可追溯、不丢失。
---
## 2. 项目背景
当前项目内已有一个基于 Python + pytest 的接口自动化测试框架，但仍处于基础框架阶段，距离“接口自动化测试平台”还有较大差距。
主人当前希望建设的，不只是一个接口自动化测试框架，而是一个能够围绕接口文档、功能测试用例、抓包结果等输入，辅助或自动生成接口自动化测试资产，并统一进行编写、执行、管理与扩展的接口自动化测试平台。
未来所有生成的接口自动化测试用例，都应基于该平台进行编写和执行。
---
## 3. 产品目标
建设一个接口自动化测试平台，使其具备以下能力：
1. 能根据用户提供的不同输入信息，生成或辅助生成接口自动化测试代码。
2. 生成的接口自动化测试代码必须统一适配项目内部既定的 Python + requests + pytest 技术体系。
3. 平台既支持自动生成，也支持人工确认、人工补充、半自动生成。
4. 平台最终应支持通过 Web 服务方式访问使用。
5. 平台最终应支持打包为可直接使用的桌面应用程序。
6. 桌面应用优先支持 Windows，架构设计时需提前考虑后续支持 macOS 的打包与运行方式。
7. 平台后续生成的接口方法、测试用例、测试数据、执行结果，应形成统一的资产沉淀与管理方式。
---
## 4. 当前阶段目标
当前阶段**不是直接进入完整功能开发阶段**，而是优先完成以下工作：
1. 梳理与完善整体产品需求。
2. 完善全局产品说明书。
3. 对现有接口自动化测试框架进行分析。
4. 明确未来平台化改造方向。
5. 先完成总体架构设计。
6. 后续在产品说明书确认后，再进入详细测试用例设计与 TDD 开发阶段。
---
## 5. 开发流程要求（强约束）
后续开发工作需要严格遵循以下流程：
1. **整理并确认产品说明书**。
2. **所有开发都必须基于详细的产品说明书进行**。
3. **产品说明书确认后，先按照说明书编写详细测试用例**。
4. **再按照产品说明书和详细测试用例进行开发**。
5. **开发完成后，执行编写的测试用例，所有测试通过才算开发完成**。
6. **最后对照产品说明书，核对开发结果与测试结果是否满足要求**。
7. **整体开发模式要求采用 TDD 思路推进**。
### 5.1 当前达成的一致原则
- 当前优先事项是“需求整理、产品说明书完善、架构设计”，不是直接堆功能。
- 所有后续新功能，都应先确认产品说明书，再进入实现。
- 后续需求讨论内容应优先沉淀到本地文档，防止信息丢失。

## 6. 当前项目现状
当前仓库内已有一个基础接口自动化测试框架，主要特征如下：
1. 技术栈为 Python + requests + pytest。
2. 已具备基础 API 请求封装能力。
3. 已具备基础 pytest 执行能力。
4. 已有简单测试用例示例。
5. 当前更偏“接口自动化测试框架”，尚不是完整的“接口自动化测试平台”。
### 6.1 已知现状判断
当前项目更接近：
- 一个可运行的接口自动化测试框架雏形；
- 适合作为未来平台演进的基础底座；
- 需要在资产管理、生成能力、流程化能力、规则校验能力、平台化能力上进行系统改造。

## 7. 核心需求（当前已明确）
### 7.1 生成目标
平台需要能够直接生成，或按照用户要求辅助生成：
- 基于 Python 的接口方法封装；
- 基于 requests 调用的接口自动化测试代码；
- 基于 pytest 的测试用例；
- 可落地、可执行、可维护的接口自动化测试资产。
### 7.2 接口封装规范要求
1. 每个接口应封装为单独的接口方法。
2. 接口方法应存放在单独的 `.py` 文件中。
3. 接口方法需要按照业务模块进行归类。
4. 类似 `/api/module1`、`/api/bs/module1`、`/api/app/module1` 等接口，应根据项目约定归类到模块文件内。
5. 实际模块分组与命名规范，需要结合测试项目进一步明确。
### 7.3 测试用例组织要求
1. 测试用例应使用单独的 `.py` 文件。
2. 测试用例应基于测试场景、功能测试用例、用户使用场景调用现有接口方法进行拼接。
3. 测试用例仍需按模块区分，分文件管理。
4. 生成结果必须统一接入现有或改造后的接口自动化测试平台框架中。
---
## 8. 输入路线（当前已确认）
平台未来需要同时支持三条输入路线，但开发上分批推进。
### 8.1 路线一：接口文档驱动
适用场景：
- 用户提供 OpenAPI / Swagger / Markdown / 其他接口文档。
- 平台根据接口文档批量生成接口方法。
- 在具备足够信息时，可进一步生成基础测试骨架。
当前判断：
- 这是非常重要的一条路线；
- 适合作为平台自动生成能力的重要基础；
- 对接口文档质量依赖较高。
### 8.2 路线二：功能测试用例驱动
适用场景：
- 用户提供详细功能测试用例或业务场景说明。
- 平台基于已有接口方法，自动或半自动拼接为接口自动化测试用例。
当前判断：
- 这是从“业务场景”到“自动化测试”的关键路线；
- 非常适合测试工程师实际使用；
- 依赖接口资产层先沉淀稳定。
### 8.3 路线三：抓包驱动
适用场景：
- 没有完整文档或完整测试用例；
- 通过代理抓包记录用户页面操作或客户端操作请求；
- 平台基于抓包过程提取接口、参数、调用顺序、依赖关系；
- 进一步生成接口方法和完整测试用例。
当前判断：
- 这是非常重要的一条补充路线；
- 特别适合没有完整文档、但有真实业务流程的项目；
- 后续实现时需要重点解决抓包清洗、去重、参数归一化、动态值识别、依赖提取等问题。

## 9. 产品形态要求（当前已确认）
### 9.1 最终产品形态
平台最终需要同时支持：
1. **Web 服务访问使用**
   - 启动服务后，可通过浏览器直接访问使用。
2. **桌面应用程序使用**
   - 可打包为可直接使用的应用程序。
   - 优先满足 Windows 系统使用。
   - 后续尝试支持 macOS。
### 9.2 架构约束
在架构设计时，必须提前考虑：
- 核心引擎与表现层解耦；
- 能同时支撑 Web 形态与桌面应用形态；
- Windows 打包优先；
- 后续扩展 macOS 打包时，尽量不重写核心业务逻辑。
---
## 10. 运行模式要求（当前已确认）
平台最终需要同时支持：
1. 本地单机模式；
2. 后续服务化/集中化扩展能力。
当前阶段明确采用：
- **先按本地单机模式设计核心能力**；
- 后续预留中心化服务能力扩展空间。
### 10.1 本地单机模式优先的原因
当前平台的核心能力天然依赖本地环境，包括但不限于：
- 本地项目目录读写；
- 本地代码生成；
- 本地 pytest 执行；
- 本地代理抓包；
- 本地应用打包；
- 本地调试与问题定位。
因此前期设计时，优先围绕本地闭环设计更合理。

## 11. 目标用户（当前已确认）
### 11.1 优先用户
优先满足：
- **测试工程师**
- **开发工程师**

- 产品人员

但非技术用户的使用方式，后续应建立在：

- 生成结果仍可由测试/开发人员复核。

## 12. 平台总体能力方向（当前初步结论）
为了同时支撑三条输入路线，平台后续建议具备以下能力层：
1. **输入接入层**
   - 文档输入
   - 功能测试用例输入
   - 抓包输入
2. **解析与标准化层**
   - 将不同来源的信息转换为统一的中间数据结构
   - 统一接口定义、参数定义、依赖关系、业务场景步骤、断言候选信息
3. **资产管理层**
   - 管理接口方法资产
   - 管理测试用例资产
   - 管理测试数据
   - 管理生成记录与元数据
4. **AI 生成编排层**
   - 根据输入信息生成接口方法与测试用例草稿
   - 补全断言
   - 建议命名、模块归类、依赖拼接
5. **规则校验层**
   - 校验生成结果是否符合项目规范
   - 校验目录结构、命名、引用关系、代码模板、参数合法性
6. **执行验证层**
   - 执行 pytest
   - 生成执行报告
   - 将“生成完成”提升为“可执行且验证通过”
> 说明：以上为当前阶段的初步总体方向，后续需要继续细化成正式架构设计章节。
---
## 13. AI 在平台中的职责边界（当前初步结论）
平台应引入 AI 能力，但 AI 的职责应明确边界：
### 13.1 AI 适合负责
- 将文档转换为接口方法草稿；
- 将功能测试用例转换为自动化测试草稿；
- 将抓包结果整理为接口定义与场景草稿；
- 补充断言建议；
- 辅助识别依赖关系；
- 辅助生成测试数据与场景描述。
### 13.2 AI 不应直接等同于最终正确结果
- AI 生成结果不能默认视为最终可信资产；
- 仍需经过规则校验；
- 仍需经过人工确认或半自动确认；
- 仍需通过测试执行验证。
### 13.3 当前建议原则
- **事实来源优先级**：接口文档 / 功能测试用例 / 抓包记录 > AI 推断
- **AI 角色定位**：生成器、补全器、编排器、推荐器
- **最终结果标准**：以可审查、可执行、可验证为准

3. 明确模块目录规范；

6. 增强断言模板与公共断言能力；
7. 增强动态参数提取与传递规范；
8. 增加生成记录与可追溯能力；
9. 提升项目结构对 AI 生成与维护的友好性；
10. 为后续平台化能力预留接口。

## 15. 外部调研得到的可借鉴方向（当前初步结论）
通过在线调研，目前可借鉴的行业实现方向包括：
1. **规范驱动**（OpenAPI / Swagger）
   - 可借鉴自动生成接口定义、测试骨架、参数结构、schema 校验等思路。
2. **状态流 / 依赖推断**
   - 可借鉴 producer-consumer 依赖识别、多接口场景拼接、状态机测试等思路。
3. **模糊 / 属性测试**
   - 可借鉴边界值、随机化输入、异常输入探索能力。
4. **抓包录制 / 回放**
   - 可借鉴录制真实请求、重放、清洗、结构化提取、动态值替换等思路。
5. **LLM 辅助生成**
   - 可借鉴“文档/用例/抓包 → 代码草稿”的生成模式，但必须纳入规则校验和测试验证流程。

## 16. 版本规划（当前已确认的阶段性结论）
### 16.1 V1 方向
V1 采用：**A + 明确 V2 路线**。
即：
- V1 优先定位为“需求验证版 / 架构奠基版”；
- 优先证明整体平台路线可行；
- 不追求一次性覆盖三条输入路线的完整能力；
- 明确 V2 的扩展方向与演进边界。
### 16.2 V1 目标范围（初步）
V1 优先聚焦以下内容：
1. 全局产品说明书、架构设计说明书、版本规划文档建立与沉淀；
2. 现有接口自动化测试框架的系统化分析与首轮改造设计；
3. 平台核心架构骨架设计；
4. 面向“接口文档驱动”的最小可行闭环设计；
5. 面向“测试执行验证”的最小可行闭环设计；
6. 为 AI 生成接口方法与测试用例预留统一生成与校验接口；
7. 明确平台目录结构、资产结构、模板结构、生成规则基础规范。
### 16.3 V1 不追求一次性完成的内容（初步）
V1 阶段原则上不强求一次性完整实现：
1. 三条路线全部打通；
2. 完整抓包驱动能力；
3. 完整桌面应用打包体验；
4. 完整多人协作/服务化平台能力；
5. 非技术用户友好化交互闭环。
### 16.4 V2 方向（当前已明确需要预留）
V2 需要在 V1 稳定基础上，重点扩展：
1. 功能测试用例驱动的场景化生成能力；
2. 抓包驱动的接口识别、依赖提取、用例拼接能力；
3. 更完整的 Web 使用能力；
4. 更完整的本地应用封装能力（优先 Windows）；
5. 更强的审核、预览、修订、确认能力；
6. 更完整的执行调度、报告展示与结果管理能力。
### 16.5 V3 及后续方向（初步）
后续版本可进一步考虑：
1. 更成熟的桌面应用体验；
2. macOS 打包支持；
3. 更强的多项目管理能力；
4. 更强的协作与权限管理能力；
5. 面向非技术用户的低门槛使用模式；
6. 更高级的 AI 自愈、自动修复、风险分析与生成治理能力。

## 17. 外部产品调研补充（AI 自动化测试 / AI 驱动接口测试）
### 17.1 本轮调研目标
基于已确认的产品方向，进一步调研市面上“AI 自动化测试”和“AI 驱动接口自动化测试”相关产品，用于补充本平台的设计参考。
### 17.2 本轮调研中观察到的典型产品方向
#### （1）Postman Agent Mode
观察到的方向：
- 强调使用自然语言完成 API 调试、探索、文档改进与工作流加速；
- 强调 AI 与现有 API 平台深度集成；
- 支持上下文注入、审批、自动执行、工具链集成。
可借鉴点：
- AI 不是单点功能，而是嵌入整个 API 生命周期；
- 强调“上下文选择 + 审批确认 + 自动执行”的组合模式；
- 非常适合作为本平台后续 AI 交互层设计参考。
#### （2）KushoAI
观察到的方向：
- 支持从 OpenAPI 规范或 Postman Collection 自动生成测试；
- 强调契约测试、持续安全扫描、全链路验证、自愈能力；
- 强调 CI/CD 集成与持续维护。
可借鉴点：
- 文档/规范驱动是重要切入点；
- 生成后不仅要执行，还要考虑持续维护与自愈；
- 可作为后续“契约测试 + 持续测试 + 生成后治理”方向参考。
#### （3）ACCELQ
观察到的方向：
- 强调 AI 驱动、无代码/低代码、跨 Web / API / Desktop / Backend 的统一平台；
- 强调测试设计、执行、管理、协作的一体化平台能力。
可借鉴点：
- 平台不是单纯代码生成器，而应考虑统一测试资产与流程管理；
- 支持不同角色协作是平台化的重要方向；
- 可作为后续“测试资产统一管理与多角色协作”参考。
#### （4）Katalon
观察到的方向：
- 强调 no-code / low-code / full-code 并存；
- 强调 AI 参与需求分析、测试设计、执行优化、结果分析；
- 强调覆盖 Web、移动端、API、桌面端和生产后洞察。
可借鉴点：
- 平台未来应允许不同技术深度的用户共存；
- AI 不应只服务于生成代码，也应服务于测试设计、执行与结果分析；
- 可作为“分层用户能力设计”参考。
### 17.3 本轮调研提炼出的共性模式
本轮调研显示，主流产品通常具备以下共性：
1. **以现有测试资产或接口资产为基础，而非完全从零凭空生成**；
2. **强调自然语言 + 上下文注入 + 审批确认的协同模式**；
3. **强调契约/规范驱动能力**；
4. **强调自动生成后的执行验证、维护、自愈与持续集成**；
5. **强调测试资产统一管理，而不是孤立生成测试代码**；
6. **强调从专业用户起步，再逐步扩展到低门槛用户**。
### 17.4 对本平台的直接启发
结合本项目目标，当前可明确以下启发：
1. 本平台应优先做“统一测试资产平台”，而不是只做“代码生成器”；
2. AI 生成功能需要与上下文管理、审批确认、执行验证联动；
3. 文档驱动路线适合作为 V1/V2 的重要主线；
4. 抓包驱动和场景驱动更适合作为后续增强能力逐步接入；
5. 后续应预留“持续维护 / 自愈 / 差异分析 / 结果洞察”能力扩展位。
### 17.5 基于本轮调研新增明确的产品原则
结合本轮产品与开源项目调研，新增以下明确原则：
1. **平台定位原则**：平台应定位为“统一接口测试资产平台 + AI 生成/编排能力”，而不是单点代码生成工具。
2. **事实优先原则**：OpenAPI / Swagger / 功能测试用例 / 抓包记录等事实输入优先于 AI 自由推断。
3. **中间模型原则**：不允许输入直接跳到最终代码，后续必须建设统一中间模型层，用于承接接口定义、场景步骤、变量绑定、断言候选与依赖关系。
4. **模板生成原则**：接口方法、测试用例、断言、数据文件等生成结果，应优先通过模板化方式生成，而不是完全依赖一次性自由文本生成。
5. **依赖编排原则**：平台的高价值能力不仅是单接口生成，更要支持 producer-consumer 依赖识别、动态变量提取与场景链路编排。
6. **分级执行原则**：平台后续需要支持结构校验、规则校验、smoke 执行、深度执行等分级验证能力。
7. **资产沉淀原则**：所有生成物需要保留来源、生成时间、输入摘要、审核状态、执行状态等元信息，形成可追溯资产。

## 19. 当前结论
当前已形成一致共识：
1. 该项目未来目标是建设“接口自动化测试平台”，而不只是保留为“接口自动化测试框架”。
2. 平台未来需要同时支持文档驱动、功能测试用例驱动、抓包驱动三条路线。
3. 当前阶段优先完成全局需求梳理、产品说明书完善、总体架构设计、现有框架改造分析。
4. 后续所有开发工作必须先基于确认后的产品说明书，再按 TDD 模式推进。
5. 平台最终需要同时支持 Web 访问与桌面应用，Windows 优先，架构预留 macOS 扩展能力。
6. 平台核心运行模式先按本地单机闭环设计，同时保留后续服务化扩展空间。
7. 目标用户优先满足测试工程师与开发工程师，后续再逐步尝试推广给非技术用户。
8. 平台应优先定位为“统一接口测试资产平台 + AI 生成/编排能力”。
9. 后续架构应围绕中间模型、模板化生成、规则校验、分级执行、资产沉淀展开。
