# 总体架构设计说明书
## 1. 文档说明
- **文档名称**：总体架构设计说明书
- **所属项目**：接口自动化测试平台
- **文档目标**：用于明确平台的总体架构、开发方式、技术选型、模块划分、数据流、运行形态、版本演进与实施边界，为后续详细设计、测试设计和 TDD 开发提供基础依据。
- **当前阶段**：总体架构设计阶段
- **适用范围**：V1 架构设计、V2 扩展预留、V3 演进方向参考
---
## 2. 架构设计目标
本说明书面向以下核心目标：
1. 将当前“接口自动化测试框架”演进为“接口自动化测试平台”。
2. 支持文档驱动、功能测试用例驱动、抓包驱动三条输入路线。
3. 建立统一中间模型，统一承接多种输入。
4. 通过模板化生成、规则校验、执行验证和资产沉淀，形成平台闭环。
5. 支持本地单机优先运行模式，同时预留后续 Web / 服务化扩展能力。
6. 最终支撑 Web 服务访问与桌面应用（Windows 优先，后续支持 macOS）。
---
## 3. 架构设计原则
## 3.1 平台化原则
平台不是简单的代码生成工具，而是统一测试资产平台。设计时要优先考虑：
- 资产沉淀
- 流程闭环
- 可追溯
- 可扩展
- 可复用
## 3.2 本地优先原则
当前阶段优先以本地单机模式为核心，确保：
- 本地目录读写稳定
- 本地 pytest 执行稳定
- 本地抓包可行
- 本地应用打包可控
## 3.3 事实优先原则
OpenAPI / Swagger / 功能测试用例 / 抓包记录 / 既有接口资产等事实输入，高于 AI 推断。
## 3.4 中间模型原则
所有输入必须先转为统一中间模型，再进入生成与执行链路。
## 3.5 模板化原则
所有代码生成应优先依赖模板体系和规则体系，而不是直接自由文本生成最终代码。
## 3.6 分层解耦原则
核心引擎、输入解析、生成编排、执行验证、资产管理、展示层必须分层设计，避免强耦合。
## 3.7 分级执行原则
生成结果应支持结构校验、规则校验、smoke 执行、场景执行、深度执行等分级验证模式。
## 3.8 可演进原则
V1 聚焦架构奠基与最小闭环，但设计上必须为 V2 / V3 的抓包驱动、场景编排、桌面应用和 Web 能力预留扩展位。
---
## 4. 平台总体定位
平台定位为：
> **统一接口测试资产平台 + AI 生成/编排能力 + 规则校验与执行验证闭环**
平台未来要管理的不只是代码，还包括：
- 接口资产
- 测试用例资产
- 测试数据资产
- 断言资产
- 生成记录
- 执行结果
- 报告资产
- 审核记录
---
## 5. 总体架构概览
建议平台采用以下总体分层：
1. **接入层（Input Layer）**
2. **解析标准化层（Parsing & Normalization Layer）**
3. **统一中间模型层（Unified Intermediate Model Layer）**
4. **生成编排层（Generation & Orchestration Layer）**
5. **模板与规则层（Template & Rule Layer）**
6. **资产管理层（Asset Management Layer）**
7. **执行验证层（Execution & Validation Layer）**
8. **报告与分析层（Reporting & Analysis Layer）**
9. **交互层（Web / Desktop / CLI）**
10. **基础设施层（Storage / Config / Runtime / Packaging）**
---
## 6. 各层详细设计
## 6.1 接入层
### 6.1.1 目标
负责接收用户输入，并统一进入后续解析链路。
### 6.1.2 输入类型
1. OpenAPI / Swagger 文档
2. Markdown / 文本文档形式的接口说明
3. 功能测试用例文档
4. 抓包结果
5. 现有接口代码 / 测试代码资产
6. 用户自然语言补充说明
### 6.1.3 设计要求
- 每类输入都要有标准入口；
- 输入必须保留来源信息；
- 输入接入层不负责直接生成代码；
- 接入层仅做接收、校验、预处理和路由。
---
## 6.2 解析标准化层
### 6.2.1 目标
将不同来源的输入转换为统一可处理的结构化数据。
### 6.2.2 主要职责
1. 文档解析
2. OpenAPI / Swagger 解析
3. 用例文本解析
4. 抓包数据解析
5. 路径、方法、参数、响应结构提取
6. 依赖候选提取
7. 断言候选提取
8. 模块归类候选提取
### 6.2.3 设计要求
- 不直接输出最终代码；
- 输出统一的标准化结果；
- 对异常输入进行错误提示或回退；
- 为后续 AI 编排提供明确上下文。
---
## 6.3 统一中间模型层
### 6.3.1 目标
统一承接多种输入路线，是平台核心枢纽。
### 6.3.2 为什么必须有中间模型
如果没有统一中间模型，三条输入路线会分别走不同逻辑，导致：
- 生成代码风格不一致；
- 审核与校验难统一；
- Web / Desktop / CLI 无法共用核心；
- 后续修改成本极高。
### 6.3.3 中间模型建议包含的实体
#### 1. API Module
- 模块名称
- 模块路径
- 模块分组标签
#### 2. API Operation
- 接口名称
- 方法（GET/POST/PUT/DELETE 等）
- 路径
- 描述
- 请求头
- Query 参数
- Path 参数
- Body 结构
- 鉴权要求
- 响应结构
#### 3. Variable Binding
- 提取来源
- 提取表达式
- 变量名
- 作用范围
#### 4. Assertion Candidate
- 断言对象
- 断言类型
- 断言值候选
- 来源说明
#### 5. Test Scenario
- 场景名称
- 场景说明
- 场景步骤顺序
- 前置依赖
- 清理步骤
#### 6. Generation Metadata
- 输入来源
- 生成时间
- 生成策略
- 审核状态
- 执行状态
- 版本标识
### 6.3.4 中间模型要求
- 可序列化
- 可被模板层消费
- 可被审核与修订
- 可被多端复用
---
## 6.4 生成编排层
### 6.4.1 目标
负责把中间模型转换为可生成资产的结构，并调度 AI / 规则 / 模板流程。
### 6.4.2 主要职责
1. 接口方法生成编排
2. 测试用例生成编排
3. 场景步骤拼接
4. 断言建议补全
5. 命名建议
6. 模块归类建议
7. 动态变量引用建议
### 6.4.3 AI 在本层中的角色
- 生成草稿
- 补全不完整字段
- 解释生成结果
- 提供修订建议
- 辅助依赖推断
### 6.4.4 设计要求
- AI 输出不能直接落盘为最终生产资产；
- 必须进入模板与规则层；
- 所有生成过程必须保留元数据。
---
## 6.5 模板与规则层
### 6.5.1 目标
保证生成结果统一、稳定、可维护。
### 6.5.2 模板类型
1. 接口方法模板
2. 测试用例模板
3. 断言模板
4. 测试数据模板
5. 模块索引模板
6. 生成记录模板
7. 报告模板
### 6.5.3 规则类型
1. 命名规范规则
2. 模块归档规则
3. 参数规范规则
4. 变量命名规则
5. 引用关系规则
6. 断言规范规则
7. 目录结构规则
### 6.5.4 设计要求
- 模板应与生成逻辑解耦；
- 规则应集中管理；
- 生成资产必须先过规则校验再落盘；
- 后续支持模板升级与规则版本化。
---
## 6.6 资产管理层
### 6.6.1 目标
统一管理平台产出的所有资产。
### 6.6.2 管理对象
1. 接口方法资产
2. 测试用例资产
3. 测试数据资产
4. 断言资产
5. 生成记录
6. 审核记录
7. 执行记录
8. 测试报告
### 6.6.3 每类资产建议记录的元信息
- 资产名称
- 资产类型
- 来源
- 生成时间
- 生成版本
- 最近修改时间
- 审核状态
- 执行状态
- 所属模块
### 6.6.4 设计要求
- 资产应可检索；
- 资产应可追溯；
- 资产应可重生成；
- 资产应支持后续 diff / 版本演进。
---
## 6.7 执行验证层
### 6.7.1 目标
将“生成完成”提升为“执行通过并验证有效”。
### 6.7.2 执行分级建议
1. 结构校验
2. 规则校验
3. 生成代码校验
4. smoke 执行
5. 场景执行
6. 深度执行 / 回归执行
### 6.7.3 主要职责
- 调用 pytest
- 控制执行范围
- 输出执行结果
- 回传失败信息
- 为后续 AI 修订提供上下文
### 6.7.4 设计要求
- 执行能力与 UI 解耦；
- 支持本地运行；
- 后续预留服务化执行能力。
---
## 6.8 报告与分析层
### 6.8.1 目标
统一展示执行结果、生成效果和问题分析信息。
### 6.8.2 主要职责
1. 报告生成
2. 执行结果可视化
3. 失败分类
4. 差异分析
5. 后续风险/自愈能力预留
### 6.8.3 V1 要求
V1 不要求做到非常复杂，但至少应支持：
- 本地执行结果展示
- 用例通过/失败统计
- 失败详情查看
- 报告文件归档
---
## 6.9 交互层
### 6.9.1 组成
1. CLI / 本地命令入口
2. Web 界面
3. Desktop 应用界面
### 6.9.2 设计原则
- 交互层不直接实现核心业务；
- 所有业务通过核心引擎 / 应用服务层调用；
- 多入口共用同一套核心逻辑。
---
## 6.10 基础设施层
### 6.10.1 组成
- 本地文件系统
- 配置管理
- 日志管理
- Python 运行环境
- 应用打包机制
- 可选本地数据库 / 索引存储
### 6.10.2 要求
- V1 优先用轻量本地方案；
- 后续扩展服务化时逐步增强。
---
## 7. 核心工作流设计
## 7.1 文档驱动工作流
1. 用户导入 OpenAPI / 接口文档
2. 解析标准化
3. 转换为中间模型
4. AI 补全 / 编排
5. 模板生成接口方法与基础测试骨架
6. 规则校验
7. 落盘生成资产
8. 执行 smoke 测试
9. 生成结果与报告沉淀
## 7.2 功能测试用例驱动工作流
1. 用户提供功能测试用例 / 场景说明
2. 解析场景步骤
3. 绑定已有接口资产
4. 推导依赖与变量关系
5. 生成场景测试用例草稿
6. 校验与确认
7. 执行与沉淀
## 7.3 抓包驱动工作流
1. 采集抓包数据
2. 清洗与去重
3. 提取接口与顺序
4. 识别动态参数与依赖链
5. 转中间模型
6. 生成接口方法与场景骨架
7. 规则校验与确认
8. 执行验证与沉淀
---
## 8. 技术选型建议
## 8.1 核心语言与执行体系
建议继续以 **Python** 作为平台核心实现语言。
### 原因
1. 当前已有 Python 接口测试框架基础。
2. requests + pytest 生态成熟。
3. 便于快速接入文档解析、测试执行、模板渲染、AI 编排。
4. 有利于与当前项目已有资产复用。
---
## 8.2 测试执行层技术
建议继续使用：
- pytest
- requests
- pytest-html / 后续可评估 Allure
### 原因
- 与现有项目兼容；
- 测试组织与执行成本低；
- 后续平台化时仍可持续复用。
---
## 8.3 文档解析能力
建议：
- OpenAPI / Swagger 解析优先使用 Python 生态库或自研轻量解析封装；
- Markdown / 文本说明优先做结构化解析层；
- 不建议直接把文档原文塞给生成器作为唯一依据。
---
## 8.4 模板引擎
建议：
- 使用 **Jinja2** 或同类模板引擎构建代码模板体系。
### 原因
- Python 生态成熟；
- 可维护；
- 易读；
- 方便多模板扩展。
---
## 8.5 配置与数据存储
### V1 建议
- 配置优先采用本地配置文件；
- 资产元数据可先采用本地文件 + 轻量索引方式；
- 如需结构化存储，可评估 SQLite 作为本地元数据存储。
### 原因
- 本地单机优先；
- 部署成本低；
- 适合 Windows 应用封装。
---
## 8.6 Web 技术选型建议
### 后端建议
优先建议：
- **Python + FastAPI**
### 原因
1. 与核心引擎同语言，减少切换成本；
2. API 定义清晰；
3. 容易提供本地服务供 Web / Desktop 共用；
4. 有利于未来服务化扩展。
### 前端建议
优先建议：
- **React + TypeScript**
### 原因
1. 生态成熟；
2. 后续同时支持 Web 与 Desktop 容易复用；
3. 适合复杂表单、流程、资产管理与报告展示。
---
## 8.7 Desktop 技术选型建议
### 总体建议
桌面应用建议采用：
- **前端壳 + 本地服务内核** 的方式设计。
### 桌面壳候选
1. **Tauri**
2. **Electron**
### 选型建议
#### 推荐优先评估：Tauri
原因：
- 打包更轻量；
- 性能和资源占用更好；
- 更适合作为本地工具型应用。
#### 若后续对生态要求更强可评估：Electron
原因：
- 生态成熟；
- 前端开发资料多；
- 包体更大、资源占用更高。
### 架构约束
无论最终选择哪种壳，核心业务不应写死在桌面层，而应调用统一核心引擎 / 本地服务。
---
## 8.8 抓包能力选型建议
V1 暂不要求完整实现抓包驱动，但架构上建议：
- 预留抓包适配层；
- 将抓包解析独立为单独模块；
- 不把抓包逻辑耦合进核心生成器。
---
## 9. 推荐系统架构形态
## 9.1 V1 推荐形态
> **Python 核心引擎 + 本地文件系统资产管理 + pytest 执行 + 文档驱动最小闭环 + 可扩展 Web/API 外壳**
### V1 的关键特点
- 本地单机闭环
- 底座先稳
- 文档驱动优先
- 模板化生成优先
- 为 Web / Desktop 预留接口
## 9.2 V2 推荐形态
在 V1 基础上扩展：
- FastAPI 应用服务层
- Web 界面
- 更完整的资产管理
- 功能测试用例驱动 / 抓包驱动接入
- Windows 应用封装增强
## 9.3 V3 推荐形态
进一步扩展：
- macOS 支持
- 更完整的协作能力
- 更强的分析与自愈能力
---
## 10. 目录与代码组织建议
以下为建议方向，具体路径后续可再细化：
1. `core/`：核心引擎、生成编排、规则校验
2. `parsers/`：文档、用例、抓包解析器
3. `models/`：中间模型定义
4. `templates/`：模板体系
5. `assets/`：资产管理相关逻辑
6. `executors/`：pytest 执行与结果采集
7. `reports/`：报告生成与处理
8. `services/`：应用服务层 / API 层
9. `ui/`：Web / Desktop 前端（后续）
10. `product_document/`：产品与架构文档

## 11. 如何开发（研发策略）
## 11.1 开发方法论
平台后续开发必须遵循：
1. 产品说明书确认
2. 详细测试用例设计
3. TDD 开发
4. 测试通过
5. 对照说明书验收
## 11.2 V1 开发策略
建议采用：
- **先改造底座，再接文档驱动最小闭环**
即：
1. 先改造现有测试框架，使其适合承接 AI 生成资产；
2. 再实现文档驱动最小闭环；
3. 最后再逐步接入更复杂能力。
## 11.3 实施顺序建议
建议总体顺序：
1. 定义中间模型
2. 定义模板体系
3. 定义规则体系
4. 改造现有框架目录与公共层
5. 实现文档解析最小闭环
6. 实现接口方法生成最小闭环
7. 实现基础测试用例生成最小闭环
8. 接入 pytest 执行闭环
9. 接入结果沉淀
---
## 12. 测试策略建议
## 12.1 需要测试的层次
1. 解析层测试
2. 中间模型测试
3. 模板渲染测试
4. 规则校验测试
5. 生成器测试
6. 执行器测试
7. 集成流程测试
## 12.2 测试原则
- 所有功能先写测试再实现；
- 单元测试覆盖核心逻辑；
- 集成测试覆盖最小闭环；
- 生成结果至少经 smoke 执行验证。
---
## 13. 风险与控制点
## 13.1 风险一：AI 生成不稳定
控制方式：
- 引入模板体系与规则校验；
- AI 仅生成草稿或结构化补全。
## 13.2 风险二：三条路线同时推进导致失控
控制方式：
- V1 聚焦文档驱动；
- V2 再扩展用例驱动与抓包驱动。
## 13.3 风险三：现有框架不适合承接平台化
控制方式：
- V1 先做底座改造设计；
- 明确目录、模板、规则与中间模型。
## 13.4 风险四：桌面与 Web 双端耦合过重
控制方式：
- 核心引擎与 UI 解耦；
- 统一本地服务 / 应用服务层。
## 13.5 风险五：范围膨胀
控制方式：
- V1 严控范围；
- 所有需求变更先更新产品说明书。
