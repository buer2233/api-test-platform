## 2. V2 阶段定位
V2 不是继续补 V1 的零散增强项，而是：
> **从“文档驱动最小闭环”升级到“场景驱动优先、服务化落地、可用型产品入口承接”的能力增强阶段**

V2 的核心价值不是一次性做成完整平台，而是：
1. 正式落地 V1 预留的场景、变量、依赖和审核能力；
2. 让功能测试用例驱动成为第一主线；
3. 让抓包驱动以“草稿化接入”方式进入正式阶段规划；
4. 让 Django + DRF + MySQL 从预留方向升级为正式承载层；
5. 让 Web / Windows 入口具备可用型流程，能够承接导入、预览、确认、执行和结果查看；
6. 为 V3 的平台化深化、协作治理和产品体验升级打下稳定基础。
---
## 3. V2 进入条件与承接关系
### 3.1 进入 V2 的前提
截至 2026-04-03，V1 已完成正式验收，已满足进入 V2 的前提条件：
1. `platform_core` 已完成文档驱动最小闭环；
2. `api_test` 已完成通用化与公开基线治理；
3. 中间模型、模板、规则、执行与资产摘要已形成 V1 稳定底座；
4. `run` / `inspect` 的服务摘要契约已稳定；
5. V1 文档、测试说明书和正式验收报告已同步完成。

### 3.2 V2 直接承接的 V1 成果
V2 直接复用以下 V1 成果，不重新推翻：
1. 文档驱动解析与中间模型底座；
2. 接口方法模板、基础 pytest 模板、断言模板和生成记录模板；
3. 资产清单、执行记录、服务摘要和工作区检查能力；
4. 公开基线测试站点与 `api_test` 通用底座；
5. Web / Windows 共用核心逻辑、前后端分层和本地优先原则。

### 3.3 V2 与 V1 / V3 的边界
1. V1 已完成，后续不再把新增能力继续记在 V1 名下；
2. V2 重点是场景能力、服务化落地和可用型入口；
3. V3 重点是平台化深化、协作治理、完整产品体验、抓包完整执行闭环和高级 AI 能力。
---
## 4. V2 总目标
V2 需要完成的核心目标如下：
1. 正式落地 `TestScenario`、`ScenarioStep`、`VariableBinding`、`DependencyLink`、`ReviewRecord` 等核心模型；
2. 打通“功能测试用例驱动 -> 场景草稿 -> 审核确认 -> 正式场景资产 -> 执行验证”的主闭环；
3. 打通“抓包输入 -> 清洗归一化 -> 场景草稿 -> 审核确认”的草稿化副闭环；
4. 建立审核、预览、修订、确认的资产生命周期机制；
5. 建立 Django + DRF + MySQL 的正式服务承载层，使数据库成为核心事实源；
6. 明确本地工作区作为导入导出、生成落盘和 pytest 执行载体的定位；
7. 建立可用型 Web / Windows 入口，承接导入、预览、确认、执行和结果查看；
8. 扩展模板体系、规则体系、执行层和报告层，以支持场景化测试；
9. 建立 V2 的详细测试文档、进度记录、风险记录和验收标准；
10. 为 V3 的多项目、协作治理和产品化深化预留稳定边界。
---
## 5. V2 不做什么
为了控制范围，V2 当前明确不要求完成以下内容：
1. 不把抓包驱动完整执行闭环作为阶段必达项；
2. 不做完整成熟的 Web 产品化体验；
3. 不做多人协作、权限体系和中心化组织管理；
4. 不做完整的桌面产品分发、升级和运维体系；
5. 不做 macOS 支持；
6. 不做高级 AI 自愈、自动修复和大规模自治编排；
7. 不把 AI 自由改写结果直接落为正式资产；
8. 不把 V2 扩张为 V3 的替代版本。
---
## 6. V2 正式闭环定义
V2 的完成，不定义为“三条输入路线全部成熟”，而定义为以下闭环同时成立：

### 6.1 功能测试用例驱动正式闭环
1. 输入功能测试用例或场景描述；
2. 解析并标准化；
3. 生成场景草稿；
4. 建立变量与依赖关系；
5. 进入预览、审核与修订流程；
6. 确认为正式场景资产；
7. 导出工作区并执行；
8. 结果回写数据库与本地执行产物。

### 6.2 抓包驱动草稿化闭环
1. 导入抓包数据；
2. 清洗、去重、归一化；
3. 提取接口、顺序和动态值候选；
4. 生成场景草稿；
5. 进入审核确认链路。

### 6.3 可用型入口闭环
1. Web / Windows 入口支持导入；
2. 支持草稿预览；
3. 支持审核确认；
4. 支持触发执行；
5. 支持查看结果摘要。

## 8. V2 详细设计方案
## 8.1 场景驱动核心模型设计方案
### 核心要求
1. 场景、步骤、变量、依赖、审核对象必须结构化落地；
2. 需同时可被模板层、规则层、服务层、数据库层和入口层消费；
3. 保留来源、状态、版本和审核元信息。

### 设计收益
1. 让功能测试用例驱动从描述提升为正式资产；
2. 让抓包驱动具备统一落点；
3. 让服务化和产品入口有稳定事实模型可依赖。

## 8.2 功能测试用例驱动设计方案
### 核心要求
1. 功能测试用例驱动为 V2 第一主线；
2. 输入不能直接跳到 pytest 代码，必须先转为场景模型；
3. 优先复用 V1 已生成的接口资产和服务摘要能力；
4. 场景草稿必须经过审核确认后再成为正式资产。

### 设计收益
1. 把业务测试场景正式接入平台；
2. 提升测试工程师的直接使用价值；
3. 为依赖编排、变量提取和执行调度提供正式入口。

## 8.3 抓包驱动设计方案
### 核心要求
1. 抓包驱动在 V2 先做草稿化接入；
2. 抓包数据必须先经过清洗、去重、归一化；
3. 不允许抓包结果直接产出正式资产；
4. 生成的场景草稿必须进入审核确认链路。

### 设计收益
1. 为无完整文档场景提供事实输入来源；
2. 为 V3 的抓包完整闭环做准备；
3. 避免把脏流量直接带入正式资产体系。

## 8.4 审核、预览、修订与确认设计方案
### 核心要求
1. 默认可审核，但关键节点必须确认；
2. 同时支持结构化修订和 AI 辅助改写；
3. AI 输出必须经过规则校验和确认门禁；
4. 所有状态变更必须留痕。

### 设计收益
1. 平衡效率与治理；
2. 避免 AI 越权成为事实源；
3. 为后续多人协作和权限体系预留基础。

## 8.5 服务化与数据持久化设计方案
### 核心要求
1. Django + DRF + MySQL 为阶段必达项；
2. 数据库成为核心资产与记录的事实源；
3. 本地工作区继续保留，但降为导入导出与执行载体；
4. 服务层与入口层保持严格解耦。

### 设计收益
1. 让 V2 真正进入平台化落地阶段；
2. 为 V3 的多项目和协作治理铺路；
3. 降低后续入口形态变化对核心能力的冲击。

## 8.6 可用型 Web / Windows 入口设计方案
### 核心要求
1. 入口只做到可用型，不追求完整产品化体验；
2. Web 与 Windows 的流程语言保持一致；
3. 所有能力优先调用服务层，不直接耦合核心引擎。

### 设计收益
1. 让 V2 有可演示、可使用的正式入口；
2. 保持研发重点仍在核心能力；
3. 为 V3 的产品体验深化留下空间。

1. 场景驱动能力可正式落地；

## 16. V3 承接建议
V3 建议重点承接以下内容：
1. 多项目管理能力；
2. 协作与权限管理；
3. 抓包驱动从“草稿化接入”升级为“正式执行闭环”；
4. 更完整的 Web 产品体验与 Windows 客户端体验；
5. macOS 支持；
6. 更成熟的执行调度中心、差异分析与治理能力；
7. 更高级的 AI 修订建议、自愈与自动修复能力。

V2 与 V3 的边界应保持：
1. V2 重点是核心能力成型、服务化落地和可用型入口承接；
2. V3 重点是平台化深化、协作治理和产品体验升级；
3. 凡涉及多人协作、权限管理、完整产品体验、抓包完整执行闭环、macOS 支持和高级 AI 自愈的内容，原则上归入 V3。

## 17. 结论
V2 不是继续补 V1 的零散增强项，而是正式进入：
> **场景驱动能力建设 + 服务化落地 + 可用型入口承接**

因此，当前 V2 阶段工作的主线应明确为：
1. 功能测试用例驱动优先；
2. 抓包驱动草稿化接入；
3. 审核、修订、确认成为正式生命周期；
4. 数据库成为事实源；
5. Web / Windows 入口承接稳定核心能力；
6. V2 的完成以“正式闭环成立 + P0 通过”为准，而不是以愿景描述为准。
