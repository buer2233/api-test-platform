## 2. V3 阶段定位
V3 不再是对 V2 的零散增强，而是：
> **从“能力成型”转向“平台治理深化”的正式阶段**

V3 的核心主轴已经确认：
1. **平台治理优先**，先建立项目、环境、场景集、版本、执行记录和入口承接的统一治理边界；
2. 采用 **P0 / P1 / P2 分层推进**，避免把所有能力同时拉入一个失控版本；
3. **P0 必须先稳住多项目与环境治理**，并做到执行隔离级；
4. **P1 再承接权限体系、抓包正式执行、产品入口深化和调度中心**；
5. **P2 只保留高级 AI 治理边界与验收要求**，不把自动自愈和自动修复作为当前轮承诺交付；
6. Windows 端在 V3 中需要产出可实际测试的 Demo，但采用“**先 Web 稳定验证，再轻量套壳到 Windows**”的路线；
7. `macOS` 明确保留为后续要做的方向，但 **当前阶段暂不做正式实现**。

## 3. 进入 V3 的前提与承接关系
### 3.1 进入 V3 的前提
截至 2026-04-10，V2 已完成正式验收，已满足进入 V3 的前提条件：
1. 功能测试用例驱动正式闭环已成立；
2. 抓包驱动草稿化接入已成立；
3. Django + DRF + MySQL 服务化承载层已正式落地；
4. 可用型工作台入口、来源追溯、执行历史、差异摘要和建议治理已完成当前轮交付；
5. V2 未做项与转入 V3 的边界已经在 V2 计划文档与正式验收报告中明确归档。

### 3.2 V3 直接承接的 V2 成果
V3 直接复用以下 V2 成果，不重新推翻：
1. `platform_core` 中已有的场景模型、变量绑定、依赖关系和场景执行闭环；
2. `scenario_service` 中已建立的服务化事实源、修订留痕、建议治理和执行结果回写能力；
3. 基于 Django + DRF + MySQL 的服务边界和本地真实 MySQL 开发基线；
4. 工作台入口和统一服务摘要契约；
5. V2 已形成的文档、阶段验收口径和治理边界。

### 3.3 V3 与 V2 / 后续阶段的边界
1. V2 已结束，后续新增能力不再继续记在 V2 名下；
2. V3 重点是平台治理深化、执行隔离、项目化管理、正式产品入口深化和阶段化验收；
3. 企业治理级配额、复杂共享市场、重型协作版本治理、macOS 正式落地和高度自治的 AI 自愈能力，不纳入当前轮必达范围；
4. 任何 V3 能力都必须继续遵守“**数据库事实源优先、AI 只做建议不做事实**”的底线。

## 4. V3 总目标
V3 需要完成的核心目标如下：
1. 建立多项目、多环境、多场景集的正式治理模型；
2. 建立执行隔离级的平台边界，避免工作区、执行记录、导入导出和环境绑定出现跨项目串扰；
3. 为场景集建立标签、归档和基线版本治理能力；
4. 完成 V2 历史资产到默认项目 / 默认环境 / 默认场景集的平滑迁移；
5. 在浏览器端提供最小治理入口，并以此完成 P0 治理闭环验证；
6. 在 P1 建立权限体系、审核留痕和审计日志能力；
7. 在 P1 把抓包从“草稿化接入”升级为“正式执行闭环”；
8. 在 P1 深化 Web 正式入口，并交付可实际测试的 Windows Demo；
9. 在 P1 建立调度与执行中心；
10. 在 P2 明确高级 AI 治理边界、回退机制和独立验收要求；
11. 让 P0 / P1 / P2 都具备独立阶段验收，不再只依赖最终总验收。
---
## 5. V3 不做什么
为了控制范围，V3 当前明确不要求完成以下内容：
1. 不做 `macOS` 正式客户端实现，仅保留规划与承接说明；
2. 不做更重的分支/合并/版本回滚协作治理；
3. 不做企业治理级的配额、复杂共享市场与跨项目复杂引用策略；
4. 不做完全自治的 AI 自动修复和无人审批执行；
5. 不把 Windows Demo 做成完整桌面产品分发、升级与运维体系；
6. 不把 V3 扩张成“一次性做成完整企业级测试平台”的失控版本；
7. 不因入口体验诉求反向牵引核心治理设计；
8. 不允许抓包结果、AI 建议或前端壳逻辑绕过服务层和治理门禁直接成为正式事实。

## 6. V3 分层目标定义
### 6.1 P0：多项目与环境治理
P0 是 V3 的首轮必达项，目标是：
1. 建立项目、测试环境、场景集、标签、归档与基线版本治理对象；
2. 建立执行隔离级的平台边界；
3. 完成 V2 历史资产默认项目迁移；
4. 通过浏览器端最小治理入口完成治理能力的真实验证；
5. 形成 P0 独立验收结论。

### 6.2 P1：平台能力深化
P1 建立在 P0 稳定验收之后，目标是：
1. 建立权限体系、项目级访问边界、审核流和审计日志；
2. 打通抓包正式执行闭环；
3. 深化 Web 正式入口，并交付可实际测试的 Windows Demo；
4. 建立批量执行、队列、重试、调度策略和结果聚合能力；
5. 形成 P1 独立验收结论。

### 6.3 P2：高级 AI 治理边界
P2 当前只负责：
1. 定义多轮建议边界；
2. 定义审批门禁和回退要求；
3. 定义失败回滚原则和测试设计要求；
4. 形成 P2 独立验收结论；
5. 不把高级 AI 自动自愈能力伪装成本轮已交付实现。

### 6.4 分层验收原则
V3 从现在开始明确采用：
1. **P0 独立验收**
2. **P1 独立验收**
3. **P2 独立验收**
4. **V3 总体验收**

任何一层未形成明确验收结论，都不能直接被“后续能力已开发”掩盖。

### 7.2 P0-G2：场景集、标签、归档与基线版本治理
P0 至少要支持：
1. 项目下的场景集分组；
2. 标签与筛选；
3. 归档状态；
4. 基线版本号；
5. 冻结基线；
6. 当前生效版本；
7. 执行记录关联版本。

### 7.3 P0-G3：执行隔离与事实边界
P0 的“执行隔离级”必须落到以下实际边界：
1. 项目级工作区隔离；
2. 项目级执行记录隔离；
3. 导入导出边界隔离；
4. 环境绑定执行；
5. 数据库事实源与本地工作区执行载体的职责边界清晰；
6. 不允许不同项目之间互相污染执行结果、报告路径和导入产物。

### 7.4 P0-G4：V2 历史资产默认项目迁移
P0 必须定义并落地：
1. 现有 V2 场景、修订、建议、执行记录的默认项目迁移策略；
2. 默认环境与默认场景集的承接方式；
3. 迁移后来源追溯和执行历史不丢失；
4. 迁移前后查询口径和执行口径一致；
5. 迁移失败的回退与校验方案。

### 7.5 P0-G5：最小治理入口
P0 不只做后端治理，还必须提供一个浏览器端最小治理入口，用于验证：
1. 项目切换；
2. 环境切换；
3. 场景集筛选；
4. 版本查看；
5. 导入归属确认；
6. 执行时的项目 / 环境归属；
7. 隔离效果可视验证。

## 10. V3 详细设计方案
### 10.1 治理对象与统一事实源设计
1. 项目、环境、场景集、版本、执行记录必须围绕统一事实源建模；
2. 数据库继续作为核心事实源；
3. 本地工作区继续保留，但只能作为项目 / 环境下的执行载体；
4. 所有治理对象都要能被服务层、入口层和后续审计层共同消费。

### 10.2 执行隔离级设计方案
1. 隔离必须覆盖工作区、报告路径、执行记录、环境变量和导入导出路径；
2. 执行结果必须可追溯到项目、环境、场景集和版本；
3. 同名场景在不同项目中允许独立存在，但结果、历史和状态不能串扰；
4. 调度和入口能力都必须建立在这个隔离边界上。

### 10.3 V2 历史资产默认项目迁移设计方案
1. 现有 V2 数据统一迁入默认项目 / 默认环境 / 默认场景集；
2. 迁移必须保留执行历史、修订记录、建议记录和来源追溯；
3. 迁移后需可继续查询、执行与回看结果；
4. 迁移策略需支持校验、重跑和回退说明。

### 10.4 浏览器优先与 Windows 套壳策略
1. P0 入口验证优先走浏览器端；
2. P1 在浏览器端稳定后再做 Windows Demo；
3. Windows Demo 优先选择轻量套壳方案，默认 Tauri；
4. 打包节奏采用“平时浏览器验证，阶段性打包复验”的方式；
5. 入口层始终只消费统一服务层和治理对象，不直接反向修改核心状态。

### 10.5 权限与审计设计原则
1. 权限先围绕项目边界，不一开始就做复杂全局 IAM；
2. 审核流与审计日志必须可追溯；
3. 执行、迁移、版本切换和建议采纳等关键操作都必须有留痕。

### 10.6 抓包正式执行设计原则
1. 抓包在 V3 才升级为正式执行闭环；
2. 即使进入正式执行，也必须经过审核、绑定确认和规则门禁；
3. 不能因追求效率而跳过治理流程。

### 10.7 调度与执行中心设计原则
1. 调度中心必须建立在多项目与环境隔离边界稳定之后；
2. 批量执行与重试策略不能破坏项目边界；
3. 结果聚合需要保留版本、环境和触发来源上下文。

### 10.8 高级 AI 治理设计原则
1. AI 输出只能作为建议，不是事实源；
2. 所有自动化建议都必须经过审批、回退和审计设计；
3. 即使 P2 后续进入实现，也不能突破“建议而非事实”的底线。

## 12. V3 测试方案

V3 测试目标是：
1. 验证治理对象与事实边界成立；
2. 验证迁移策略与执行隔离成立；

4. 验证每个分层阶段都具备独立验收能力；
5. 验证 Windows Demo 的真实可测性与重复验证便捷性。

### 12.2 测试分层
V3 建议采用以下分层：
1. `V3-DOC`：文档一致性与阶段边界测试；
2. `V3-MODEL`：项目、环境、场景集、版本等核心治理对象测试；
3. `V3-MIG`：V2 历史数据迁移与兼容测试；
4. `V3-ISO`：项目级工作区、执行记录、导入导出和环境绑定隔离测试；
5. `V3-API`：治理服务与查询接口契约测试；
6. `V3-EXEC`：多项目 / 多环境下的执行路由与结果回写测试；
7. `V3-UI`：最小治理入口、正式 Web 入口和 Windows Demo 测试；
8. `V3-INT`：阶段验收级集成回归与浏览器 / Windows 冒烟测试。

### 12.3 分层验收标准
#### P0 验收标准
1. V2 历史资产完成默认项目迁移并可追溯；
2. 项目、环境、场景集、版本、执行记录形成统一归属关系；
3. 项目级执行隔离真实成立，不出现跨项目串扰；
4. 浏览器端最小治理入口能完成“切项目 -> 选环境 -> 看场景集 -> 执行 -> 回看结果”的真实闭环；
5. P0 自动化测试、迁移检查和入口冒烟全部通过。

#### P1 验收标准
1. 权限体系和审计日志已形成项目级边界；
2. 抓包正式执行闭环已形成正式验收证据；
3. Web 正式入口深化通过浏览器验收；
4. Windows Demo 可完成真实测试，并支持阶段性打包复验；
5. 调度与执行中心在不破坏项目隔离的前提下通过验收。

#### P2 验收标准
1. AI 治理边界、审批门禁、回退机制和测试设计已文档化；
2. P2 的独立验收结论明确区分“已实现能力”和“仅保留边界”；
3. 不允许把未实现的 AI 能力包装成已交付能力。

#### V3 总体验收标准
1. P0 / P1 / P2 全部形成独立验收结论；
2. README、阶段文档、测试文档和正式验收报告同步完成；
3. V3 / 后续阶段边界清晰；
4. 当前轮未做项已明确转入后续阶段承接。
