```bash
$env:DJANGO_SETTINGS_MODULE='platform_service.settings'; .\.venv_service\Scripts\python.exe manage.py makemigrations scenario_service --check --dry-run --settings=platform_service.migration_settings
```

结果：

- `tests/platform_core`：`71 passed`
- 根测试 `tests`：`79 passed`
- `api_test/tests`：`39 passed`
- `service_tests`：`19 passed`
- 迁移一致性检查：`No changes detected in app 'scenario_service'`

### 6.2 本次验收中的执行说明

1. `service_tests` 首次使用 `--basetemp=.pytest_tmp_service/v2_acceptance_service` 执行时出现 `FileNotFoundError`。
2. 根因是 pytest 不会自动创建缺失的父目录，而不是业务逻辑失败。
3. 调整为 `.pytest_tmp/v2_acceptance_service_fixed` 后重新执行，`service_tests` 全量通过。

结论：

- 本次异常属于验收命令参数问题，不构成 V2 功能缺陷，不影响正式验收结论。

## 8. 风险、遗留项与边界判断

### 8.1 当前已缓解的关键风险

1. MySQL 正式环境验收缺口已缓解，当前已有真实库迁移、状态检查和执行冒烟证据。
2. 服务层与入口耦合风险已缓解，当前工作台只消费统一服务摘要与 API。
3. AI 修订越权风险已缓解，当前建议必须先落为建议记录，再采纳为标准修订并重新进入审核链路。
4. 服务测试库权限不足风险已缓解，当前已用本地 MySQL 基线复用和清表夹具绕开建库权限限制。

### 8.2 当前仍保留但不阻断验收的事项

1. 双事实源冲突风险仍需在 V3 继续通过更完整的事实治理和版本治理增强。
2. 场景编排复杂度仍会随着更多变量注入、更多公开基线操作目录而继续上升。
3. 抓包噪声治理已形成基础能力，但更复杂的低置信清洗仍建议进入 V3。

### 8.3 未完成项是否影响验收

结论是不影响。

原因如下：

1. 这些事项在 V2 计划文档中本就被定义为后续增强项、P2 项或明确转入 V3 的内容。
2. 本次正式验收只对 V2 已承诺范围负责，不以 V3 范围反向否定 V2 验收。

## 10. V3 工作计划建议

### 10.1 V3 阶段目标建议

V3 建议从“能力成型”转向“平台化深化”，重点目标如下：

1. 建立多项目、多资产、多环境的统一治理模型。
2. 建立协作、权限、审核分工和操作留痕体系。
3. 将抓包驱动从“草稿导入”升级为“正式执行闭环”。
4. 建立更完整的 Web 产品体验和独立 Windows 客户端承接方案。
5. 建立更成熟的执行调度、批量运行、历史比对和差异治理能力。
6. 建立更高级的 AI 建议、自愈和自动修复治理边界。

### 10.2 V3 建议的任务分组

| 建议分组 | 目标 | 优先级 |
| --- | --- | --- |
| V3-G1 多项目与资产治理 | 项目、环境、场景集、版本、标签和归档治理 | P0 |
| V3-G2 协作与权限体系 | 用户、角色、审核流、权限边界和审计日志 | P0 |
| V3-G3 抓包正式执行闭环 | 抓包清洗确认、变量绑定确认、执行生成与结果回写 | P0 |
| V3-G4 调度与执行中心 | 批量执行、队列、重试、调度策略和结果聚合 | P1 |
| V3-G5 产品化入口深化 | React 正式前端、Windows 壳、视觉回归和交互治理 | P1 |
| V3-G6 高级 AI 治理 | 多轮建议、自动修订、自愈策略、回退与审批机制 | P2 |

### 10.3 V3 建议实施顺序

1. 先做多项目与协作治理，避免继续在单项目假设上累积技术债。
2. 再做抓包正式执行闭环和调度中心，补齐第二条输入路线的真正生产能力。
3. 然后推进产品化入口深化，把当前工作台升级为正式产品入口。
4. 最后再扩展高级 AI 治理和自动修复，避免 AI 能力先于治理边界落地。

### 10.4 V3 建议验收重点

1. 多项目隔离和权限边界必须先于大规模入口扩张通过验收。
2. 抓包正式执行闭环必须有独立的确认层和回放治理，不可直接越权执行。
3. 调度中心必须能稳定回看历史、差异和追溯来源。
4. AI 能力必须继续遵守“建议而非事实”的治理原则。
