# V3 P2 AI 治理边界设计说明

## 1. 设计目标
- 在现有 `scenario_service` 建议治理、修订留痕、权限门禁和审计日志基础上，补齐 `P2` 所要求的 AI 治理边界。
- 把当前系统从“建议可生成、可直接采纳转修订”升级为“建议受策略约束、必须人工决策、可回退、可审计、可在入口中被正确识别状态”。
- 明确 `P2` 当前交付的是治理能力，而不是自动自愈、无人审批执行或高级自治 AI。

## 2. 当前缺口
### 2.1 已有能力
- 已存在 `ScenarioSuggestionRecord`，可生成建议并查询建议列表。
- 已存在 `apply_suggestion()`，可把建议转成标准修订记录。
- 已存在 `ProjectRoleAssignmentRecord` 与 `ScenarioAuditLogRecord`，可表达项目级权限和关键动作审计。
- 已存在 `/ui/v3/workbench/`，可承接治理入口、审计、抓包正式执行和调度中心。

### 2.2 缺失能力
- 缺少正式的 AI 治理策略对象，无法结构化表达建议类型、审批门禁和回退要求。
- 缺少 AI 建议从“待审批 -> 已拒绝 / 已批准待采纳 / 已采纳 / 已回退”的正式状态机。
- 缺少建议审批、拒绝、回退及责任链记录对象，无法稳定支撑 `P2` 审计和验收。
- 缺少“未经审批不得执行”阻断逻辑，当前仍可能把建议视为普通修订入口处理。
- 缺少入口层 AI 状态分层展示，工作台当前只能展示原始建议列表。

## 3. 设计方案
### 3.1 统一对象模型
- 新增 `AiGovernancePolicyRecord` 作为项目级 AI 治理策略对象，挂接 `project`，记录：
  - `policy_id`
  - `scope_type`
  - `scope_ref`
  - `suggestion_types`
  - `approval_mode`
  - `rollback_mode`
  - `auto_execution_enabled`
  - `metadata`
- 新增 `AiSuggestionDecisionRecord` 作为建议流转责任链对象，挂接 `scenario`、`suggestion`、`project`，记录：
  - `decision_id`
  - `decision_type`，包括 `approve / reject / adopt / rollback`
  - `decision_status`
  - `actor_name`
  - `decision_comment`
  - `snapshot_before`
  - `snapshot_after`
  - `related_revision_id`
  - `related_execution_id`
  - `metadata`

### 3.2 建议状态机
- `pending_approval`：建议已创建，但尚未审批。
- `rejected`：建议被拒绝，正式事实保持不变。
- `approved`：建议已审批通过，但尚未采纳落地。
- `adopted`：建议已采纳，已转成标准修订记录。
- `rolled_back`：建议已执行回退，正式事实已恢复到采纳前状态。

### 3.3 服务层原则
- 建议创建时必须挂接项目级治理策略，并把策略摘要返回给入口层。
- 审批和拒绝必须由具备 `can_review` 权限的成员执行，并写入责任链与审计日志。
- 采纳必须以“已审批”为前置条件，且继续走现有标准修订留痕，不允许绕过修订事实层。
- 回退必须基于采纳时保存的快照恢复正式状态，并写入回退责任链与审计日志。
- 未审批建议不得进入正式执行；任何越权尝试都必须被阻断并留痕。

### 3.4 入口承接原则
- `/ui/v3/workbench/` 新增 AI 治理区域，明确显示：
  - 策略摘要
  - 待审批数量
  - 已批准待采纳数量
  - 已采纳数量
  - 已拒绝数量
  - 已回退数量
- 建议列表必须区分 `pending_approval / approved / rejected / adopted / rolled_back`，并显示审批要求与最近责任链摘要。
- UI 只消费服务层状态，不在前端拼装治理事实。

## 4. 测试与验收映射
- `DOC-001/002/003`：通过文档与实现口径检查，锁定“AI 建议不是事实源”“审批/回退不可绕过”。
- `MODEL-001/002`：通过新增治理策略对象与责任链对象的模型测试覆盖。
- `MIG-001`：通过迁移兼容测试确认 `P1` 历史事实链不被覆盖。
- `ISO-001/002`：通过项目边界、角色边界和上下文脱敏校验覆盖。
- `API-001/002/003`：通过建议查询、审批/拒绝/采纳、回退/审计接口覆盖。
- `EXEC-001`：通过未审批执行阻断测试覆盖。
- `UI-001`：通过工作台 AI 治理状态展示测试覆盖。
- `INT-001/002/003/004`：通过采纳闭环、拒绝不污染事实、回退恢复状态和独立验收口径覆盖。

## 5. 当前轮不做
- 不做自动自愈正式落地。
- 不做无人审批执行。
- 不做自动修复批量编排。
- 不做独立 AI 服务进程或新的模型推理子系统。

## 6. 结论
- `P2` 的正确实现方式不是额外造一个“AI 系统”，而是在现有建议治理、修订留痕、权限和审计底座上补齐正式治理边界。
- 当前设计已经与 `V3阶段工作计划文档.md` 和 `详细测试用例说明书(V3-P2).md` 对齐，可直接进入实施计划和 TDD 开发阶段。
