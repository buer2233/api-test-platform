# V1 最终收口设计说明

## 1. 文档信息
- 日期：2026-04-02
- 主题：完成 V1 阶段最后一轮平台化收口并形成可验收状态
- 适用范围：`platform_core/`、`api_test/`、`tests/`、README、阶段文档与测试文档

## 2. 背景与问题
截至当前分支，V1 的 P0 闭环实际上已经成立，但阶段文档仍把 V1 标记为“进行中”。当前阻塞点主要集中在三类残留：
1. `business_rule` 只支持 `non_empty_string`，规则覆盖仍偏窄；
2. `run` / `inspect` 的稳定摘要虽然已存在，但对后续 Django + DRF / 客户端直接消费来说，资产聚合字段仍偏少；
3. `api_test/requirements.txt` 仍保留宽松版本约束和未使用的 `rsa` 依赖，不符合仓库的最高优先级依赖安全规则。

这些问题不再属于“大功能缺失”，而是 V1 最后一轮边界治理和工程收口问题。

## 3. 方案比较

### 方案一：只改文档状态，不改代码
优点：
1. 成本最低。

缺点：
1. 无法解释文档中此前列出的剩余风险为何被关闭；
2. 依赖治理问题仍然存在；
3. 阶段完成判断缺少新的实质交付支撑。

### 方案二：继续做更大范围能力扩展
做法：
1. 继续实现更复杂 `business_rule` DSL；
2. 引入更多 CLI 命令或 V2 级别能力。

优点：
1. 产出更多功能。

缺点：
1. 容易越过 V1 边界；
2. 会把当前“阶段收口”重新拖回“继续扩张”。

### 方案三：做最小但足够的最终收口，推荐
做法：
1. 为 `business_rule` 增加第二个简单规则代码 `positive_integer`；
2. 为服务摘要增加稳定的资产聚合字段；
3. 把依赖安全规则落成可执行治理测试，并修正 `requirements.txt`；
4. 在完整回归通过后，把 V1 文档状态正式切换为“已完成”。

优点：
1. 直接命中当前剩余问题；
2. 改动范围可控，适合一轮 TDD 闭环；
3. 能把“V1 进行中”合理收口为“V1 已完成”。

缺点：
1. 不会实现更复杂规则 DSL；
2. 更深层结构断言与变量依赖仍需放入 V2。

## 4. 选型结论
本轮采用方案三。

原因：
1. 当前剩余问题已经是收口性质，不需要再引入更大实现面；
2. V1 的完成标准是“最小闭环成立并可验证”，不是覆盖所有未来增强项；
3. 本方案既有代码交付，又能给文档状态切换提供充分依据。

## 5. 设计范围

### 5.1 `business_rule` 扩展
本轮新增第二个通用规则代码：
- `positive_integer`

语义：
1. 读取目标路径对应值；
2. 校验值类型为整数；
3. 校验值大于 0。

约束：
1. 不扩展为 DSL；
2. 不支持跨字段关系；
3. 不让解析器自动推断该规则。

### 5.2 服务摘要收口
在现有 `run` / `inspect` 摘要基础上增加更稳定、可直接用于服务接口的聚合字段。

具体方向：
1. `run` 摘要补齐 `source_type`、`execution_id` 和资产类型分布；
2. `inspect` 摘要补齐 `source_type`、`execution_id` 和工作区资产聚合摘要；
3. 工作区资产聚合摘要至少包含：
   - 资产类型分布；
   - 生成记录审核状态分布；
   - 生成记录执行状态分布。

### 5.3 依赖治理
1. 为 `api_test/requirements.txt` 增加治理测试；
2. 所有依赖改为固定版本；
3. 删除未使用的 `rsa`；
4. 优先复用仓库中原有的版本下限，不引入新的高版本依赖。

## 6. 测试设计

### 6.1 失败测试
先补失败测试，至少覆盖：
1. `positive_integer` 的模板渲染；
2. `positive_integer` 的假响应生成；
3. `positive_integer` 的规则校验通过；
4. `run` / `inspect` 摘要的新聚合字段；
5. `api_test/requirements.txt` 不允许宽松版本和 `rsa`。

### 6.2 绿灯验证
完成实现后执行：
1. 定向红绿测试；
2. `tests/platform_core` 全量回归；
3. 根测试全量回归；
4. `api_test/tests` 全量回归；
5. 公开基线双入口回归。

## 7. 风险与控制

### 风险 1：把 V1 又拖回持续扩张
控制：
1. 只做一个新增规则代码；
2. 只做聚合摘要，不新增大命令集；
3. 把更复杂规则和变量依赖明确转入 V2。

### 风险 2：摘要字段与底层资产不一致
控制：
1. 使用服务层聚合，而不是在 CLI 中散落拼装；
2. 先写模型和服务失败测试，再实现。

### 风险 3：依赖治理只写文档不落地
控制：
1. 新增自动化治理测试；
2. 同步修正 `requirements.txt`。

## 8. 验收标准
本轮只有同时满足以下条件才算完成：
1. 新增失败测试已真实失败；
2. 最小实现后所有新增测试转绿；
3. `tests/platform_core`、根测试、`api_test/tests` 和公开基线均通过；
4. README、V1 文档、测试文档和 V2 承接说明已同步；
5. 阶段文档可据实切换到“V1 已完成”。

## 9. 结论
本轮不是再做一个“功能扩展轮”，而是把已经具备闭环能力的 V1 正式收口。

通过补齐第二档 `business_rule`、稳定服务摘要聚合字段和依赖治理约束，当前仓库将具备把 V1 从“持续重构中”切换为“阶段完成”的充分依据。
