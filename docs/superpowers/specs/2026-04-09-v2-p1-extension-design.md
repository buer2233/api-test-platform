# V2 P1 扩展设计说明书

## 1. 文档目的
本文档用于定义 V2 阶段在 `P0 主线已完成` 前提下，继续补齐 `P1` 与少量 `P2 前置` 的设计范围。

本文档只覆盖以下三段已确认方向：
1. 核心能力增强
2. AI 与修订增强
3. 入口一致性前置

本文档不进入实现细节任务拆分，不替代 `product_document/阶段文档/V2阶段工作计划文档.md`，而是作为本轮增量扩展的正式设计规格，为下一步实施计划编写提供依据。

## 2. 当前背景
截至 2026-04-09，V2 的 `P0` 主线已经完成并通过自动化验证，当前仓库已经具备以下能力：
1. 功能测试用例驱动的正式闭环已经成立；
2. 抓包驱动的草稿化接入闭环已经成立；
3. Django + DRF + MySQL 已成为场景资产、审核记录、执行结果的正式承载层；
4. Web 工作台入口已经具备导入、预览、审核、执行和结果查看的最小可用流程。

但当前仍存在以下缺口：
1. 场景的多来源追溯信息保留仍然偏弱；
2. 抓包低置信、复杂噪声和问题分类仍然较粗；
3. 执行历史沉淀仍偏向“最新一次结果”，缺少更清晰的历史视图和差异摘要；
4. AI 建议尚未进入结构化治理链路；
5. 入口虽然可用，但列表筛选、差异视图、历史摘要和双入口一致流程语言仍未定型。

因此，本轮扩展的目标不是把 V2 改写成 V3，而是在不破坏既有分层和闭环的前提下，把 V2 的 `P1` 做实，并为少量 `P2` 能力补足前置边界。

## 3. 范围结论

### 3.1 本轮要做的内容
1. 补齐多来源追溯事实、场景问题事实和执行历史事实；
2. 增强抓包低置信问题标记、复杂噪声分类和一致性检查能力；
3. 增强场景列表筛选、结果摘要、历史查询和轻量差异检查；
4. 引入 AI 建议对象和建议采纳治理链路；
5. 强化修订记录、执行记录与建议记录之间的关联追溯；
6. 固化 Web / Windows 共用的流程语言、服务契约和入口职责边界。

### 3.2 本轮明确不做的内容
1. 不把抓包驱动升级为正式执行闭环；
2. 不做完整成熟的 Web 产品体验；
3. 不做独立 Windows 壳的分发、升级和运维体系；
4. 不做多项目、多角色、权限体系和组织协作；
5. 不做高级 AI 自愈、失败归因、自动修复和大模型质量评估；
6. 不把当前轮重新定义为 V3。

### 3.3 本轮允许纳入的少量 P2 前置
1. 为后续抓包正式闭环补足稳定的事实字段和问题分类；
2. 为后续更复杂执行治理补足历史记录和差异摘要边界；
3. 为后续 Windows 独立入口补足统一服务契约和流程语言。

## 4. 路线比较与结论
本轮扩展在设计阶段评估了三条路线：

### 4.1 方案 A：核心事实层增强优先
先增强持久化事实层、服务层查询与结果治理，再让入口消费这些稳定事实。

优点：
1. 最符合当前服务分层；
2. 风险最低；
3. 最容易复用到后续 Windows 入口和 V3 平台化能力。

缺点：
1. 入口层的表面提升速度会慢于“先做页面”路线；
2. 需要先定义更稳定的数据结构。

### 4.2 方案 B：入口体验优先
先补工作台页面的筛选、对比、建议面板和历史面板，再倒推服务与持久化结构。

优点：
1. 视觉和演示效果提升更快；
2. 短期更容易展示“新能力”。

缺点：
1. 容易出现 UI 倒逼服务契约的临时设计；
2. 容易把事实层做成被页面结构绑死的脆弱实现。

### 4.3 方案 C：AI 与修订优先
先做 AI 建议、多轮确认和修订对比，再补执行历史和筛选能力。

优点：
1. “智能化”感知更强；
2. 更容易快速展示 AI 入口能力。

缺点：
1. 会推迟当前最缺的事实层治理、执行历史和结果摘要能力；
2. 容易把 AI 建议做成越权通道。

### 4.4 最终选择
本轮采用 `方案 A：核心事实层增强优先`。

原因如下：
1. 当前 V2 已经形成服务化和持久化骨架，继续增强事实层最稳；
2. 多来源追溯、执行历史和问题分类是后续筛选、差异和 AI 建议治理的共同前提；
3. 入口层应继续只承接稳定事实，不应倒逼核心层产生临时结构。

## 5. 核心设计

### 5.1 设计原则
本轮扩展继续遵循以下原则：
1. 数据库是事实源，本地工作区只是执行与产物载体；
2. AI 只能生成候选建议，不能直接成为正式事实；
3. 入口层只消费服务层输出，不直接拼接核心业务；
4. 低置信和不确定项一律保守下沉为问题清单，不自动升级为正式资产；
5. 所有新增能力都必须能被测试、留痕、追溯和后续扩展。

### 5.2 核心事实层增强
本轮在现有 `ScenarioRecord`、`ScenarioReviewRecord`、`ScenarioRevisionRecord`、`ScenarioExecutionRecord` 的基础上，补齐三类事实能力。

#### 5.2.1 多来源追溯事实
场景、步骤、变量、依赖需要补充以下信息：
1. 来源类型：功能用例、抓包、既有接口资产、AI 建议；
2. 来源引用：原始来源标识或关联记录标识；
3. 置信度：高、中、低；
4. 问题标签：用于表示低置信、待确认、来源冲突或绑定不稳。

设计要求：
1. 来源事实必须支持一个场景同时保留多种来源；
2. 场景级和步骤级都要支持追溯；
3. 修订记录和建议记录也要能关联来源事实。

#### 5.2.2 场景问题事实
当前仅有通用问题清单，本轮需要把以下问题类型结构化：
1. `missing_operation_binding`
2. `dynamic_value_uncertain`
3. `duplicate_request_group`
4. `static_noise_filtered`
5. `assertion_incomplete`
6. `source_conflict`
7. `capture_operation_needs_review`

设计要求：
1. 问题应可被服务层筛选、聚合和返回；
2. 问题应可在详情页和列表页以统一口径展示；
3. 问题标签应支持扩展，不与页面写死耦合。

#### 5.2.3 执行历史事实
当前场景记录主要保留最新一次执行结果，本轮需补齐历史沉淀能力。

设计要求：
1. 每次执行都必须保留独立记录；
2. 执行记录要能表达结果统计、失败摘要、报告路径、触发来源和时间；
3. 场景结果查询要能返回最近一次执行摘要和历史执行列表；
4. 历史记录不能被后续执行覆盖。

### 5.3 服务层增强
`scenario_service/services.py` 继续作为唯一编排入口，但从“操作服务”扩展为“治理型查询 + 生命周期服务”。

#### 5.3.1 场景列表服务增强
列表服务需要支持：
1. 按来源类型筛选；
2. 按审核状态筛选；
3. 按执行状态筛选；
4. 按低置信问题标签筛选；
5. 按更新时间排序；
6. 返回来源聚合、问题聚合和最近执行摘要。

列表服务不负责页面渲染，只返回稳定摘要结构。

#### 5.3.2 场景详情服务增强
详情服务需要支持：
1. 返回步骤、问题、审核记录、修订记录；
2. 返回来源追溯信息；
3. 返回最近执行历史；
4. 返回最近建议摘要；
5. 返回适合页面消费的轻量差异摘要。

#### 5.3.3 结果查询服务增强
结果服务需要支持：
1. 返回当前审核状态和执行状态；
2. 返回最近执行统计；
3. 返回历史执行列表；
4. 返回最近两次执行的轻量差异摘要。

差异摘要本轮只覆盖：
1. 执行状态变化；
2. 通过数、失败数、跳过数变化；
3. 失败摘要变化；
4. 关联修订版本变化。

本轮不做完整报告对比引擎。

### 5.4 抓包治理增强
`platform_core/traffic_capture.py` 继续保持“草稿化接入”的阶段定位，不升级为正式执行入口。

#### 5.4.1 低置信问题分层
当前 `capture_confidence: low` 过于粗糙，本轮需增强为：
1. 低置信标记；
2. 复杂噪声标签；
3. 请求去重来源；
4. 动态值识别不确定原因；
5. 接口绑定待确认原因。

#### 5.4.2 保守治理原则
抓包解析层继续执行以下保守策略：
1. 无法稳定识别的请求不直接产出正式事实；
2. 无法稳定绑定的操作只保留候选和问题说明；
3. 无法稳定提取的动态值只进入问题清单，不自动生成高置信绑定。

### 5.5 AI 建议与修订增强
本轮不把 AI 定义为“自动修订者”，而是把它纳入候选建议链路。

#### 5.5.1 建议对象设计
新增建议记录对象，承载以下信息：
1. 目标范围：场景、步骤、变量、依赖、断言；
2. 建议类型：改写、补充、删除、断言补全、低置信修复；
3. 基线快照：生成建议时所对应的场景版本或修订版本；
4. 建议补丁：结构化 patch；
5. 差异摘要；
6. 建议来源、置信度、生成时间、采纳状态。

设计要求：
1. 建议对象与正式场景对象分离；
2. 建议对象可以被查询、预览、比较；
3. 建议对象不能直接覆盖正式场景。

#### 5.5.2 建议生效路径
AI 建议如果要生效，必须走以下路径：
1. 生成建议；
2. 查看当前版本与建议版本差异；
3. 人工选择全部采纳或按字段采纳；
4. 生成一次标准修订记录；
5. 再进入审核门禁。

因此，AI 建议只是一种修订候选来源，而不是正式内容本身。

#### 5.5.3 建议提供者边界
本轮不把真实大模型接入质量设为阻断项，建议采用两层结构：
1. 建议提供者抽象层；
2. 规则驱动或模拟建议提供者的默认实现。

这样可以先走通：
1. 低置信问题修复建议；
2. 断言补全建议；
3. 结构化字段补全建议；
4. 差异展示和采纳留痕。

后续如接入真实模型，只替换提供者实现，不改治理链路。

### 5.6 修订历史与执行历史关联
为了让“修订 -> 执行 -> 结果变化”形成最小闭环，本轮需要补足以下关联：
1. 执行记录可标记基于哪个修订版本触发；
2. 执行记录可标记是否来自 AI 建议采纳；
3. 结果差异可表达与上一轮相比的变化摘要；
4. 详情页和结果页可回看某次执行对应的修订上下文。

### 5.7 入口一致性前置
本轮不新建独立前端工程，也不做完整产品化视觉升级，而是固化入口职责边界和双入口流程语言。

#### 5.7.1 统一流程语言
Web 与未来 Windows 入口统一使用以下阶段语义：
1. `draft`
2. `reviewing`
3. `confirmed`
4. `executing`
5. `finished`

统一动作语义如下：
1. 导入场景
2. 查看问题
3. 提交审核
4. 结构化修订
5. 查看 AI 建议
6. 触发执行
7. 查看结果
8. 查看历史
9. 查看差异

#### 5.7.2 入口职责边界
入口层只负责：
1. 收集输入；
2. 调用服务；
3. 展示摘要、详情、问题、历史和差异；
4. 触发审核、修订和执行动作。

入口层不负责：
1. 本地拼装场景编排逻辑；
2. 本地计算执行差异；
3. 本地直接修改正式场景事实。

#### 5.7.3 工作台最小增强内容
现有工作台本轮建议补充：
1. 列表筛选区；
2. 来源和问题摘要区；
3. 最近执行历史区；
4. 最近两次差异摘要区；
5. 结构化修订与 AI 建议并列入口；
6. 历史执行入口。

本轮不做：
1. 拖拽式流程编排器；
2. 富交互编辑器；
3. 独立设计系统；
4. Windows 壳自动化测试与分发能力。

### 5.8 服务契约前置
为了后续 Windows 共用，本轮需先固定服务契约结构。

#### 5.8.1 列表接口
列表接口返回：
1. 基础摘要；
2. 来源聚合；
3. 问题聚合；
4. 最近执行摘要；
5. 最近修订摘要。

#### 5.8.2 详情接口
详情接口返回：
1. 步骤列表；
2. 问题列表；
3. 来源追溯；
4. 审核记录；
5. 修订记录；
6. 建议记录；
7. 执行历史；
8. 轻量差异摘要。

#### 5.8.3 结果接口
结果接口返回：
1. 当前执行摘要；
2. 历史执行记录；
3. 最近差异摘要；
4. 关联修订或建议标识。

#### 5.8.4 错误响应
错误结构继续保持统一的 `code/message` 语义，避免为不同入口拆分不同错误格式。

## 6. 测试设计要求
本轮扩展仍然按 TDD 推进，必须先补测试再写实现。

### 6.1 核心测试范围
本轮至少补齐以下测试方向：
1. 多来源追溯持久化测试；
2. 执行历史独立沉淀测试；
3. 列表筛选与排序契约测试；
4. 结果差异摘要测试；
5. 抓包低置信问题分类测试；
6. AI 建议记录创建与查询测试；
7. AI 建议不能直接覆盖正式场景的阻断测试；
8. AI 建议采纳后生成标准修订记录的测试；
9. 多轮修订与多次执行的关联追溯测试；
10. 工作台展示来源、问题、历史和差异摘要的契约测试；
11. Web / Windows 共享流程语言的一致性契约测试。

### 6.2 阶段性通过标准
本轮扩展通过的判断标准如下：
1. 多来源追溯、问题分类和执行历史都能被服务层稳定返回；
2. 列表、详情和结果查询的契约能承接筛选、历史和差异需求；
3. 抓包低置信与复杂噪声标记更细化，但仍保持草稿化接入边界；
4. AI 建议已进入“建议 -> 采纳 -> 修订 -> 审核”的治理链路；
5. Web 工作台能够消费新的历史、差异和来源摘要；
6. Windows 后续可以复用同一套服务流程语言和契约，而不需要额外定义第二套核心逻辑。

## 7. 风险与控制

### 7.1 范围膨胀风险
风险：把本轮扩展从 `P1 + 少量 P2 前置` 拉成 V3。

控制方式：
1. 不做抓包正式执行闭环；
2. 不做完整产品体验；
3. 不做权限与协作；
4. 不做高级 AI 自愈和自治编排。

### 7.2 双事实源风险
风险：入口或本地工作区再次越权成为事实源。

控制方式：
1. 一切来源、修订、建议、执行历史都回写数据库；
2. 工作区只保留执行产物和报告引用；
3. 入口只消费服务层结构化结果。

### 7.3 AI 越权风险
风险：AI 建议绕过修订和审核门禁，直接修改正式场景。

控制方式：
1. AI 建议与正式场景分表或分对象；
2. AI 建议只能生成 patch；
3. 采纳 AI 建议必须生成标准修订记录；
4. 修订后仍需审核确认。

### 7.4 服务契约漂移风险
风险：工作台和后续 Windows 各自形成不同字段口径。

控制方式：
1. 先固定列表、详情、结果三类服务结构；
2. 入口只做消费，不反向定义核心字段；
3. 契约变更必须同步测试和文档。

## 8. 实施顺序建议
本轮实现建议顺序如下：
1. 先补核心事实层和服务契约；
2. 再补抓包问题分类与执行历史查询；
3. 再补 AI 建议对象和采纳链路；
4. 最后补工作台筛选、历史、差异和建议入口展示。

采用该顺序的原因如下：
1. 入口增强依赖事实层和服务契约稳定；
2. AI 建议治理依赖修订和历史追溯结构稳定；
3. 先做核心层可以减少前端返工和接口回改。

## 9. 结论
本轮 V2 扩展不是新起一个阶段，而是在 `V2 P0 已完成` 的前提下，对 `P1` 与少量 `P2 前置` 进行收口增强。

本轮最终设计结论如下：
1. 采用“核心事实层增强优先”的路线；
2. 先补多来源追溯、问题分类、执行历史和差异摘要；
3. 再把 AI 建议纳入结构化修订与审核门禁；
4. 最后以统一服务契约承接 Web 工作台和后续 Windows 入口；
5. 全程保持数据库事实源、AI 非最终事实、抓包仍为草稿化接入、入口不越权的边界不变。
