## 2. 设计背景
当前仓库已经具备：
1. `api_test` 作为既有接口自动化测试底座；
2. Django + DRF + MySQL 的服务化承载层；
3. 一个偏 V3 治理控制台风格的 Web 工作台。

但当前前端表达存在明显偏离：
1. 主界面过度强调治理、审计、调度和 AI 状态；
2. 没有把“方便做接口自动化”作为第一入口；
3. 没有把 `api_test` 作为唯一资产落点清晰表达出来；
4. 抓包到生成用例的核心链路被弱化。

本轮设计的目标，是把平台重新收敛为：

> **一个面向 `api_test` 框架的抓包治理、接口方法识别、接口自动化用例生成与执行平台**

---

## 3. 核心设计结论

### 3.1 平台最核心只做三件事
1. 抓包获取接口数据并进行数据治理；
2. 选取接口并判断 `api_test` 中是否已有对应接口方法；
3. 按抓包顺序拼接为完整接口自动化测试用例。

### 3.2 最终只沉淀两类核心资产
1. `api_test/core/<project>/<model>/` 下的接口测试方法；
2. `api_test/tests/<project>/<model>/` 下的接口自动化测试用例。

### 3.3 设计主线
平台的唯一主线应为：

1. 抓包；
2. 数据治理；
3. 识别已有接口方法；
4. 新增缺失接口方法；
5. 生成测试用例；
6. 执行 `pytest`；
7. 查看 `allure` 报告；
8. 失败重试。

### 4.1 当前轮设计范围
1. 主界面信息架构；
2. 模块级抓包入口与过滤机制；
3. 抓包治理后的接口候选列表；

2. 审计后台重新设计；
3. Windows Demo 扩张；
4. 独立于 `api_test` 的第二套测试执行体系；
5. 一次性做完“AI 从最终接口自动倒推所有前置依赖”的完整自治能力。

说明：
- AI 依赖反推是后续增强主线；
- 当前轮只先预留方向，不把它包装成交付完成能力。

## 5. 资产与目录规则

### 5.1 目录规范
后续接口方法与测试用例必须落到以下位置：

1. `api_test/core/<project>/<model>/`
2. `api_test/tests/<project>/<model>/`

其中：
1. `project` 表示项目目录；
2. `model` 表示模块目录；
3. 两者统一使用 **英文小写下划线** 命名。

### 5.2 资产约束
1. 平台不允许把核心资产继续散落在数据库或平行目录中长期停留；
2. 数据库中的记录只作为过程记录和治理记录；
3. 最终正式可复用资产必须落回 `api_test`。

## 6. 页面信息架构

### 6.1 总体布局
主工作台采用三段式布局：

1. 左侧树导航区；
2. 中间当前层级对象工作列；
3. 右侧详情与执行区。

### 6.2 左侧树结构
左侧树的正式层级为：

1. 项目 / URL
2. 模块
3. 子模块
4. 测试用例
5. 接口方法

交互规则：
1. 默认展开到子模块；
2. 点击子模块时，中间列默认显示该子模块下的测试用例列表；
3. 接口方法不作为默认主入口，而是同属子模块下的资产入口；
4. 点击测试用例时，右侧显示该用例详情；
5. 点击接口方法时，右侧显示该方法详情和被哪些用例引用。

### 6.3 中间工作列
中间列只负责：
1. 当前层级对象列表；
2. 当前层级最核心动作；
3. 当前选中对象的状态摘要。

交互规则：
1. 点击项目 / URL：显示模块列表和项目摘要；
2. 点击模块：显示子模块列表和模块摘要；
3. 点击子模块：默认显示测试用例列表；
4. 接口方法作为次级入口，不抢默认展示位。

当点击模块 / 子模块时，中间列显示的核心动作固定为：
1. 开始抓包；
2. 停止抓包；
3. 导入抓包；
4. 查看接口方法；
5. 新增测试用例。

### 6.4 右侧详情与执行区
右侧采用混合式结构：

上半区固定显示：
1. 用例基本信息；
2. 执行测试；
3. 失败重试；
4. 最近结果；
5. 查看最新报告。

下半区标签页切换：
1. 方法链；
2. 历史执行；
3. 测试报告。

## 8. 抓包设计

### 8.1 抓包入口位置
抓包入口只允许出现在：
1. 模块级；
2. 子模块级。

不允许单独设置项目级抓包中心。

原因：
1. 抓包的目标是新增接口方法和新增测试用例；
2. 抓包结果必须落到明确模块 / 子模块归属；
3. 避免先抓一堆无归属数据，再二次人工整理。

### 8.2 抓包方式
抓包采用本地代理端口方式，形态接近 Fiddler：
1. 平台启动本地代理监听端口；
2. 用户将浏览器代理指向该端口；
3. 用户在浏览器中真实操作业务；
4. 平台获取经过代理的请求流量。

### 8.3 抓包过滤机制
抓包必须支持**前置过滤**，并且该过滤不是“只显示过滤结果”，而是：

> **开启过滤后，平台只记录和获取匹配过滤条件的抓包数据。**

过滤能力要求：
1. 支持按 URL 域名过滤；
2. 支持按 IP 过滤；
3. 后续可扩展路径前缀过滤，但当前轮先以 URL / IP 为核心；
4. 支持在开始抓包前设置过滤条件；
5. 抓包会话中必须记录本次过滤条件。

示例：
- 过滤条件：`https://weapp.mulinquan.cn`

平台行为：
1. 只记录该域名相关的请求；
2. 其余无关请求不进入平台记录；
3. 从源头减少无用抓包；
4. 降低后续数据治理复杂度和时间成本。

### 8.4 抓包会话要求
每次抓包会话应记录：
1. 所属项目；
2. 所属模块；
3. 所属子模块；
4. 代理端口；
5. 过滤条件；
6. 开始时间；
7. 停止时间；
8. 抓到的原始请求数量；
9. 治理后有效请求数量。

## 9. 抓包数据治理

### 9.1 治理目标
抓包结果不能直接生成代码，必须先进行治理，目标是把原始流量整理为“可进入 `api_test` 的接口候选集合”。

### 9.2 治理步骤
治理分为四层：

1. 清洗：
   - 去除静态资源；
   - 去除无关噪声请求；
   - 去除重复请求；
   - 去除无意义预检请求。

2. 归类：
   - 归到当前模块 / 子模块；
   - 按方法、路径、顺序、动作类型分类。

3. 参数治理：
   - 识别 token、cookie、header；
   - 识别 path/query/body 中需参数化字段；
   - 识别响应中可提取变量。

4. 依赖治理：
   - 建立前后请求的 producer-consumer 候选关系；
   - 为后续 AI 依赖反推保留结构化事实。

### 9.3 用户操作约束
前期用户只做一类核心操作：

> **从治理后的接口候选列表中，勾选真正需要保留的接口。**

用户不再负责：
1. 调整接口调用顺序；
2. 手动拖拽请求链；
3. 手动设计依赖图。

## 10. 接口方法识别、复用与新增规则

### 10.1 识别规则
判断某接口在 `api_test` 中是否已实现，匹配主规则固定为：

> **HTTP 方法 + 完整 URL 路径**

例如：
1. `POST /api/ebuilder/coms/component/enable`
2. `GET /api/ebuilder/coms/component/enable`

必须视为两个不同接口。

### 10.2 识别结果状态
每条治理后的接口候选必须明确展示以下三种状态之一：
1. 已实现，可直接复用；
2. 已实现，但建议补充参数；
3. 未实现，需要新增。

### 10.3 已实现接口处理规则
如果接口已实现：
1. 优先复用已有接口方法；
2. 必要时只补参数或补默认值；
3. 不重复生成第二个同义方法；
4. 参数差异优先落到测试用例层，而不是复制接口方法。

### 10.4 未实现接口处理规则
如果接口未实现：
1. 才允许新增接口方法；
2. 新方法必须生成到 `api_test/core/<project>/<model>/`；
3. 方法命名必须符合现有 `api_test` 风格；
4. 方法应参数化，不得把抓包值写死。

---

## 11. 测试用例生成规则

### 11.1 顺序规则
测试用例中的接口调用顺序固定为：

> **按照抓包抓取到的接口顺序生成**

规则如下：
1. 抓包顺序就是用例调用顺序；
2. 用户只勾选保留哪些必要接口；
3. 用户不调整接口顺序；
4. 平台按抓包原始顺序直接拼接调用链。

### 11.2 后续 AI 依赖方向
后期 AI 的增强方向固定为：

1. 从最终接口开始倒推；
2. 根据最终接口请求参数，寻找这些参数来自哪些前置接口响应；
3. 自动识别并提取变量；
4. 一直追溯，直到整条依赖链完整；
5. 最终形成完整测试用例。

说明：
- 这是后续增强方向；
- 当前轮先不作为必须实现闭环。

### 11.3 用例生成责任边界
接口方法层负责：
1. 请求封装；
2. 入参接收；
3. 响应返回。

测试用例层负责：
1. 调用顺序；
2. 变量传递；
3. 断言组织；
4. 报告步骤；
5. 重试和执行入口承接。

### 11.4 测试用例文件落点
测试用例必须生成到：
- `api_test/tests/<project>/<model>/`

接口方法必须生成到：
- `api_test/core/<project>/<model>/`

## 12. AI 生成与提交门禁

### 12.1 AI 职责
AI 在当前链路中直接负责：
1. 生成完整接口自动化测试用例；
2. 生成缺失的接口方法草稿；
3. 生成断言；
4. 生成 `allure` 步骤结构。

### 12.2 提交门禁
平台提交门禁固定为：

1. AI 生成完成；
2. 自动执行 `pytest`；
3. `pytest` 通过后，才允许进入提交候选状态；
4. 提交后用户可审核和修改；
5. 用户不修改，则默认可直接使用。

即：

> **AI 生成完成 + pytest 通过 + 用户可选审核**

才算正式可用资产。

## 13. Allure 强制规范

### 13.1 适用范围
后续所有新增、生成、改造后的接口自动化测试用例，统一必须满足 `allure` 规范。

### 13.2 强制规则
1. 测试类必须添加 `@allure.feature("...")`
2. 测试用例必须添加 `@allure.story("...")`
3. 测试步骤 / 接口调用步骤必须使用 `with allure.step("...")`
4. 生成用例结构必须天然兼容 `allure` 报告展示
5. 这条规则属于平台编写测试用例的**强制规则**

### 13.3 规范示例
```python
@allure.feature("ebuilder-应用中心")
class TestAppCenter:

    @allure.story("应用中心-EB应用-新建应用")
    def test_create_eb_app(self):
        with allure.step("1.前置接口新建包含封存选项的EB表单和测试数据"):
            pass
```

### 13.4 设计结论
平台未来生成的不是普通 pytest，而是：

> **符合 `allure` 规范的 pytest 接口自动化用例**

- `allure` 测试报告

### 15.2 页面职责

2. 打开 `allure` 报告；
3. 查看失败步骤；

### 15.3 重试定义
失败重试本质上是：
1. 对已生成的 `api_test` 用例重新执行；
2. 不重新抓包；
3. 不重新生成接口方法；
4. 不重新治理原始数据。

## 16. TDD 分批改造边界

### 第一批：界面壳与信息架构
目标：
1. 把当前工作台从治理控制台收敛为三段式接口自动化主界面；
2. 先实现左树、中间列、右侧详情区；
3. 实现主题切换但只换样式不换结构。

测试范围：
1. 左树结构测试；
2. 子模块默认显示测试用例列表测试；
3. 右侧详情区结构测试；
4. 主题切换结构不变测试。

### 第二批：模块级抓包入口与过滤
目标：
1. 在模块 / 子模块级接入代理抓包；
2. 支持开始 / 停止抓包；
3. 支持抓包前置过滤。

测试范围：
1. 抓包入口可见性测试；
2. 代理端口启动 / 停止测试；
3. URL / IP 过滤后仅记录匹配请求测试；
4. 抓包结果归属当前模块 / 子模块测试。

### 第三批：抓包治理与接口识别
目标：
1. 生成治理后的接口候选列表；
2. 判断是否已在 `api_test` 中实现；
3. 给出复用 / 补参 / 新增状态。

测试范围：
1. HTTP 方法 + 完整 URL 路径匹配测试；
2. 已有方法复用测试；
3. 未实现方法新增候选测试；
4. 资产归档路径测试。

### 第四批：测试用例拼接与生成
目标：
1. 按抓包顺序生成调用链；
2. 用户只勾选保留接口；
3. 生成接口方法和测试用例资产。

测试范围：
1. 抓包顺序保持测试；
2. 勾选接口后生成调用链测试；
3. 基础变量传递测试；
4. `allure.feature / story / step` 规范测试；
5. 目录生成路径测试。

### 第五批：pytest 门禁与提交确认
目标：
1. AI 生成完整用例；
2. 自动执行 `pytest`；
3. 通过后才允许提交。

测试范围：
1. 生成后自动执行 `pytest` 测试；
2. 失败时阻断提交测试；
3. 通过后允许提交测试；
4. 提交后在主工作台可见测试。

### 第六批：报告与失败重试
目标：
1. 查看历史执行；
2. 打开 `allure` 报告；
3. 支持失败重试。

测试范围：
1. 历史执行记录展示测试；
2. `allure` 报告入口测试；
3. 失败重试测试。

## 17. 成功标准
本轮主线改造完成后，应满足以下标准：

1. 前端主界面不再以治理控制台为主，而是以测试用例展示与执行为主；
2. 抓包只在模块 / 子模块级出现，并支持前置过滤；
3. 平台只沉淀到 `api_test/core/<project>/<model>/` 与 `api_test/tests/<project>/<model>/`；
4. 已实现接口优先复用，未实现接口再新增；
5. 测试用例按抓包顺序生成；
6. 生成用例必须符合 `allure` 规范；
7. 提交前必须通过 `pytest`；
8. 主界面能执行、重试并查看报告。

---

## 18. 结论
当前设计已经正式收敛为：

> **以 `api_test` 为唯一核心资产落点的接口自动化平台**

即：
1. 前端主入口优先展示和执行测试用例；
2. 抓包只是模块级新增入口；
3. 抓包必须支持前置过滤，从源头减少无用数据；
4. 接口识别采用“HTTP 方法 + 完整 URL 路径”；
5. 测试用例调用顺序固定为抓包顺序；
6. AI 负责直接生成完整测试用例，但提交前必须通过 `pytest`；
7. 最终资产只落到 `api_test/core` 与 `api_test/tests`；
8. 所有测试用例必须遵守 `allure` 规范。

该设计适合进入下一阶段的实现计划拆解与严格 TDD 改造。
