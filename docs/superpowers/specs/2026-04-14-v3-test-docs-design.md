# V3 详细测试文档包设计说明

## 1. 文档说明
- **文档名称**：V3 详细测试文档包设计说明
- **所属项目**：接口自动化测试平台
- **文档目的**：归档本轮围绕 V3 详细测试用例说明书所完成的多轮需求确认结果，作为后续正式编写 `V3` 测试文档包的直接依据。
- **形成时间**：2026-04-14
- **当前状态**：待审阅

---

## 2. 设计背景

当前仓库已经完成：

1. `V1` 正式验收；
2. `V2` 正式验收；
3. `V3阶段工作计划文档.md` 正式化。

根据已经确认的 `V3` 阶段计划：

1. V3 采用 `P0 / P1 / P2` 分层推进；
2. `P0 / P1 / P2` 都必须各自形成独立验收；
3. `P0` 聚焦多项目与环境治理，并做到执行隔离级；
4. `P1` 承接权限体系、抓包正式执行闭环、产品入口深化与调度中心；
5. `P2` 保留高级 AI 治理边界与独立验收要求。

因此，V3 的详细测试文档不再适合继续沿用 V1 / V2 的单文件模式，而应升级为一套可支撑“分层规划 + 分阶段验收 + 独立维护”的测试文档包。

---

## 3. 已确认的设计结论

### 3.1 文档包结构
V3 测试文档采用：

> **1 个总索引 + 3 个分文档**

建议文件如下：

1. `product_document/测试文档/详细测试用例说明书(V3-总索引).md`
2. `product_document/测试文档/详细测试用例说明书(V3-P0).md`
3. `product_document/测试文档/详细测试用例说明书(V3-P1).md`
4. `product_document/测试文档/详细测试用例说明书(V3-P2).md`

### 3.2 总索引职责
总索引文档只负责：

1. 文档说明；
2. 适用范围与阶段边界；
3. 编号规则；
4. 测试分层定义；
5. `P0 / P1 / P2` 独立验收原则；
6. 阶段测试矩阵；
7. 文档映射关系；
8. 维护与更新规则。

总索引 **不重复** 编写详细测试用例本体。

### 3.3 分文档职责
三份分文档分别完整承载：

1. `P0` 详细测试用例；
2. `P1` 详细测试用例；
3. `P2` 详细测试用例。

每份分文档都直接服务于对应阶段的独立验收。

### 3.4 编写深度
本轮已确认：

1. 不是只写 `P0`；
2. 不是 `P0` 详细、`P1 / P2` 简写；
3. 而是 **`P0 / P1 / P2` 三份都需要写得很详细**。

这意味着 V3 测试文档包应视作“阶段级测试总册”，而不是仅服务近期一小轮开发的临时清单。

### 3.5 文档内部组织方式
虽然文件按 `P0 / P1 / P2` 拆分，但每份分文档内部统一采用：

> **按测试层分章**

而不是按任务组分章。

---

## 4. 统一编号与测试层规则

### 4.1 统一编号规则
编号采用全局统一体系，例如：

1. `TC-V3-P0-DOC-001`
2. `TC-V3-P0-API-012`
3. `TC-V3-P1-EXEC-008`
4. `TC-V3-P2-INT-003`

该编号体系固定表达：

1. 所属版本阶段：`V3`
2. 所属阶段：`P0 / P1 / P2`
3. 所属测试层：`DOC / MODEL / MIG / ISO / API / EXEC / UI / INT`
4. 顺序号

### 4.2 测试层固定代号
本轮已确认使用以下固定代号：

1. `DOC`：文档与阶段边界测试
2. `MODEL`：模型与治理对象测试
3. `MIG`：迁移与兼容测试
4. `ISO`：隔离与边界测试
5. `API`：服务与接口契约测试
6. `EXEC`：执行与调度测试
7. `UI`：交互入口与 Demo 测试
8. `INT`：主线集成与阶段验收测试

### 4.3 优先级规则
单条测试用例中继续保留：

1. `优先级`
2. `归属阶段验收`

二者不等价。

示例：

- 编号：`TC-V3-P1-API-012`
- 优先级：`P0`
- 归属阶段验收：`P1`

表示这是 `P1` 阶段的一条 API 用例，但它在 `P1` 阶段内部属于最高优先级测试项。

## 5. 总索引文档建议结构

建议 `详细测试用例说明书(V3-总索引).md` 采用以下结构：

1. 文档说明
2. V3 测试目标
3. 适用范围与阶段边界
4. 文档包结构说明
5. 编号规则
6. 测试分层定义
7. `P0 / P1 / P2` 独立验收原则
8. 阶段测试矩阵
9. 文档映射关系
10. 当前维护规则与更新要求

其中核心内容是：

1. 阶段到文档的映射矩阵；
2. 测试层到分文档的映射矩阵；
3. 统一编号和引用规则；
4. 独立验收与总体验收的关系说明。

---

## 6. 分文档统一模板

三份分文档统一采用以下章节结构：

1. 文档说明
2. 本阶段测试目标
3. 本阶段测试边界
4. 本阶段测试分层结构
5. 文档约束测试用例
6. 模型测试用例
7. 迁移与兼容测试用例
8. 隔离与边界测试用例
9. 服务与接口测试用例
10. 执行与调度测试用例
11. 交互入口测试用例
12. 主线集成与阶段验收测试用例
13. 本阶段通过标准
14. 本阶段不通过判定
15. 变更记录

说明：

1. `P0` 会完整覆盖全部测试层；
2. `P1` 仍保留全部结构，但 `MIG` 会弱化，`EXEC / UI / INT` 会增强；
3. `P2` 仍保留同样结构，但重点将转向 `DOC / RULE边界 / API / INT`，而不是假装其有与 `P0` 一样重的实现面。

---

## 7. 单条用例模板

每条详细测试用例统一采用超详细执行模板：

1. `编号`
2. `名称`
3. `目标`
4. `背景`
5. `前置条件`
6. `输入`
7. `执行步骤`
8. `检查点`
9. `失败判定`
10. `优先级`
11. `预期结果`
12. `归属阶段验收`
13. `后续联动测试`

该模板的定位不是“概要说明”，而是尽量接近后续可直接转为 TDD 与验收执行项的粒度。

---

## 8. 三个阶段的测试重点

### 8.1 P0 测试重点
`P0` 最重：

1. `MIG`：历史资产默认项目迁移；
2. `ISO`：项目级工作区、执行记录、导入导出和环境绑定隔离；
3. `UI`：浏览器端最小治理入口；
4. `INT`：`切项目 -> 选环境 -> 看场景集 -> 执行 -> 回看结果` 的独立验收闭环。

### 8.2 P1 测试重点
`P1` 最重：

1. 权限体系与审计治理；
2. 抓包正式执行闭环；
3. Web 正式入口深化；
4. Windows Demo；
5. 调度与执行中心。

### 8.3 P2 测试重点
`P2` 最重：

1. AI 建议边界；
2. 审批门禁；
3. 回退与留痕；
4. “AI 不能越权成为事实源”的治理闭环。

---

## 9. 当前不采用的方案

本轮已明确不采用以下方案：

1. 不采用单一 `V3` 测试总文档模式；
2. 不采用“只写 `P0`，后面再补 `P1 / P2`”的方式；
3. 不采用“总索引也重复写全部详细用例”的方式；
4. 不采用“各文件各自独立简写编号”的方式；
5. 不采用“按任务组分章”的分文档组织方式。

---
