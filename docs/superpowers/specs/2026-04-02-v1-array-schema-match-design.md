# V1 数组对象结构断言增强设计说明

## 1. 文档信息
- 日期：2026-04-02
- 主题：增强 `platform_core` 对数组对象结构的 `schema_match` 断言支持
- 适用范围：`platform_core/`、`tests/platform_core/`、README 与 V1 阶段文档
- 关联目标：
  - 补齐 V1 当前仍未落地的“更深层对象/数组结构断言”能力
  - 在不引入完整 JSON Schema 校验器的前提下，增强文档驱动最小闭环
  - 继续降低生成测试骨架与断言语义不一致的风险

## 2. 背景与问题
当前 `platform_core` 已支持：
1. `status_code`
2. `json_field_exists`
3. `json_field_equals`
4. `schema_match` 的对象和数组最小类型断言

但数组结构支持仍然过浅：
1. 对 `type=array` 的断言目前只能校验“是不是列表”；
2. 若响应为“对象数组”，解析器无法提取数组项的关键字段；
3. 模板层也无法为数组项生成字段级结构断言；
4. 生成测试骨架的假响应同样无法表达“列表内元素是对象且具备字段”的约束。

这会让当前 V1 只能覆盖“数组存在且类型正确”，还无法稳定承接更常见的列表接口结构。

## 3. 方案比较

### 方案一：增强 `schema_match`，支持数组项对象结构，推荐
做法：
1. 在解析器中识别数组项为对象的响应结构；
2. 生成带 `item_type` 和 `required_fields` 的 `schema_match`；
3. 在模板层对数组项逐个校验；
4. 在假响应生成器中构造最小对象数组。

优点：
1. 直接补齐当前 P1 缺口；
2. 与已有 `schema_match` 能力保持同一路线；
3. 对流水线、模板和规则层的改动可控。

缺点：
1. 需要为数组项结构补充一组新的最小约束；
2. 需要同步更新测试和文档。

### 方案二：直接实现 `business_rule` 断言
做法：
1. 新增业务规则表达结构；
2. 新增模板和规则解释逻辑。

优点：
1. 能扩展断言能力边界。

缺点：
1. 当前输入结构不稳定；
2. 更容易把 V1 范围拉大；
3. 不如先把现有 `schema_match` 做扎实。

## 4. 选型结论
本轮采用方案一。

原因：
1. 它直接对应《详细测试用例说明书(V1)》里的 P1 剩余项；
2. 它仍然属于现有 `schema_match` 的自然增强，而不是新能力线；
3. 它能继续服务当前“文档驱动最小闭环”主线。

## 5. 设计目标

### 5.1 主目标
把 `schema_match` 的数组支持从：

```json
{
  "type": "array"
}
```

增强为可表达数组项对象结构：

```json
{
  "type": "array",
  "item_type": "object",
  "required_fields": ["id", "name"]
}
```

### 5.2 必须达成的结果
1. 解析器可从“对象数组”响应结构中生成增强版 `schema_match`；
2. 模板层可渲染数组项对象字段校验；
3. 规则层可校验 `item_type` 与 `required_fields` 的合法性；
4. 生成测试骨架的假响应可生成最小对象数组；
5. 文档驱动流水线和全量回归继续通过。

### 5.3 非目标
1. 不实现完整 JSON Schema 数组项递归校验；
2. 不支持数组索引路径语法；
3. 不实现 `business_rule`；
4. 不扩展功能用例驱动和抓包驱动。

## 6. 最小结构设计

### 6.1 `expected_value` 结构
当目标字段是对象数组时，`schema_match` 结构如下：

```json
{
  "type": "array",
  "item_type": "object",
  "required_fields": ["id", "name"]
}
```

约束：
1. `type` 必须为 `array`；
2. `item_type` 可选，但若存在则必须为支持的类型之一；
3. `required_fields` 仅在 `item_type=object` 时有意义；
4. 当前只支持对数组项的第一层对象字段做存在性校验。

### 6.2 模板行为
渲染后的断言至少包括：
1. 目标路径存在；
2. 目标值为列表；
3. 若指定 `item_type`，则逐项校验类型；
4. 若 `item_type=object` 且存在 `required_fields`，则逐项校验字段存在。

### 6.3 假响应构造
当断言为对象数组时，生成测试骨架中的假响应应至少生成：

```json
{
  "data": [
    {
      "id": "sample-id",
      "name": "sample-name"
    }
  ]
}
```

## 7. 测试设计

### 7.1 模板层
新增测试验证：
1. `schema_match` 能渲染数组项对象结构断言；
2. 假响应生成器能生成对象数组。

### 7.2 解析层
新增测试验证：
1. 对象数组能生成 `item_type=object`；
2. `required_fields` 正确提取。

### 7.3 规则层
新增测试验证：
1. 数组断言非法 `item_type` 会被拦截；
2. `item_type` 不是 `object` 时不允许携带 `required_fields`。

### 7.4 集成层
新增测试验证：
1. 文档驱动流水线生成的测试骨架包含数组项断言；
2. 该测试骨架可执行通过。

## 8. 风险与控制

### 风险 1：把数组结构断言做成递归 Schema 校验
控制：
1. 只支持数组项第一层结构；
2. 不引入外部 schema 校验库。

### 风险 2：规则设计过重
控制：
1. 仅对 `item_type` 和 `required_fields` 做最小合法性检查；
2. 不扩展到更多断言类型。

## 9. 验收标准
只有同时满足以下条件，本轮才算完成：
1. 新增数组对象结构断言测试先失败后通过；
2. `tests/platform_core` 全量通过；
3. 根测试与 `api_test` 回归未被破坏；
4. README 与 V1 阶段文档同步到真实状态。

## 10. 结论
本轮仍然属于 V1 的模板/规则覆盖增强，而不是新路线扩张。

通过把数组对象结构纳入 `schema_match` 的最小支持范围，当前平台对列表型接口的承接能力会更完整，也更接近后续产品形态需要的稳定结构断言能力。
