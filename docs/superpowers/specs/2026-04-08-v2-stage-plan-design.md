# V2阶段工作计划文档设计说明

## 1. 文档说明
- **文档名称**：V2阶段工作计划文档设计说明
- **所属项目**：接口自动化测试平台
- **文档目的**：在正式改写 `product_document/阶段文档/V2阶段工作计划文档.md` 前，固化 V2 阶段的定位、范围、任务组、测试策略、验收标准和文档结构设计，作为后续计划文档编写依据。
- **形成时间**：2026-04-08
- **当前状态**：待审阅

---

## 2. 背景与问题

当前仓库的 V1 已于 2026-04-03 完成正式验收，`platform_core` 和 `api_test` 已具备文档驱动最小闭环、统一中间模型雏形、模板体系雏形、规则校验雏形、资产清单与执行记录能力。

但当前 `product_document/阶段文档/V2阶段工作计划文档.md` 仍处于“简版规划（待后续细化）”状态，只有方向性条目，缺少以下关键内容：

1. V2 阶段定位与 V1/V3 的边界；
2. V2 的进入条件、正式闭环定义与不做项；
3. 任务组拆分、实施顺序与承接关系；
4. 开发方案、测试方案、进度记录、风险记录；
5. 服务化与数据持久化深度；
6. 产品入口目标深度与可交付边界。

如果继续维持当前占位稿状态，后续 V2 开发和测试会失去统一阶段依据，并重新暴露“多方向并行失控”的风险。

---

## 3. V2 的阶段定位

V2 应定义为：

> **从“文档驱动最小闭环”升级到“场景驱动优先、服务化落地、可用型产品入口承接”的能力增强阶段。**

V2 的核心含义如下：

1. 不再只补 V1 收口项，而是正式落地 V1 预留的场景、变量、依赖和审核能力；
2. 同时做核心能力和产品入口，但阶段优化重点放在核心能力；
3. 把 Django + DRF + MySQL 从“预留方向”升级为“阶段必达能力”；
4. 让本地工作区从唯一事实源降级为导入导出与 pytest 执行载体；
5. 让功能测试用例驱动成为 V2 第一主线，抓包驱动作为同阶段草稿化接入能力。

---

## 4. 已确认的 V2 关键决策

### 4.1 总体主线
V2 同时覆盖核心能力和产品入口，但优先优化核心能力。

### 4.2 第一优先输入路线
V2 第一主线为“功能测试用例驱动优先”。

### 4.3 抓包驱动在 V2 的目标深度
抓包驱动纳入 V2，但先做到：

1. 抓包导入；
2. 清洗、去重、归一化；
3. 动态值候选识别；
4. 接口识别与顺序提取；
5. 场景草稿生成；
6. 审核确认入口接入。

V2 不把“抓包驱动完整执行闭环”作为必达项。

### 4.4 产品入口深度
Web / Windows 入口采用“可用型入口”目标：

1. 支持导入；
2. 支持预览；
3. 支持审核确认；
4. 支持触发执行；
5. 支持结果查看。

V2 不追求完整成熟产品体验。

### 4.5 服务化与数据持久化深度
V2 采用“核心数据先服务化落地”：

1. Django + DRF + MySQL 进入阶段必达范围；
2. 结构化资产、记录和状态以数据库为事实源；
3. 本地工作区继续保留，但主要承担导入导出、生成落盘和 pytest 执行载体职责。

### 4.6 审核与确认策略
V2 采用“默认可审核、关键节点需确认”：

1. 允许先生成草稿和预览；
2. 在“正式入库为场景资产”前需要人工确认；
3. 在“触发正式执行”前需要人工确认。

### 4.7 修订策略
V2 同时支持：

1. 结构化修订；
2. AI 辅助改写。

但 AI 改写结果不能直接成为正式事实资产，仍需经过规则校验和关键确认节点。

---

## 5. V2 不做什么

V2 应明确不做以下内容：

1. 不把抓包驱动完整执行闭环作为阶段必达项；
2. 不做完整成熟的 Web 产品化体验；
3. 不做多人协作、权限体系和中心化组织管理；
4. 不做高级 AI 自愈、自动修复和大规模自治编排；
5. 不把 AI 自由改写结果直接落为正式资产；
6. 不把 V2 写成 V3 的替代版本。

---

## 6. V2 核心任务组与实施顺序

V2 建议拆分为以下 6 个任务组。

### 6.1 任务组 A：场景驱动核心模型扩展
目标：

1. 将 `TestScenario`、`ScenarioStep`、`VariableBinding`、`DependencyLink`、`ReviewRecord` 正式落地；
2. 建立数据库模型映射；
3. 建立服务层稳定契约；
4. 建立与 V1 文档驱动资产的关联方式。

### 6.2 任务组 B：功能测试用例驱动闭环
目标：

1. 支持功能测试用例输入；
2. 解析为场景模型；
3. 绑定已有接口资产；
4. 推导变量与依赖；
5. 生成场景草稿与测试资产；
6. 经确认后进入执行闭环。

### 6.3 任务组 C：审核、预览、修订与确认链路
目标：

1. 定义草稿、已确认、已驳回、已修订等状态；
2. 支持结构化修订；
3. 支持 AI 辅助改写；
4. 支持关键节点确认门禁；
5. 形成资产生命周期治理能力。

### 6.4 任务组 D：服务化与数据持久化落地
目标：

1. 建立 Django 应用服务边界；
2. 建立 DRF API；
3. 建立 MySQL 表结构与迁移策略；
4. 建立数据库事实源与本地工作区执行载体的关系；
5. 让审核、执行、结果和状态能统一查询与回写。

### 6.5 任务组 E：抓包驱动草稿化接入
目标：

1. 支持抓包导入；
2. 支持抓包清洗和去重；
3. 支持参数模板化和动态值候选识别；
4. 支持接口识别与场景草稿生成；
5. 进入审核确认链路。

### 6.6 任务组 F：可用型 Web / Windows 入口
目标：

1. 让 Web / Windows 入口能承接导入、预览、审核确认、执行、结果查看；
2. 确保入口只消费服务层稳定能力，不反向绑死核心流程；
3. 保持 Web 与 Windows 的统一流程语言。

### 6.7 推荐实施顺序

1. 任务组 A：场景驱动核心模型扩展
2. 任务组 B：功能测试用例驱动闭环
3. 任务组 C：审核、预览、修订与确认链路
4. 任务组 D：服务化与数据持久化落地
5. 任务组 E：抓包驱动草稿化接入
6. 任务组 F：可用型 Web / Windows 入口

该顺序的核心原则是：

1. 先落模型与场景主线；
2. 再落审核与状态流；
3. 再落服务化与持久化；
4. 再让抓包与入口承接稳定核心能力。

## 7. V2 正式闭环定义

V2 的阶段完成，不应定义为“三条路线全部成熟”，而应定义为以下闭环同时成立：

### 7.1 功能测试用例驱动正式闭环
1. 输入功能测试用例；
2. 解析并标准化；
3. 生成场景草稿；
4. 建立变量与依赖；
5. 审核确认；
6. 形成正式场景资产；
7. 导出工作区并执行；
8. 结果回写数据库与本地执行产物。

### 7.2 抓包驱动草稿化闭环
1. 导入抓包数据；
2. 清洗、去重、归一化；
3. 提取接口与顺序；
4. 生成场景草稿；
5. 进入审核确认链路。

### 7.3 可用型产品入口闭环
1. Web / Windows 入口完成导入；
2. 能预览场景草稿；
3. 能执行审核确认；
4. 能触发执行；
5. 能查看结果摘要。

## 8. V2 测试策略与验收标准

### 8.1 建议新增独立测试文档
建议新建 `product_document/测试文档/详细测试用例说明书(V2).md`，不在 V1 测试文档上继续叠加。

### 8.2 V2 测试分层
V2 至少应覆盖以下 8 层：

1. 文档与阶段约束测试；
2. 中间模型测试；
3. 解析与标准化测试；
4. 模板与规则测试；
5. 服务层与持久化测试；
6. 执行与调度测试；
7. 交互入口测试；
8. 主线集成闭环测试。

### 8.3 V2 必须满足的验收标准
以下条件同时满足时，才能宣称 V2 完成：

1. 功能测试用例驱动能够生成可审核场景草稿，并在确认后形成正式场景资产；
2. 场景资产支持变量绑定、依赖关系和多步骤执行；
3. 抓包驱动能够完成清洗、归一化、接口识别和场景草稿生成，并进入审核确认链路；
4. Django + DRF + MySQL 已成为核心资产与记录的正式承载层；
5. Web / Windows 入口能够完成导入、预览、确认、执行、结果查看；
6. AI 改写结果必须经过规则校验和关键确认节点；
7. V2 定义的 P0 测试项全部通过。

## 12. 可借鉴的外部设计模式

### 12.1 RESTler
可借鉴点：

1. producer-consumer 依赖关系推断；
2. 分级执行思路；
3. stateful 场景执行优先于纯单点生成。

来源：https://github.com/microsoft/restler-fuzzer

### 12.2 mitmproxy2swagger
可借鉴点：

1. 抓包不能直接进最终资产；
2. 必须先做路径模板化和人工筛选；
3. 抓包路线天然需要中间清洗层。

来源：https://github.com/alufers/mitmproxy2swagger

### 12.3 Schemathesis
可借鉴点：

1. 契约驱动执行入口；
2. CLI / CI 友好的验证链路；
3. schema-first 的执行增强思路。

来源：https://github.com/schemathesis

### 12.4 openapi-python-client / OpenAPI Generator
可借鉴点：

1. 模板与生成器解耦；
2. 配置驱动生成；
3. tag / module 归组策略。

来源：
- https://github.com/openapi-generators/openapi-python-client
- https://github.com/OpenAPITools/openapi-generator

## 13. 结论

V2 不是“继续补 V1 的一些小能力”，而是正式进入：

> **场景驱动能力建设 + 服务化落地 + 可用型入口承接**

因此，正式版 `V2阶段工作计划文档.md` 应从占位稿升级为完整阶段计划文档，并围绕以下主线展开：

1. 功能测试用例驱动优先；
2. 抓包驱动草稿化接入；
3. 审核、修订、确认成为正式生命周期；
4. 数据库成为事实源；
5. Web / Windows 入口承接稳定核心能力；
6. V2 的完成以“正式闭环成立 + P0 通过”为准，而不是以愿景描述为准。
